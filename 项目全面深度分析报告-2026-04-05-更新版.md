## 一、项目全景（更新）

### 1.1 产品定位

云客CRM是一个 **SaaS + 私有部署双模式** 的完整CRM商业产品体系，包含5个子系统 + 1个新增会员中心模块：

| 子系统 | 技术栈 | 文件数（最新统计） | 端口 | 定位 |
|--------|--------|--------|------|------|
| **Backend** | Node.js + Express + TypeORM + MySQL | **291个TS文件** | 3000 | 统一后端API服务 |
| **CRM前端** | Vue 3 + TS + Element Plus + Pinia | **254个Vue + 151个TS** | 5173 | 主业务系统（员工使用） |
| **Admin后台** | Vue 3 + TS + Element Plus + Axios | **40个Vue + 11个TS** | 5174 | 平台管理后台（运营方使用） |
| **Website官网** | Vue 3 + TS + 纯手写SCSS | **25个Vue + 10个TS** | 8080 | 产品展示 + 注册 + 支付 + 会员中心 |
| **crmAPP** | uni-app + Vue 3 + TS | **18个Vue + 16个TS** | -- | 电销外呼手机APP |

**全项目代码总量：约 370,000+ 行（较3月29日增长约40,000行）**

### 2.2 认证体系（更新 - 三套独立JWT）

| 认证系统 | 中间件 | JWT Secret | 有效期 | 用途 |
|---------|--------|-----------|--------|------|
| **CRM用户认证** | `authenticateToken` | JWT_SECRET | 7天(生产)/365天(开发) | 员工登录CRM系统 |
| **Admin管理员认证** | `adminAuthMiddleware` | ADMIN_JWT_SECRET | -- | 管理员登录Admin后台 |
| **会员中心认证（新）** | `memberAuth` | MEMBER_JWT_SECRET | 7天 | 租户登录会员中心 |

### 2.3 通信方式

| 连接类型 | 技术 | 用途 |
|---------|------|------|
| HTTP REST | Express + JWT | 所有CRUD操作 |
| WebSocket (Socket.IO) | socket.io | PC端实时消息推送 |
| WebSocket (ws) | ws | 移动端通话指令/状态同步 |
| 定时任务 | node-cron | 授权过期/支付提醒/消息清理/超时提醒/业绩报表/订阅扣款/扣款重试 |

### 3.3 多租户隔离最新状态

**4月5日状态**：全面覆盖 + 安全审计通过
**4月8日状态**：WebSocket + MessageController + 移动端登录安全审查修复完成

三层保护机制：
| 层级 | 组件 | 功能 |
|------|------|------|
| 第1层 | `tenantContextMiddleware` | AsyncLocalStorage 设置租户上下文 |
| 第2层 | `getTenantRepo()` Proxy | 自动拦截 find/save/update/delete/createQueryBuilder，注入 tenant_id |
| 第3层 | `tenantSQL()` / `tenantWHERE()` | 原生SQL场景的租户过滤辅助函数 |
| 第4层(新) | WebSocket 租户房间隔离 | `tenant_{id}` 房间 + 角色/部门房间含租户前缀 |

4月8日深度安全审查发现并修复7个问题（详见第十九章）：
| 问题 | 风险 | 修复 |
|------|------|------|
| WebSocket房间无租户前缀 | 🔴高 | 房间名改为 `tenant_{id}_role/department_{id}` |
| WebSocket broadcast跨租户 | 🔴高 | 新增租户范围广播 `broadcast(event, data, tenantId)` |
| 部门订阅配置查询无租户过滤 | 🔴高 | 通过Department关联过滤tenantId |
| WebSocket事件缺少租户上下文 | 🟡中 | 新增 `withTenantContext()` 方法恢复AsyncLocalStorage |
| 公告WebSocket推送跨租户 | 🟡中 | 传入tenantId限定推送范围 |
| 移动端登录SaaS未强制tenantId | 🟡中 | 添加 `deployConfig.isSaaS()` 校验 |
| 通知渠道sendToRole跨租户 | 🟡中 | 传入当前tenantId到sendToRole |

## 十、扩容管理系统（新增）

### 10.1 功能概述

支持租户独立购买额外用户数和存储空间，不需要升级套餐：

| 扩容类型 | 说明 | 累加方式 |
|---------|------|----------|
| 用户数扩容 | 购买额外N个用户席位 | tenants.extra_users += N, max_users同步更新 |
| 存储空间扩容 | 购买额外N GB存储 | tenants.extra_storage_gb += N, max_storage_gb同步更新 |

### 10.2 安全修复（验收审查发现7个问题）

| 问题 | 严重度 | 修复 |
|------|--------|------|
| activateCapacity竞态条件 | 严重 | 改为原子CAS操作 |
| 自动关闭误杀扩容订单 | 严重 | 扩容/套餐订单独立处理 |
| 类型与价格不匹配 | 严重 | 添加type校验 |
| 退款状态语义不一致 | 中 | ENUM新增refunded值 |
| CSV导出缺订单类型列 | 中 | 新增列 |
| 价格配置缺业务校验 | 中 | 单价>0 + min<=max校验 |
| 未校验租户状态 | 中 | 暂停/过期租户禁止扩容 |

### 10.3 续费保留机制

套餐续费时保留扩容额度：
```sql
max_users = 新套餐基础用户数 + COALESCE(extra_users, 0)
max_storage_gb = 新套餐基础存储 + COALESCE(extra_storage_gb, 0)
```

## 十八、2026-04-08 Admin租户详情页配置模块样式修复（新增）

### 18.1 问题概述

管理后台租户客户详情页（`tenant-customers/Detail.vue`）在之前重构拆分为子组件时，**9个子组件的样式全部丢失**，导致「功能模块授权」配置区域从横向网格卡片布局变成竖排无样式排列。

Vue 的 `<style scoped>` 通过编译时为元素和 CSS 选择器添加唯一 `data-v-xxxxx` 属性实现样式隔离。当单文件组件被拆分为多个子组件后，父组件的 scoped 样式无法穿透到子组件内部元素，而子组件本身未定义任何样式。

## 二十、2026-04-08 物流管理模块API配置审查与修正（新增）

### 20.1 问题背景

物流管理模块支持9家主流快递公司API对接，其中顺丰(SF)已验证正确，其余8家公司在初始开发时存在签名算法错误、URL不一致、字段映射混乱等问题，无法直接投入生产使用。

### 20.2 修复清单

| 物流公司 | 问题 | 修复方案 |
|----------|------|---------|
| **中通(ZTO)** | 签名算法使用MD5，应为HMAC-SHA256+Base64 | 后端签名改为`crypto.createHmac('sha256', appSecret).update(data).digest('base64')`，URL更新为`traceInterfaceNewTraces` |
| **圆通(YTO)** | 字段映射混乱(appId存AppKey、appKey存AppSecret、appSecret存UserId)；请求体字段名错误 | 简化为3字段(appId→AppKey, appSecret→SecretKey, customerId→客户编码)；请求体`data`→`param`，`waybillNo`→`Number` |
| **极兔(JTSD)** | 签名编码hex应为base64；沙箱域名错误 | `.digest('hex')`→`.digest('base64')`；域名`openapi-test`→`uat-openapi` |
| **德邦(DBL)** | 签名编码hex应为base64 | `.digest('hex').toUpperCase()`→`.digest('base64')` |
| **邮政EMS** | 前端URL缺少具体接口路径，与后端不一致 | URL补全为`/openapi/mailTrack/query` |
| **韵达(YD)** | URL路径含typo(`logictis`) | 修正为标准端点`/api/queryTraceInfo` |
| **申通(STO)** | 缺少配置引导 | 添加setupSteps（签名和接口结构已正确） |
| **京东(JD)** | 缺少配置引导 | 添加setupSteps（签名和接口结构已正确） |
| **快递100** | 响应解析bug：`returnCode==='200'`永远为false | 改为`message==='ok'`，修复data/state字段路径 |

### 20.3 查询优先级逻辑

实现官方API优先、快递100降级的智能查询策略：
- 官方API已配置 → 优先使用，失败降级快递100

- 两者都未配置 → 返回配置提示
