## 问题一：来电响铃时号码不显示

### 1.1 问题描述

- **现象**: 工作手机客户呼入时，来电提醒弹窗和 APP 在响铃期间显示"未知客户"/"未知来电"
- **待机后**: 挂断后能自动补全看到号码
- **影响机型**: 荣耀手机、OPPO 手机（已确认），预期所有 Android 10+ 设备
- **严重程度**: **高** — 租户无法在来电时判断是老客户还是新客户，影响客户服务质量

### 1.2 完整调用链路

```
手机系统来电
  ├─ PhoneStateListener.onCallStateChanged(state=1, phoneNumber="")  ← 号码为空（Android 10+ 隐私限制）
  ├─ PHONE_STATE 广播接收器 → incoming_number 也可能为空
  │
  └─ handleCallStateChange(state=1, phoneNumber="")
       ├─ phoneNumber 为空 → quickResolveAndReport()
       │    ├─ onRingingDetected("未知来电")        ← 🔴 立即上报 CRM 弹窗，号码为"未知来电"
       │    ├─ startCallLogPolling()               ← 每300ms 轮询 CallLog
       │    │    └─ 响铃/RINGING 期间 CallLog 未写入 → 永远查不到 ❌
       │    │    └─ IDLE/挂断后 CallLog 已写入   → 能查到 ✅ → 但弹窗已关闭
       │    │
       │    └─ reportIncomingToServer("未知来电")   ← WebSocket/HTTP 上报
       │         └─ 后端 MobileWebSocketService.handleIncomingCallDetected()
       │              └─ 创建 call_records: customer_phone="未知来电"
       │              └─ 推送 CRM 端 CALL_INCOMING → 弹窗显示"未知客户"
       │
       └─ 挂断: onIncomingCallEnded()
            ├─ tryResolveCallerFromCallLog()   ← 此时 CallLog 已写入，成功获取号码
            ├─ tryResolveCallerFromRecordingFile()  ← 从录音文件名补获
            └─ 更新 call_records + 通知CRM端  ← 挂断后号码才显示
```

### 1.3 根因逐层分析

#### 第一层：PhoneStateListener 不返回号码（系统层面）

**代码位置**: [`crmAPP/src/services/incomingCallService.ts#L266-L288`](file:///d:/kaifa/CRM - 1.8.0/crmAPP/src/services/incomingCallService.ts#L266-L288)

```typescript
// PhoneStateListener 回调
onCallStateChanged: function() {
  const args = Array.prototype.slice.call(arguments)
  const state = Number(args[0]) || 0
  const rawNumber = args[1]
  const pn = rawNumber ? ('' + rawNumber).trim() : ''
  // phoneNumber 在 Android 10+ 设备上经常为空
  self.handleCallStateChange(state, pn)
},
```

**原因**: Android 10 引入了新的隐私限制。即使应用声明 `READ_CALL_LOG` 权限，`PhoneStateListener.onCallStateChanged` 的 `incomingNumber` 参数也经常返回空字符串。华为/荣耀 EMUI/HarmonyOS、OPPO ColorOS 等国产 ROM 会进一步强化这一限制。

#### 第二层：空号码立即上报"未知来电"

**代码位置**: [`crmAPP/src/services/incomingCallService.ts#L985-L1000`](file:///d:/kaifa/CRM - 1.8.0/crmAPP/src/services/incomingCallService.ts#L985-L1000)

```typescript
private async quickResolveAndReport() {
  // 立即用"未知来电"启动流程，确保弹窗不延迟
  this.onRingingDetected('未知来电')   // ← 🔴 关键问题所在！
  this.startCallLogPolling()
}
```

```typescript
// handleCallStateChange 中的 RINGING 分支
case 1: // RINGING
  if (phoneNumber && phoneNumber.length >= 3 && phoneNumber !== '未知来电') {
    this.onRingingDetected(phoneNumber)    // 有号码 → 正常流程
  } else if (!this.currentIncoming) {
    this.quickResolveAndReport()           // 无号码 → 直接报"未知来电"
  }
```

**问题**: 代码为了"弹窗不延迟"，在没有拿到真实号码的情况下，立即以 `"未知来电"` 启动完整流程——包括创建通话记录、推送 CRM 弹窗。此时弹窗已经显示"未知客户"，后续即使轮询获取到号码，用户的第一印象已经形成。

#### 第三层：CallLog 在响铃期间 100% 查询失败

**代码位置**: [`crmAPP/src/services/incomingCallService.ts#L1024-L1026`](file:///d:/kaifa/CRM - 1.8.0/crmAPP/src/services/incomingCallService.ts#L1024-L1026)

```typescript
// 通话结束(IDLE)后再多查 20 次（6秒），因为大部分设备在 IDLE 后才写入 CallLog
if (this.lastPhoneState === 0) {
  idleCount++
  if (idleCount > 20) { /* 停止 */ }
}
```

**这是关键**: Android 系统的 CallLog 数据库**只在通话结束后才写入记录**。在 RINGING（响铃）或 OFFHOOK（通话中）期间，`content://call_log/calls` 查询**永远返回空或旧记录**。代码注释中也确认了这点。这意味着：

> **轮询策略在响铃期间注定失败，号码只能挂断后获取。**

### 1.4 为什么"之前能显示"、"有时串号"

| 时期 | 情况 | 原因 |
|------|------|------|
| 早期 | 能显示号码 | Android 9 及以下系统，`PhoneStateListener` 正常返回号码 |
| 中期 | 有时能显示、有时串号 | Android 10+ 部分设备仍能返回，但 CallLog 定时查询可能获取到上一通的号码（串号） |
| 现在 | 完全不显示 | 荣耀/OPPO 最新系统彻底拦截，`PhoneStateListener` 完全返回空 |

### 1.5 修复方案

#### 方案 A（推荐）：延迟上报 + 等待广播补号

**不立即以"未知来电"上报**，设置 800ms~1500ms 的延迟窗口：

1. 在 `handleCallStateChange` RINGING 分支中，当 phoneNumber 为空时：
   - 先创建 `currentIncoming` 但不调用 `onRingingDetected`
   - 启动 1500ms 超时定时器
   - 同时注册一个临时的 PHONE_STATE 广播监听
2. 1500ms 内如果收到带号码的广播：
   - 用真实号码调用 `onRingingDetected`
3. 超时后仍无号码：
   - 才以 `onRingingDetected('未知来电')` 上报
   - 同时在 APP 端显示"客户来电中..."而不是"未知来电"

#### 方案 B：强化 PHONE_STATE 广播接收器

当前代码已有 `registerPhoneStateReceiver`（Activity Context）和 `registerAltPhoneStateReceiver`（Application Context）。可通过以下措施增强：

1. 提升广播接收器的优先级（`filter.setPriority(2147483647)` 已做）
2. 添加 `RECEIVER_EXPORTED` 标志（Android 13+，已做）
3. 在广播中获取号码后立即更新 `currentIncoming` 并重新通知 CRM 端

#### 方案 C：使用 TelecomManager API（Android 9+）

考虑调用 `TelecomManager` 获取当前通话信息：

```java
TelecomManager tm = getSystemService(TelecomManager.class);
if (tm.isIncomingCallPermitted()) {
    // 可能通过 getCallCapablePhoneAccounts 等获取来电信息
}
```

但此 API 在非系统拨号器应用上权限有限。

#### 方案 D：APP 端 UI 优化

即使拿到号码需要时间，也可以在 APP 的来电页面显示"客户来电中..."而非"未知来电"，避免给用户"系统不知道是谁"的负面印象。

## 问题二：OPPO 手机自动录音检测不到/未上传

### 2.1 问题描述

- **现象**: OPPO 在拨号应用开启了自动录音，但 APP 没有检测到并上传录音
- **待确认**: 其他安卓机型是否也存在类似问题
- **严重程度**: **中** — 影响录音自动上传功能，但可通过手动上传补救

### 2.2 完整调用链路

```
通话结束
  └─ processRecordingAsync()
       └─ recordingService.processCallRecording()
            ├─ checkPermissions()                    ← 检查存储权限
            ├─ 等待 2 秒（wait for recording file）
            ├─ findMatchingRecording()               ← 扫描录音目录 + 匹配
            │    ├─ scanRecordingFolders()           ← 扫描预定义路径
            │    ├─ getPhoneVariants()               ← 电话号码变体
            │    └─ 时间匹配 + 文件名匹配 + 大小匹配
            └─ uploadRecordingFile()
```

### 2.3 根因分析

#### 问题 2.3.1：OPPO 录音路径可能不完整

**代码位置**: [`crmAPP/src/services/recordingService.ts#L32-L58`](file:///d:/kaifa/CRM - 1.8.0/crmAPP/src/services/recordingService.ts#L32-L58)

当前 OPPO 专属路径：
```typescript
'/storage/emulated/0/Recordings/Call Recordings/',
'/storage/emulated/0/Music/Recordings/Call Recordings/',
```

**可能缺失的路径**:

| ColorOS 版本 | 实际录音路径 | 是否已覆盖 |
|-------------|-------------|-----------|
| ColorOS 12 | `/storage/emulated/0/Music/Recordings/Call Recordings/` | ✅ |
| ColorOS 13/14 | `/storage/emulated/0/Recordings/Call Recordings/` | ✅ |
| 部分机型 | `/storage/emulated/0/Recordings/`（根目录） | ❌ |
| OPPO 录音机 App | `/storage/emulated/0/Android/data/com.coloros.soundrecorder/` | ❌ |
| 文件管理器可访问 | `/storage/emulated/0/Music/Recordings/` | ❌（仅在 scan 列表，不在专属路径） |

#### 问题 2.3.2：录音文件名匹配不适应 OPPO 格式

OPPO 录音文件命名格式：
- `联系人名称_20260709_143022.aac`（以联系人为名）
- `13812345678_20260709_143022.amr`（以号码为名）
- `通话录音_20260709143022.mp3`

当前 `getPhoneVariants()` 只生成手机号的标准变体（138****, 86格式等），**不处理"联系人名称"格式**。如果用户在 OPPO 通讯录中保存了该客户的姓名，录音文件会以联系人名称命名，而非号码，导致号码匹配失败。

#### 问题 2.3.3：系统 Settings 键值未覆盖 OPPO

`checkSystemRecordingSetting()` 尝试读取系统 Settings 键值判断录音是否开启，但 OPPO ColorOS 的录音开关可能存储在：

- `com.android.phone` 的 SharedPreferences 中（APP 内部，外部无法读取）
- 或 `Settings.Global` 中未被探测到的键名

#### 问题 2.3.4：OPPO 通过拨号 APP 开启录音的检测盲区

用户提到"在拨号应用开启的"——OPPO 的通话自动录音开关在 ColorOS 电话APP → 设置 → 通话录音 中设置，**不在系统全局设置**中。这意味着 `checkSystemRecordingSetting()` 和 MediaStore 检测都可能找不到。

### 2.4 修复方案

#### 方案 A：补充 OPPO 录音路径

在 `CALL_SPECIFIC_PATHS` 和 `getBrandSpecificPaths` 中增加：

```
'/storage/emulated/0/Recordings/'         // OPPO 录音根目录
'/storage/emulated/0/Recordings/Call/'    // 子目录
'/storage/emulated/0/Music/Recordings/'   // 通用音乐目录
```

#### 方案 B：改进文件名匹配，降低对号码的依赖

当前 `findMatchingRecording()` 的评分权重中，时间匹配仅占 50 分（满分），文件名中的号码匹配占 40 分。对于 OPPO 以联系人名称命名的文件：

1. **提高时间匹配权重**：当时间窗口高度吻合（±5秒内）时给予高分
2. **增加文件大小估算范围**：增加 `.aac` 格式的预估码率（AAC 约为 5-8KB/s，与 amr 不同）
3. **OPPO 设备放宽阈值**：在 OPPO 设备上，匹配分数阈值从 50 降低到 35

#### 方案 C：增强录音上传的重试与延迟

OPPO 录音文件写入可能有 3-5 秒延迟，当前 `processCallRecording` 固定等待 2 秒不够：

```typescript
// 当前：固定 2 秒等待
await new Promise((resolve) => setTimeout(resolve, 2000))

// 建议：递增重试 2s → 3s → 5s → 8s
const delays = [2000, 3000, 5000, 8000]
```

#### 方案 D：增加 MediaStore 对 OPPO 路径的查询

在 `checkMediaStoreForCallRecordingsStrict()` 中增加 OPPO 专属路径条件：

```sql
LOWER(data) LIKE '%/recordings/call%'
OR LOWER(data) LIKE '%/recordings/%通话录音%'
```

## 问题三：APP 登录错误提示不正确

### 3.1 问题描述

- **现象**: 密码错误、用户不存在时，APP 提示"登录已过期"而非实际错误信息
- **严重程度**: **中** — 用户无法判断登录失败的原因，增加客服支持成本

### 3.2 根因定位

**核心问题在 `request.ts` 的 401 错误处理逻辑**。

**代码位置**: [`crmAPP/src/utils/request.ts#L137-L145`](file:///d:/kaifa/CRM - 1.8.0/crmAPP/src/utils/request.ts#L137-L145)

```typescript
// Token过期 - 尝试静默重新登录并重试
if (res.statusCode === 401) {
  const pages = getCurrentPages()
  const currentPage = pages[pages.length - 1]
  const currentPath = currentPage?.route || ''

  if (currentPath.includes('login')) {
    reject(new Error('登录已过期'))    // ← 🔴 BUG！丢弃了后端返回的真实错误消息
    return
  }
  // ... 静默重登逻辑
}
```

### 3.3 错误链路分析

| 场景 | 后端 HTTP | 后端 message | 后端 code | 前端实际显示 | 是否正确 |
|------|----------|-------------|----------|-------------|---------|
| 用户名或密码错误 | 401 | `用户名或密码错误` | `AUTH_FAILED` | ❌ `登录已过期` | **错误** |
| 用户不存在 | 401 | `用户名或密码错误` | `AUTH_FAILED` | ❌ `登录已过期` | **错误** |
| 租户编码不存在 | 400 | `租户编码不存在` | `INVALID_TENANT` | ✅ `租户编码不存在` | 正确 |
| 缺少租户编码 | 400 | `SaaS模式下必须提供租户编码才能登录` | `TENANT_REQUIRED` | ✅ 正确显示 | 正确 |
| 用户名密码为空 | 400 | `用户名和密码不能为空` | `INVALID_PARAMS` | ✅ 正确显示 | 正确 |

**结论**：HTTP 400 的业务错误前端能正确显示，但 HTTP 401 的认证错误**全部被 `'登录已过期'` 覆盖**。

### 3.4 后端错误返回详情

**代码位置**: [`backend/src/routes/mobile/auth.ts`](file:///d:/kaifa/CRM - 1.8.0/backend/src/routes/mobile/auth.ts)

后端 login 接口对各类错误有完善的返回：

```typescript
// 用户名/密码为空 — HTTP 400
if (!username || !password) {
  res.status(400).json({ message: '用户名和密码不能为空', code: 'INVALID_PARAMS' })
}

// 租户编码不存在 — HTTP 400
if (tenantRows.length === 0) {
  res.status(400).json({ message: '租户编码不存在', code: 'INVALID_TENANT' })
}

// SaaS模式缺少租户 — HTTP 400
if (deployConfig.isSaaS() && !tenantId) {
  res.status(400).json({ message: 'SaaS模式下必须提供租户编码才能登录', code: 'TENANT_REQUIRED' })
}

// 用户不存在 — HTTP 401
if (users.length === 0) {
  res.status(401).json({ message: '用户名或密码错误', code: 'AUTH_FAILED' })
}

// 密码错误 — HTTP 401
if (!isValidPassword) {
  res.status(401).json({ message: '用户名或密码错误', code: 'AUTH_FAILED' })
}
```

### 3.5 修复方案

#### 方案 A（推荐，1 行改动）：保留后端错误消息

修改 `request.ts` 中登录页的 401 处理，不覆盖后端返回的消息：

```typescript
// 修复前：
if (currentPath.includes('login')) {
  reject(new Error('登录已过期'))
  return
}

// 修复后：
if (currentPath.includes('login')) {
  // 登录页的 401 是认证失败（密码错误/无用户），而非 token 过期
  // 直接使用后端返回的错误信息
  const errorMsg = data.message || '登录失败，请检查用户名和密码'
  reject(new Error(errorMsg))
  return
}
```

#### 方案 B（锦上添花）：后端细分错误类型

可选优化，将用户不存在和密码错误分开提示：

```typescript
// 当前：统一返回
message: '用户名或密码错误'

// 建议：
// 用户不存在 → message: '该用户不存在，请检查用户名'
// 密码错误   → message: '密码错误，请重试'
```

> **注意**：出于安全考虑，有些系统故意不区分"用户不存在"和"密码错误"以防止用户名枚举攻击。如果租户租客场景下不存在此风险，可以区分。
