### 3.2 审查中发现并已修复的问题

1. **`setSilenceCall` API级别**：该方法为 API 29+，插件 minSdk 24 下编译期虽可通过（compileSdk 33）但存在低版本运行风险 → 移除该调用（默认即不静音）。
2. **ASI 隐患**：`getCallScreeningStatus`/`requestCallScreeningRole` 中 `return` 后跟不可达代码触发 esbuild 警告 → 改用 `#ifndef APP-PLUS` 条件编译，警告清零。
3. **本地插件目录位置**：CLI 项目的 `nativeplugins` 必须与 `manifest.json` 同级（`src/` 下），初版放在项目根目录 → 已移动到 `crmAPP/src/nativeplugins/`。
4. **`app-settings:` scheme**：设置页"去授权"用的是 iOS scheme，Android 无效 → 改为品牌授权链。
5. **重复路径**：`/录音/通话录音/` 在路径表中重复声明 → 去重。
6. **HBuilderX 打包报"插件在nativePlugins目录下不存在"**：HBuilderX 的查找位置取决于导入的是哪个目录——导入 `dist/build/app` 时找 `dist/build/app/nativeplugins`，直接导入 `crmAPP` 项目根时找 `crmAPP/nativeplugins`（项目根）。而 CLI 构建不会自动携带插件目录 → 新增 `scripts/copy-nativeplugins.js` 并挂入 `build:app`，构建后从源头 `src/nativeplugins` 自动同步到**项目根目录 + dist/build/app + dist/dev/app** 三处（根目录与 dist 下的均为自动生成副本，修改插件只改 `src/nativeplugins`），已验证三处目录及 AAR 均存在。
7. **AAR 编译依赖问题（架构重构）**：原设计含 UniModule 插件模块，编译需 1GB+ 的 uni-app 离线SDK → 重构为"Service 为主"架构：号码传递走 SharedPreferences + 应用内广播（`包名.CALL_SCREENED`，setPackage 限定本应用），角色检查/请求由 JS 经 `plus.android` 直调 `RoleManager`。已在本机用 `javac + android.jar(API 33)` 完成编译并打包出 `crm-callscreening-release.aar`，`npm run build:app` 后自动随 dist 携带。JS 侧通过 `importClass` 服务类是否成功判断插件是否打进APK，未集成时自动回退 CallLog 旧方案。
8. **HBuilderX 报"plugins节点下存在为空的对象"**：`plugins: []` 空数组无法通过 HBuilderX 描述文件校验 → 补回一个极简 UniModule 模块 `CallScreeningModule`（仅两个无参同步方法，读服务的静态缓存，不持有Context）。编译采用**桩类方案**脱离离线SDK：`native-src/CRM-CallScreening/stubs/` 下声明与官方签名一致的 `UniModule`/`UniJSMethod` 空实现，仅参与编译、不打进 AAR（classes.jar 只收录 `com/xianhu/**`），运行时按全限定名解析到基座内真实类，二进制兼容。已验证 AAR 内含 Service+Module 两个class，三处 nativeplugins 目录（项目根/src/dist）内容一致。

**B. 录音检测不到/不上传（荣耀实测反馈）修复的5个缺陷（`recordingService.ts` + `incomingCallService.ts`）**

1. **MediaStore 查询参数构造缺陷（关键根因）**：`plus.android.newObject('java.lang.String[]', ...)` 构造查询参数在部分环境不可靠，导致 MediaStore 兜底扫描/严格检测静默失败 → 时间戳改为直接内联进 selection（纯数字无注入风险），去掉参数数组。受影响的三处查询全部修复。
2. **MediaStore 只在 File 扫描为空时才兜底**：荣耀上 File API 可能扫到部分公开目录（非空）但读不到受保护的 `Sounds/CallRecord` → 改为**始终合并** MediaStore 索引结果。
3. **`processRecordingAsync` 无 callId 直接跳过**：来电号码未及时上报服务器时 `info.callId` 为空，录音处理整体跳过 → 传入与通话结束上报一致的本地兜底 callId（`IN-local-开始时间`），录音仍可关联上传。
4. **duration=0 直接跳过**：部分ROM拦截 OFFHOOK 状态导致测不到接通，实际已通话有录音 → 改为按"响铃开始→结束"时间窗口查找，找到录音说明确实接通，照常上传并纠正记录为已接通。
5. **Android 11~12 权限检查 Promise 挂起**：旧实现依赖弹窗回调 resolve，后台场景弹窗不显示 → Promise 永久挂起、录音永不处理 → 改为非阻塞（引导走24小时节流弹窗，立即按当前权限状态返回，READ_EXTERNAL_STORAGE 授予时 MediaStore 兜底可用）。

**附带加固（防串号）**：录音匹配的时间窗口从"加分项"改为**硬性条件**（窗口外文件一律不参与匹配，防止同一客户上次通话的旧录音被误认）；窗口后沿从+30秒放宽到+90秒（适配落盘/媒体库收录滞后）；号码未知时不再用空字符串参与文件名匹配（原先恒真加分）。

## 五、已知边界与风险提示

1. **来电识别角色是排他的**：若用户手机已把"来电显示与骚扰拦截"授予腾讯手机管家等应用，授权给本APP会顶掉原应用的骚扰拦截，引导文案已注明"不会拦截或影响接听"。
2. **Android 9 及以下**：CallScreeningService 角色机制不可用（`supported=false`），自动回退 CallLog 方案，设置页显示"系统不支持"。
3. **品牌权限页组件名随ROM版本变动**：跳转失败会自动逐级降级（品牌页→详情页→文案），不会出现点了没反应。
4. **系统设置键检测**：各厂商键名未公开文档化，采集自社区与实测，检测为"或"逻辑且只增不减，不会把已开启误判为未开启的概率高于误报（检测不到时还有目录/文件/MediaStore四级兜底）。
5. **AAR 重新编译**：修改 `native-src` 下 Java 源码后需重新编译 AAR 并覆盖到 `src/nativeplugins/.../android/`。因架构已去除 UniModule 依赖，用 `javac + android.jar` 即可（步骤见 README），无需下载 uni 离线SDK，也无需 Android Studio。
