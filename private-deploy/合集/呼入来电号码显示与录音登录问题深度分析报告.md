## 一、问题1：呼入响铃时不显示来电号码

### 1.1 问题描述

- **现象**：客户呼入工作手机，响铃期间来电提醒弹窗和APP页面显示"未知来电"/"未知客户"，无法显示来电号码
- **设备**：荣耀手机 → 换到OPPO手机，均复现
- **环境**：系统最新版本 + 正式打包应用
- **对比**：挂断后号码能自动补全显示
- **之前状态**：之前能获取到号码显示，但有时会串号

### 1.3 来电号码获取的多通道架构

当前代码设计了**5层数据获取通道**，按优先级排列：

```
通道1: PhoneStateListener.onCallStateChanged(state, phoneNumber)
       ├── Android 9及以下: RINGING时直接返回号码 ✅
       ├── Android 10-11: LISTEN_CALL_STATE(32) 号码被系统清空 ❌
       └── Android 12+: LISTEN_PHONE_NUMBERS(512) 部分ROM有效 ⚠️

通道2: PHONE_STATE 广播接收器（Activity + ApplicationContext双注册）
       ├── 第一条广播: state=RINGING, 无 incoming_number
       └── 第二条广播: 携带 incoming_number（需READ_CALL_LOG权限）
           ├── 国产ROM可能直接投递号码 ✅（部分设备）
           └── ROM AppOps拦截，不投递号码 ❌（OPPO/Honor新系统）

通道3: CallLog ContentObserver（监听通话记录变化）
       └── 大部分ROM在通话结束(IDLE)后才写入CallLog ⚠️
           响铃期间CallLog无新记录 → 无法获取号码

通道4: CallLog 轮询（每300ms查询一次，最多36秒）
       └── 同通道3，响铃期间CallLog无记录 → 无法获取号码 ❌

通道5: 录音文件名解析（仅通话结束后）
       └── 响铃期间禁用（防串号），仅用于挂断后补号 ✅
```

### 1.4 根因分析

#### 根因A：Android 10+ PhoneStateListener 号码限制

**代码位置**：`incomingCallService.ts` 第338-365行 `initAndroidListener()`

```typescript
// 关键代码：
const PhoneStateListener = plus.android.implements('android.telephony.PhoneStateListener', {
  onCallStateChanged: function() {
    const args = Array.prototype.slice.call(arguments)
    const state = Number(args[0]) || 0
    const rawNumber = args[1]  // ← Android 10+ 此参数为空
    const pn = rawNumber ? ('' + rawNumber).trim() : ''
    self.handleCallStateChange(state, pn)
  },
})

let listenFlags = 32  // LISTEN_CALL_STATE
if (sdkInt >= 31) {
  listenFlags = 544 // 32 | 512 (LISTEN_PHONE_NUMBERS)
}
```

**分析**：
- Android 10 (API 29) 起，Google在`PhoneStateListener.onCallStateChanged(int state, String phoneNumber)`中**不再传递phoneNumber参数**，无论是否有`READ_CALL_LOG`权限
- Android 12+ (API 31) 新增`LISTEN_PHONE_NUMBERS`(512)标志，通过`onCallStateChanged(int state)`回调配合`TelephonyManager.getLine1Number()`获取号码，但此API获取的是**本机号码**而非来电号码
- `LISTEN_PHONE_NUMBERS`在大多数国产ROM上**不生效**，OPPO ColorOS和荣耀HarmonyOS均不传递来电号码
- **结论**：通道1在Android 10+上无法获取来电号码，这是平台限制，非代码Bug

#### 根因B：PHONE_STATE广播被ROM拦截

**代码位置**：`incomingCallService.ts` 第690-740行 `handlePhoneStateBroadcastIntent()`

```typescript
private handlePhoneStateBroadcastIntent(intent: any) {
  const state = intent.getStringExtra('state')
  const incomingNumber = intent.getStringExtra('incoming_number')
  // ↑ Android 9+系统会发两条广播：
  //   第一条: state=RINGING, incoming_number=null
  //   第二条: state=RINGING/OFFHOOK, incoming_number=实际号码
  //   但第二条需要READ_CALL_LOG权限且不被AppOps拦截
}
```

**分析**：
- Android 9+ (API 28) 起，`PHONE_STATE`广播的`incoming_number`字段需要`READ_CALL_LOG`权限才能接收
- OPPO ColorOS 13/14和荣耀HarmonyOS 4.x的AppOps层会**静默拦截**通话记录读取，即使运行时权限已授予
- 代码已实现了Activity Context + ApplicationContext双重广播注册（`registerPhoneStateReceiver` + `registerAltPhoneStateReceiver`），但**两条带号码的广播都被ROM层拦截**
- 代码已有`diagnoseCallLogAccess()`深度诊断方法，能区分三种根因：APK未含权限/运行时未授权/AppOps拦截
- **结论**：通道2在OPPO/荣耀新系统上被ROM层AppOps拦截，非代码Bug

#### 根因C：CallLog写入时机问题（核心原因）

**代码位置**：`incomingCallService.ts` 第980-1010行 `quickResolveAndReport()` 和 `startCallLogPolling()`

```typescript
private async quickResolveAndReport() {
  // 立即用"未知来电"启动流程，确保弹窗不延迟
  this.onRingingDetected('未知来电')
  // 启动 CallLog 轮询持续尝试获取真实号码
  this.startCallLogPolling()
}

private startCallLogPolling() {
  // 每 300ms 查询一次通话记录，直到获取到号码或超时
  // maxAttempts = 120 → 最多轮询36秒
  // 但大部分ROM在IDLE(挂断)后才写入CallLog！
}
```

**分析**：
- **这是导致"响铃时不显示号码、挂断后补全"的直接原因**
- Android系统的CallLog（通话记录数据库）由Dialer应用（拨号器）在**通话状态变为IDLE后**才写入，而非响铃时实时写入
- 代码在`handleCallStateChange`的`case 0: // IDLE`分支中，当号码仍未知时会延迟1秒再处理结束流程，等待CallLog写入
- 在`onIncomingCallEnded()`中有递增延迟补号逻辑（500ms→1000ms→...→5000ms），最终能从CallLog或录音文件名获取号码
- **这解释了为什么"挂断就能自动补全看到号码"**

#### 根因D：标准基座 vs 自定义基座问题

**代码位置**：`incomingCallService.ts` 第542-548行 `diagnoseCallLogAccess()`

### 1.5 为什么"之前能做到"——历史回顾与串号分析

#### 之前能获取号码的原因

之前能在响铃时获取号码，可能依赖以下任一条件：

1. **旧版Android系统**：Android 9及以下，`PhoneStateListener.onCallStateChanged`仍会传递来电号码
2. **旧版ROM**：旧版荣耀Magic UI / EMUI的PHONE_STATE广播第二条会投递`incoming_number`，不受AppOps限制
3. **旧版OPPO ColorOS**：ColorOS 12及以下可能对`READ_CALL_LOG`的AppOps管控较松

#### 串号原因分析

之前"有时候能正确匹配显示，有时候会串号"，根本原因是：

**场景：快速连续来电**

```
时间线：
T0: 客户A来电 → 响铃 → RINGING无号码 → 报"未知来电" → CallLog轮询启动
T1: 用户拒接/挂断 → IDLE → CallLog轮询继续（IDLE后还查20次≈6秒）
T2: CallLog终于写入客户A的号码 → 轮询获取到号码A
T3: 客户B来电 → 新的RINGING → 此时轮询的号码A被赋给客户B的会话 → 串号！
```

**代码已有的防串号措施**（说明之前已修复过此问题）：

| 防护措施 | 代码位置 | 说明 |
|---------|---------|------|
| 时间窗口校验 | `extractNumberFromCursor()` 第2230-2250行 | CallLog记录时间必须 > 来电开始时间-5秒 |
| 会话句柄比对 | `handleCallLogChange()` / `onIncomingCallEnded()` | 补号期间检测`this.currentIncoming !== session`则中止 |
| 录音文件时间戳校验 | `tryResolveCallerFromRecordingFile()` | 文件名中的14位时间戳必须在本通窗口内 |
| 响铃态禁用录音解析 | `tryResolveNumber()` | `if (this.lastPhoneState !== 1)` 才尝试录音文件 |
| 后端120秒窗口+号码比对 | `MobileWebSocketService.ts` 第700-720行 | 仅复用号码未知或号码相同的响铃记录 |
| IDLE后补号新来电检测 | `backgroundResolveNumber()` | `if (this.currentIncoming) return` 中止补号 |

**串号根因**：尽管有上述防护，在极端情况下（ROM延迟写入CallLog、快速连续来电），仍可能出现时间窗口重叠导致匹配到错误号码。当前代码的防护已经相当完善，串号问题在最新代码中已大幅缓解。

### 1.6 可修复方案

#### 方案1：接入CallScreeningService原生插件（推荐，Android 10+最佳方案，彻底解决）

**原理**：Android 10 (API 29) 引入`CallScreeningService`，它是Telecom框架的一部分。当有来电时，系统会在**响铃的第一时间**主动调用该Service的`onScreenCall(Call.Details)`方法，通过`Call.Details.getHandle()`可直接获取来电号码的Uri（`tel:13812345678`），**不依赖CallLog、不受AppOps静默拦截**。

**为什么能响铃时拿到号码**：

```
传统方式（当前代码）：
  来电 → 系统Dialer应用响铃 → PhoneStateListener(无号码) / 广播(被拦截) / CallLog(未写入)
  → 响铃期间所有通道均失败 ❌

CallScreeningService方式：
  来电 → Telecom框架 → 调用 CallScreeningService.onScreenCall(Call.Details)
  → Call.Details.getHandle() = tel:13812345678 → 立即拿到号码 ✅
  → （之后系统Dialer应用才响铃）
```

CallScreeningService的回调时机**早于或等于系统拨号器响铃**，因为它的设计目的就是让第三方应用在来电时做筛选决策（接听/拒接/静音）。系统会将来电号码通过`Call.Details`对象传递给Service，无需查询CallLog。

**优势**：
- Android 10+官方推荐的来电筛选API，由Telecom框架直接调用
- 响铃时即可获取号码，不依赖CallLog的写入时机
- 不受AppOps静默拦截限制（仅需`READ_CALL_LOG`运行时权限）
- OPPO ColorOS / 荣耀HarmonyOS / 小米MIUI均支持（系统级API，非厂商私有）
- 可同时实现来电拦截、静音等功能（如需）

**限制**：
- 仅支持Android 10+ (API 29+)，Android 9及以下需回退到现有方案
- 必须通过AndroidManifest注册Service组件，**uni-app的`plus.android.implements`无法实现**
- 需要编写uni-app原生插件（Android原生Java/Kotlin代码）
- 用户需在系统设置中将本APP设为"来电筛选应用"（`ROLE_CALL_SCREENING`）

##### 第二步：编写CallScreeningService（Java原生代码）

`CRMCallScreeningService.java`：

```java
package com.xianhu.crm;

import android.telecom.Call;
import android.telecom.CallScreeningService;
import android.net.Uri;
import android.util.Log;

/**
 * 来电筛选服务 — 响铃时由Telecom框架调用，直接获取来电号码
 * 
 * 注意：此Service必须在AndroidManifest.xml中注册，
 * 且APP需要被设为"来电筛选应用"（通过RoleManager请求ROLE_CALL_SCREENING）
 */
public class CRMCallScreeningService extends CallScreeningService {

    private static final String TAG = "CRMCallScreening";
    // 静态变量：供uni-app插件读取最新来电号码
    public static volatile String lastIncomingNumber = "";
    public static volatile long lastIncomingTime = 0;
    // 回调接口：供插件设置，Service触发时回调到JS层
    public static volatile OnIncomingCallListener listener = null;

    public interface OnIncomingCallListener {
        void onIncomingCall(String phoneNumber, long timestamp);
    }

    @Override
    public void onScreenCall(Call.Details callDetails) {
        // 获取来电号码（Uri格式：tel:13812345678）
        Uri handle = callDetails.getHandle();
        String phoneNumber = "";
        if (handle != null) {
            phoneNumber = handle.getSchemeSpecificPart(); // 提取号码部分
        }

        long timestamp = System.currentTimeMillis();
        Log.i(TAG, "onScreenCall: 响铃获取到来电号码 = " + phoneNumber);

        // 保存到静态变量（供轮询方式读取）
        lastIncomingNumber = phoneNumber;
        lastIncomingTime = timestamp;

        // 通过回调通知JS层（如果已设置）
        if (listener != null) {
            listener.onIncomingCall(phoneNumber, timestamp);
        }

        // 允许来电响铃（不做拦截）
        // CallResponse.Builder 必须调用，否则系统认为不允许来电
        respondToCall(callDetails, new CallResponse.Builder()
                .setDisallowCall(false)      // 允许来电
                .setRejectCall(false)        // 不拒接
                .setSilenceCall(false)       // 不静音
                .setSkipCallLog(false)       // 不跳过通话记录
                .setSkipNotification(false)  // 不跳过通知
                .build());
    }
}
```

##### 第四步：编写插件入口（CallScreeningPlugin.java）

```java
package com.xianhu.crm;

import android.app.role.RoleManager;
import android.content.Context;
import android.content.Intent;
import android.os.Build;
import android.util.Log;

import com.alibaba.fastjson.JSONObject;

import io.dcloud.feature.uniapp.annotation.UniJSMethod;
import io.dcloud.feature.uniapp.common.UniModule;

/**
 * uni-app原生插件入口
 * 在JS中通过 uni.requireNativePlugin('CRM-CallScreening') 调用
 */
public class CallScreeningPlugin extends UniModule {

    private static final String TAG = "CRM-CallScreening-Plugin";

    /**
     * 检查是否支持CallScreeningService（Android 10+）
     */
    @UniJSMethod(uiThread = true)
    public void isSupported(JSONObject options, com.alibaba.fastjson.JSONObject callback) {
        boolean supported = Build.VERSION.SDK_INT >= Build.VERSION_CODES.Q; // API 29
        JSONObject result = new JSONObject();
        result.put("supported", supported);
        result.put("sdkInt", Build.VERSION.SDK_INT);
        callback(result);
    }

    /**
     * 请求"来电筛选应用"角色（系统会弹窗让用户确认）
     * Android 10+必需，否则Service不会被调用
     */
    @UniJSMethod(uiThread = true)
    public void requestCallScreeningRole(JSONObject options, final com.alibaba.fastjson.JSONObject callback) {
        try {
            Context context = mUniSDKInstance.getContext();
            if (Build.VERSION.SDK_INT < Build.VERSION_CODES.Q) {
                JSONObject result = new JSONObject();
                result.put("granted", false);
                result.put("reason", "unsupported");
                callback.invoke(result);
                return;
            }

            RoleManager roleManager = (RoleManager) context.getSystemService(Context.ROLE_SERVICE);
            if (roleManager == null) {
                JSONObject result = new JSONObject();
                result.put("granted", false);
                result.put("reason", "no_role_manager");
                callback.invoke(result);
                return;
            }

            // 检查是否已获得角色
            if (roleManager.isRoleAvailable(RoleManager.ROLE_CALL_SCREENING) &&
                roleManager.isRoleHeld(RoleManager.ROLE_CALL_SCREENING)) {
                JSONObject result = new JSONObject();
                result.put("granted", true);
                callback.invoke(result);
                return;
            }

            // 请求角色（系统弹窗）
            Intent intent = roleManager.createRequestRoleIntent(RoleManager.ROLE_CALL_SCREENING);
            mUniSDKInstance.startActivityForResult(intent, new com.alibaba.fastjson.JSONObject(),
                new com.alibaba.fastjson.JSONObject() {
                    // uni-app的Activity回调机制
                });

            // 简化：延迟检查结果
            new android.os.Handler().postDelayed(() -> {
                RoleManager rm = (RoleManager) mUniSDKInstance.getContext()
                    .getSystemService(Context.ROLE_SERVICE);
                JSONObject result = new JSONObject();
                result.put("granted", rm != null && rm.isRoleHeld(RoleManager.ROLE_CALL_SCREENING));
                callback.invoke(result);
            }, 3000);
        } catch (Exception e) {
            Log.e(TAG, "requestCallScreeningRole error", e);
            JSONObject result = new JSONObject();
            result.put("granted", false);
            result.put("reason", e.getMessage());
            callback.invoke(result);
        }
    }

    /**
     * 注册来电号码回调（JS层设置回调函数）
     * Service触发onScreenCall时，通过此回调将号码传给JS
     */
    @UniJSMethod(uiThread = true)
    public void registerCallback(JSONObject options, final com.alibaba.fastjson.JSONObject callback) {
        CRMCallScreeningService.listener = new CRMCallScreeningService.OnIncomingCallListener() {
            @Override
            public void onIncomingCall(String phoneNumber, long timestamp) {
                JSONObject result = new JSONObject();
                result.put("phoneNumber", phoneNumber);
                result.put("timestamp", timestamp);
                // 回调到JS层
                callback.invokeAndKeepAlive(result);
            }
        };
        Log.i(TAG, "来电号码回调已注册");
    }

    /**
     * 获取最近一次来电号码（轮询方式，作为回调的补充）
     */
    @UniJSMethod(uiThread = true)
    public void getLastIncomingNumber(JSONObject options, com.alibaba.fastjson.JSONObject callback) {
        JSONObject result = new JSONObject();
        result.put("phoneNumber", CRMCallScreeningService.lastIncomingNumber);
        result.put("timestamp", CRMCallScreeningService.lastIncomingTime);
        callback.invoke(result);
    }
}
```

```typescript
/**
 * 轮询方式获取CallScreening号码（作为回调的补充，防止回调丢失）

private pollCallScreeningNumber() {
  // #ifdef APP-PLUS
  if (!this.callScreeningSupported || !this.callScreeningPlugin) return
  let pollCount = 0
  const pollTimer = setInterval(() => {
    pollCount++
    if (!this.currentIncoming || (this.currentIncoming.callerNumber &&
        this.currentIncoming.callerNumber !== '未知来电')) {
      clearInterval(pollTimer)
      return
    }
    this.callScreeningPlugin.getLastIncomingNumber({}, (res: any) => {
      if (res.phoneNumber && res.phoneNumber.length >= 3 &&
          res.timestamp > (this.currentIncoming?.startTime || 0) - 5000) {
        console.log('[IncomingCallService] ✅ CallScreening轮询获取号码:', res.phoneNumber)
        clearInterval(pollTimer)
        if (this.currentIncoming && this.currentIncoming.callerNumber === '未知来电') {
          this.currentIncoming.callerNumber = res.phoneNumber
          this.hasReportedIncoming = false
          this.currentIncoming.callId = undefined
          this.reportIncomingToServer(res.phoneNumber)
          uni.$emit('incoming:number_updated', { callerNumber: res.phoneNumber })
        }
      }
    })
    if (pollCount > 60) clearInterval(pollTimer) // 30秒超时
  }, 500)
  // #endif
}
```

### 2.3 OPPO录音检测的5层策略及失效分析

当前`checkRecordingEnabled()`方法使用5层检测策略：

```
策略1: 读取系统Settings键值 → ❌ OPPO不使用标准Settings存储
策略2: 通话录音专属目录 + 音频文件 → ⚠️ 路径可能不匹配
策略3: 通用目录中查找通话录音文件 → ⚠️ 命名规则可能不匹配
策略4: MediaStore精确查询 → ⚠️ OPPO可能不索引通话录音
策略5: 动态发现目录 + 文件检查 → ⚠️ 受限于扫描深度
```

#### 失效点1：系统Settings检测（策略1）完全失效

**代码位置**：`recordingService.ts` 第1420-1480行 `checkSystemRecordingSetting()`

```typescript
// 尝试读取的OPPO相关键值：
'oppo_call_record',
'oppo_auto_call_recording',
'oplus_call_recording',
```

**分析**：
- OPPO ColorOS的通话录音开关**不存储在`Settings.System`/`Settings.Secure`/`Settings.Global`中**
- OPPO将录音设置存储在**拨号应用（com.coloros.dialer）的私有SharedPreferences**中
- 第三方应用**无法读取**其他应用的私有SharedPreferences
- **结论**：策略1对OPPO完全无效，且无法通过代码修复（系统限制）

#### 失效点2：录音文件路径不匹配（策略2）

**代码位置**：`recordingService.ts` 第40-42行

```typescript
// OPPO专属路径声明：
'/storage/emulated/0/Recordings/Call Recordings/',
'/storage/emulated/0/Music/Recordings/Call Recordings/',
```

**分析**：
- OPPO ColorOS 13/14的实际录音保存路径可能为：
  - `/storage/emulated/0/Recordings/Call Recordings/`（带空格）← 已覆盖
  - `/storage/emulated/0/Recordings/CallRecordings/`（无空格）← **未覆盖**
  - `/storage/emulated/0/Record/Call/` ← 已在GENERIC_SCAN_PATHS中
  - `/storage/emulated/0/Music/Recordings/Call Recordings/` ← 已覆盖
  - **OPPO私有存储**：`/data/user/0/com.coloros.dialer/files/` ← **不可访问**
- OPPO ColorOS 14可能将录音保存在拨号应用的**私有目录**中，第三方应用即使有`MANAGE_EXTERNAL_STORAGE`权限也无法读取
- **缺失路径**：`/storage/emulated/0/Recordings/CallRecordings/`（无空格变体）

#### 失效点3：文件命名规则不匹配（策略3/5）

**代码位置**：`recordingService.ts` 第1250-1280行 `isCallRecordingFileName()`


**分析**：
- OPPO ColorOS的录音文件命名格式可能为：
  - `录音_2026_07_09_14_30_22.m4a`（中文"录音" + 时间，无号码）→ **不匹配**（"录音"后需有数字才匹配，但格式可能不含连续11位号码）
  - `CallRecording_13812345678_20260709143022.m4a` → 可匹配
  - `20260709_143022_13812345678.m4a` → 可匹配（含手机号）
- 如果文件名中不含手机号、不含"call"、不含"通话"，则策略3/5均不匹配

#### 失效点4：录音上传的文件匹配逻辑

## 三、问题3：APP登录密码错误/无用户/无租户等提示不全

### 3.1 问题描述

- **现象**：APP登录密码错误时没有正确提示
- **需求**：补全登录遇到问题的提示（密码错误、无用户、无租户等）

### 3.2 涉及代码文件

| 文件 | 作用 |
|------|------|
| `crmAPP/src/pages/login/index.vue` | 登录页面（表单+登录逻辑） |
| `crmAPP/src/utils/request.ts` | HTTP请求封装（错误处理） |
| `crmAPP/src/api/auth.ts` | 登录API定义 |
| `backend/src/routes/mobile/auth.ts` | 后端登录接口 |

### 3.3 后端错误返回分析

**后端登录接口** `POST /mobile/login` 的错误返回：

| 场景 | HTTP状态码 | message | code |
|------|-----------|---------|------|
| 用户名/密码为空 | 400 | "用户名和密码不能为空" | INVALID_PARAMS |
| 租户编码不存在 | 400 | "租户编码不存在" | INVALID_TENANT |
| SaaS模式无租户编码 | 400 | "SaaS模式下必须提供租户编码才能登录" | TENANT_REQUIRED |
| 用户不存在 | 401 | "用户名或密码错误" | AUTH_FAILED |
| 密码错误 | 401 | "用户名或密码错误" | AUTH_FAILED |
| 服务器异常 | 500 | "登录失败" | SERVER_ERROR |

**注意**：后端对"用户不存在"和"密码错误"返回相同的消息"用户名或密码错误"，这是**安全最佳实践**（防止枚举用户名攻击）。

### 3.4 根因分析

#### 根因A：request.ts的401拦截导致登录错误消息被覆盖（核心Bug）

**代码位置**：`request.ts` 第158-185行

```typescript
// request.ts 中的401处理：
if (res.statusCode === 401) {
  const pages = getCurrentPages()
  const currentPage = pages[pages.length - 1]
  const currentPath = currentPage?.route || ''

  if (currentPath.includes('login')) {
    reject(new Error('登录已过期'))  // ← BUG！密码错误时显示"登录已过期"
    return
  }
  // ... 尝试静默重登 ...
}
```

**分析**：
- 后端登录失败返回**HTTP 401**（用户不存在/密码错误）
- `request.ts`的通用401处理逻辑**拦截了登录接口的401响应**
- 当用户在登录页面时，401被错误地当作"token过期"处理，返回"登录已过期"
- **用户看到的是"登录已过期"而非"用户名或密码错误"**
- 这是最核心的Bug

#### 根因B：双重Toast显示导致消息闪烁/覆盖

**代码位置**：`request.ts` 第194-199行 + `login/index.vue` 第154-158行

```typescript
// request.ts 中（非401的其他错误）：
const errorMsg = data.message || '请求失败'
uni.showToast({ title: errorMsg, icon: 'none', duration: 2000 })  // ← 第1个Toast
reject(new Error(errorMsg))

// login/index.vue 的 catch 中：
} catch (e: any) {
  uni.showToast({ title: e.message || '登录失败', icon: 'none' })  // ← 第2个Toast
}
```

**分析**：
- 对于400错误（参数为空/租户不存在/无租户），`request.ts`先显示一个Toast
- 然后`reject`后，`login/index.vue`的catch块又显示一个Toast
- 两个Toast可能**互相覆盖**，用户只看到最后一个（或闪烁）
- 对于401错误（密码错误），`request.ts`不显示Toast直接reject，但catch显示的是"登录已过期"

#### 根因C：网络错误提示不友好

**代码位置**：`request.ts` 第211-218行

```typescript
fail: (err) => {
  uni.hideLoading()
  uni.showToast({ title: '网络连接失败', icon: 'none', duration: 2000 })  // ← 通用提示
  reject(new Error(err.errMsg || '网络错误'))
}
```

**分析**：
- 服务器无法连接时显示"网络连接失败"，不区分DNS错误、超时、拒绝连接等
- 登录页面catch会再显示一次`err.errMsg`（可能是"request:fail timeout"等技术信息）

### 3.5 可修复方案

#### 方案1：修复request.ts的401拦截逻辑（P0，核心修复）

```typescript
// request.ts 中，对 /mobile/login 接口跳过401的token过期处理：
if (res.statusCode === 401) {
  // 登录接口的401是认证失败（密码错误/用户不存在），不应触发token续期逻辑
  if (options.url.includes('/mobile/login')) {
    const errorMsg = data.message || '用户名或密码错误'
    reject(new Error(errorMsg))
    return
  }
  // ... 原有401处理逻辑 ...
}
```

#### 方案2：统一错误提示，避免双重Toast（P0）

```typescript
// 方案A：request.ts 增加参数控制是否显示Toast
interface RequestOptions {
  // ...原有字段...
  silentError?: boolean  // 为true时request不自动显示Toast，由调用方处理
}

// login/index.vue 调用时：
const result = await login(loginData)  // login函数内部传 silentError: true

// login/index.vue 的catch中统一显示错误提示：
} catch (e: any) {
  uni.showToast({ title: e.message || '登录失败', icon: 'none', duration: 3000 })
}
```

#### 方案3：后端区分错误类型（P1，可选）

如果需要更精确的提示（区分"用户不存在"和"密码错误"），可修改后端返回：

```typescript
// backend/src/routes/mobile/auth.ts

// 用户不存在时：
return res.status(401).json({
  success: false,
  message: '用户不存在，请检查用户名或租户编码',
  code: 'USER_NOT_FOUND'
})

// 密码错误时：
return res.status(401).json({
  success: false,
  message: '密码错误，请重新输入',
  code: 'PASSWORD_ERROR'
})
```

**安全提示**：区分"用户不存在"和"密码错误"会降低系统安全性（攻击者可枚举用户名），建议仅在内部部署、信任网络环境下使用。SaaS模式下建议保持统一提示。

#### 方案4：前端根据code显示不同提示（P1，推荐）

不改后端，在前端根据后端返回的`code`字段显示更友好的提示：

```typescript
// login/index.vue 的catch中：
} catch (e: any) {
  let message = e.message || '登录失败'
  // 根据后端返回的code进一步细化提示
  if (message.includes('租户编码不存在')) {
    message = '租户编码不存在，请检查后重新输入'
  } else if (message.includes('必须提供租户编码')) {
    message = '请输入租户编码'
  } else if (message.includes('用户名或密码错误')) {
    message = '用户名或密码错误，请重新输入'
  } else if (message.includes('登录已过期')) {
    message = '用户名或密码错误'  // 修正被request.ts覆盖的消息
  }
  uni.showToast({ title: message, icon: 'none', duration: 3000 })
}
```

### 5.1 来电号码获取流程图（当前实现）

```
客户来电
  │
  ├─ PhoneStateListener RINGING ──→ 号码为空(Android 10+)
  │     │
  │     └─ quickResolveAndReport()
  │         ├─ onRingingDetected('未知来电') → 弹窗显示"未知来电"
  │         └─ startCallLogPolling() → 每300ms查CallLog
  │               │
  │               ├─ 响铃期间CallLog无记录 → 持续失败
  │               └─ IDLE后CallLog写入 → 获取号码 → 更新弹窗
  │
  ├─ PHONE_STATE广播 ──→ 第二条带号码？
  │     ├─ 是 → onRingingDetected(号码) → 弹窗显示号码 ✅
  │     └─ 否（ROM拦截）→ 无操作 ❌
  │
  ├─ CallLog ContentObserver ──→ 响铃期间无变化
  │     └─ IDLE后CallLog写入 → handleCallLogChange()
  │         └─ 获取号码 → 更新弹窗
  │
  └─ 通话结束(IDLE)
        ├─ onIncomingCallEnded() → 延迟补号（CallLog + 录音文件名）
        └─ backgroundResolveNumber() → 后台持续补号（最长40秒）
```
