## 一、现状总结：三层并存的迁移机制

项目目前存在 **三种不同的数据库变更方式**，混用情况严重：

| 机制 | 位置 | 触发时机 | 覆盖范围 |
|------|------|----------|----------|
| **A. 启动时自动修复脚本** | `backend/src/scripts/initOrderSettings.ts`<br>`backend/src/scripts/initSensitiveInfoPermissions.ts` | 后端启动时 `initializeDatabase()` 内调用 | 4张表、约15个字段 |
| **B. 路由级懒加载迁移** | 散落在 20+ 路由/服务文件中 | 首次访问该路由时执行 | ~25张表、~60个字段 |
| **C. 手动SQL迁移文件** | `backend/database-migrations/*.sql` (10个文件) | 仅在"一键更新"时由 `UpdateService.runMigrations()` 执行 | ~10个变更 |

TypeORM 配置关键参数：

```typescript
// backend/src/config/database.ts:194-213
synchronize: false,    // 不自动同步表结构（正确）
migrations: [],        // 空数组，未使用 TypeORM 内置迁移系统
```

## 二、详细分类：哪些表/字段依赖什么机制

### 2.1 机制A：启动时自动修复（安全，幂等）

这2个脚本在 `database.ts:initializeDatabase()` 中启动时自动执行，**检测到缺失表/字段会自动创建**：

| 脚本 | 处理的表 | 自动创建内容 |
|------|----------|-------------|
| `initOrderSettings.ts` | `department_order_limits` | 整张表（如不存在）+ `tenant_id` 列 + 联合唯一索引 |
| | `payment_method_options` | `tenant_id` 列 + `is_system` 列 + 联合唯一索引 + 6条预设数据 |
| | `system_configs` | `tenant_id` 列 + 联合唯一索引 |
| `initSensitiveInfoPermissions.ts` | `sensitive_info_permissions` | 整张表（如不存在）+ `tenant_id` 列 + 联合唯一索引 + 35条默认权限数据 |

**代码位置**：`backend/src/services/UpdateService.ts:610-720`

核心逻辑：
1. 读取 `database-migrations/*.sql` 文件列表
2. 查询 `migration_history` 表获取已执行文件
3. 对未执行的文件逐条执行SQL语句
4. 忽略 "already exists" / "Duplicate column" / "Duplicate key name" 错误
5. 记录执行结果到 `migration_history` 表

## 三、当前机制的核心问题

| # | 问题 | 严重性 | 说明 |
|---|------|--------|------|
| 1 | **迁移逻辑散落20+文件** | 高 | 难以追踪哪些表有自动迁移、哪些没有 |
| 2 | **同一表多处重复迁移** | 高 | `tenants` 表至少在4个文件中重复添加字段，`tenant_packages` 在3个文件中重复 |
| 3 | **无统一执行记录** | 高 | 机制B的路由级迁移无任何执行记录，不知道是否执行过 |
| 4 | **启动时不执行SQL迁移文件** | 中 | `database-migrations/` 下的10个SQL文件仅在一键更新时执行，新部署可能遗漏 |
| 5 | **错误静默吞掉** | 中 | 大量 `.catch(() => {})` 隐藏了潜在错误 |
| 6 | **依赖访问触发** | 中 | 未访问的路由不会触发迁移，可能导致功能异常 |
| 7 | **无回滚机制** | 中 | 所有迁移都是单向的，无down脚本 |
| 8 | **TypeORM migrations未启用** | 低 | `migrations: []` 空配置，内置迁移系统闲置 |

### 3.1 重复迁移示例

`tenants` 表的 `user_limit_mode` 字段至少在3处重复添加：

| 文件 | 行号 | 代码 |
|------|------|------|
| `routes/admin/packages.ts` | 364 | `ALTER TABLE tenants ADD COLUMN user_limit_mode ENUM('total','online')...` |
| `services/MemberService.ts` | 384 | `ALTER TABLE tenants ADD COLUMN user_limit_mode ENUM('total','online')...` |
| `services/CapacityService.ts` | (间接) | 通过 `extra_users` 等字段间接重复 |

虽然都是幂等的（`.catch(() => {})`），但每次访问都执行一次无意义的 `INFORMATION_SCHEMA` 查询。

## 四、自动化迁移方案设计

### 4.1 方案目标

将三种机制统一为一套**启动时自动检测+创建+记录**的迁移系统，同时保持与现有代码兼容。

### 4.2 方案架构

```
后端启动
  │
  ├── 1. TypeORM initialize (synchronize: false, 不变)
  │
  ├── 2. 【新增】统一迁移引擎 AutoMigrationService
  │     ├── 2a. 扫描所有 @Entity 实体 → 对比数据库 → 缺失表/字段自动创建
  │     ├── 2b. 执行 database-migrations/*.sql（原UpdateService逻辑前移到启动时）
  │     ├── 2c. 执行现有 initOrderSettings + initSensitiveInfoPermissions（保持不变）
  │     └── 2d. 记录到 migration_history 表
  │
  └── 3. 启动完成
```

### 4.3 分阶段实施

#### 阶段1：启动时执行SQL迁移文件（低风险）

**做法**：将 `UpdateService.runMigrations()` 的核心逻辑抽取为独立服务，在启动时自动执行 `database-migrations/*.sql`。

**核心逻辑**：
```
1. 确保 migration_history 表存在
2. 读取 database-migrations/*.sql 文件列表（按文件名排序）
3. 查询已执行的文件列表
4. 对未执行的文件：
   a. 读取SQL内容
   b. 按 ";" 分割语句
   c. 逐条执行
   d. 忽略 "already exists" / "Duplicate column" 错误
   e. 记录到 migration_history
```

**影响评估**：
- ✅ 现有10个SQL文件都是幂等的（已有 `already exists` 错误处理）
- ✅ `migration_history` 表已存在，有执行记录
- ⚠️ 启动时间增加约0.5-2秒（取决于待执行文件数）
- ⚠️ 需确保SQL文件不出错，否则启动失败

#### 阶段2：基于实体的自动建表（中风险）

**做法**：利用 TypeORM 的实体元数据，**仅在表不存在时创建，不修改已有表结构**。

两种实现路径：

| 路径 | 做法 | 风险 |
|------|------|------|
| **路径A：synchronize: true** | 开启TypeORM自动同步 | 🔴 高风险：会自动修改已有表列类型、删除列，可能丢数据 |
| **路径B：自定义建表** | 遍历实体元数据，仅对 `information_schema` 中不存在的表执行 `CREATE TABLE` | 🟢 低风险：只建不删不改 |

**推荐路径B**，核心逻辑：

```
for each entity in entities:
    if table not exists in information_schema:
        generate CREATE TABLE SQL from entity metadata
        execute SQL
        log "Created table: {tableName}"
```

**影响评估**：
- ✅ 只创建缺失表，不修改已有表
- ✅ 对已有数据库无任何影响
- ⚠️ 新建表的列定义可能与手动SQL略有差异（类型映射）
- ⚠️ 需处理索引、外键、字符集等

#### 阶段3：基于实体的自动补字段（较高风险）

**做法**：遍历实体元数据，对比数据库现有列，缺失的列自动 `ALTER TABLE ADD COLUMN`。

**影响评估**：
- ✅ 只添加缺失列，不删除/修改已有列
- ⚠️ 需处理列类型映射（TypeORM类型 → MySQL类型）
- ⚠️ 需处理默认值、NULL约束
- ⚠️ 可能与机制B的路由级迁移冲突（同一字段被迁移两次）
- ⚠️ 需确保幂等性

## 五、自动化迁移对当前项目的影响评估

| 影响项 | 评估 | 说明 |
|--------|------|------|
| **现有数据安全** | 🟢 安全 | 只创建不删除，不改已有列 |
| **启动时间** | 🟡 增加1-3秒 | 首次启动需建表/补字段，后续启动仅检查（<0.5秒） |
| **现有路由级迁移** | 🟡 可能冲突 | 机制B的代码仍在，会重复执行，但都是幂等的 |
| **生产环境** | 🔴 需谨慎 | 生产环境首次启用需先备份、先在测试环境验证 |
| **多租户数据** | 🟢 安全 | 不涉及数据修改，只涉及结构变更 |
| **性能** | 🟢 无影响 | 启动后无额外开销 |
| **回滚能力** | 🔴 无 | 自动创建的表/字段无回滚脚本，需手动删除 |
| **与UpdateService的冲突** | 🟡 需处理 | 启动时已执行SQL迁移，一键更新时再执行会重复（但幂等） |

### 5.1 风险矩阵

| 风险点 | 概率 | 影响 | 缓解措施 |
|--------|------|------|----------|
| SQL迁移文件语法错误 | 低 | 启动失败 | 预检SQL语法、测试环境验证 |
| 实体与数据库类型映射不一致 | 中 | 新建表结构略有差异 | 对比验证、手动修正 |
| 与路由级迁移重复执行 | 高 | 无实际影响（幂等） | 可后续逐步清理路由级迁移 |
| 生产环境首次启用异常 | 低 | 服务无法启动 | 先备份、可在环境变量中关闭 |
| 多租户表遗漏 tenant_id | 中 | 新建表缺少租户隔离 | 实体检查、自动补 tenant_id |

## 六、推荐实施路径

```
阶段1（低风险）          阶段2（中风险）           阶段3（可选）
启动时执行SQL迁移  →  自动创建缺失表  →  自动补缺失字段
       ↓                    ↓                    ↓
    1-2天               2-3天               按需实施
    先做这个           稳定后再做           风险较高谨慎
```

### 6.1 关键原则

1. **只增不删**：永远不删除表、列、索引
2. **只建不改**：永远不修改已有列的类型、约束
3. **幂等执行**：多次执行结果一致
4. **完整记录**：所有变更记录到 `migration_history`
5. **可关闭**：通过环境变量 `AUTO_MIGRATION=false` 可关闭
6. **先备份**：生产环境首次执行前自动备份当前结构

### 6.2 各阶段实施细节

#### 阶段1：启动时执行SQL迁移文件

**新增文件**：`backend/src/services/AutoMigrationService.ts`

**修改文件**：`backend/src/config/database.ts`（在 `initializeDatabase` 末尾调用）

**核心逻辑**：
```typescript
// 伪代码
async function runStartupMigrations(): Promise<void> {
  // 1. 确保 migration_history 表存在
  await ensureMigrationHistoryTable();

  // 2. 读取 database-migrations/*.sql
  const sqlFiles = fs.readdirSync(migrationDir)
    .filter(f => f.endsWith('.sql'))
    .sort();

  // 3. 获取已执行记录
  const executed = await getExecutedFiles();

  // 4. 执行未执行的文件
  for (const file of sqlFiles) {
    if (!executed.includes(file)) {
      await executeSqlFile(file);
      await recordMigration(file);
    }
  }
}
```

**环境变量控制**：
```env
# .env

AUTO_MIGRATION=false  # 关闭自动迁移
```

```typescript
async function autoCreateMissingTables(): Promise<void> {
  for (const entity of AppDataSource.entityMetadatas) {
    const tableName = entity.tableName;
    const exists = await tableExists(tableName);

    if (!exists) {
      const createSql = generateCreateTableSQL(entity);
      await AppDataSource.query(createSql);
      log.info(`[AutoMigration] 创建表: ${tableName}`);

  }
}
```

#### 阶段3：基于实体的自动补字段

**核心逻辑**：
```typescript
// 伪代码
async function autoAddMissingColumns(): Promise<void> {
  for (const entity of AppDataSource.entityMetadatas) {
    const tableName = entity.tableName;
    const dbColumns = await getTableColumns(tableName);
    const entityColumns = entity.columns.map(c => c.databaseName);

    for (const column of entity.columns) {
      if (!dbColumns.includes(column.databaseName)) {
        const alterSql = generateAddColumnSQL(tableName, column);
        await AppDataSource.query(alterSql);
        log.info(`[AutoMigration] 补字段: ${tableName}.${column.databaseName}`);
      }
    }
  }
```
