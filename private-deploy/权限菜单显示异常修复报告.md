# 权限菜单显示异常修复报告

> 修复日期：2026-04-12  
> 问题描述：租户修改共享角色权限后，成员登录看到了本不该有的菜单（如订单审核、取消代收、客户分组等管理员专属功能）

---

## 一、问题现象

- 在管理系统角色权限设置中，只在共享角色上**新增了1-2个权限**
- 成员登录后，却能看到**全部二级菜单**，包括客户、订单审核、取消代收审核、商品管理、系统管理等**仅管理员才有的权限菜单**
- 即使数据库中该角色的权限记录是正确的（只有授权的权限），前端菜单仍然全部可见

---

## 二、根本原因分析

### Bug 1（主因）：自定义角色完全绕过菜单权限检查

**位置**：`src/utils/menu.ts` → `hasMenuPermission()` 函数

**原代码逻辑**（第62-75行）：
```typescript
const systemRoles = ['super_admin', 'admin', 'department_manager', 'manager', 'sales_staff', 'customer_service']
const isSystemRole = systemRoles.includes(userRole)

// 只对系统预设角色进行角色检查，自定义角色跳过
if (menuItem.roles && menuItem.roles.length > 0 && isSystemRole && ...) {
    if (!menuItem.roles.includes(effectiveRole)) return false
}

return true  // ← 自定义角色直接走到这里，所有菜单返回true
```

**问题**：当用户角色码不在 `systemRoles` 列表中（如自定义角色 `TONGYONG` 等），`isSystemRole = false`，导致角色白名单检查**整个被跳过**，函数直接返回 `true`。

**影响范围**：所有自定义角色的用户看到**全部菜单**，与实际权限完全无关。

### Bug 2（次因）：父模块权限反向匹配过于宽泛

**位置**：`src/utils/menu.ts` → `matchPermission()` 内部函数（第37-38行）

**原代码**：
```typescript
const parentPerm = requiredPerm.split(/[:.]/)[0]
if (userPerms.includes(parentPerm)) return true
```

**问题**：拥有模块级权限如 `order`，会自动匹配所有子权限如 `order:audit`、`order:cod_application`、`finance:cod` 等。这意味着：
- 销售员有 `order` → 能看到 `订单审核`（本应只有管理员和客服可见）
- 销售员有 `customer` → 能看到 `客户分组`、`客户标签`（本应只有管理员可见）

**权限传导链示例**：
```
用户权限: ['order', 'order.list', 'order.add']
菜单要求: 'order:audit'
父模块提取: 'order'
匹配结果: 用户有 'order' → true (错误！用户并没有 order:audit 权限)
```

---

## 三、修复方案

### 修复1：`src/utils/menu.ts` - `matchPermission` 移除父模块反向匹配

```diff
  const matchPermission = (requiredPerm, userPerms) => {
    if (userPerms.includes(requiredPerm)) return true
    // ... 格式转换匹配 ...
-   const parentPerm = requiredPerm.split(/[:.]/)[0]
-   if (userPerms.includes(parentPerm)) return true
+   // 不再做父模块反向匹配（拥有'order'不能自动匹配'order.audit'等子菜单权限）
    // 前缀匹配保留：用户有 'customer.list.view' 可匹配父菜单 'customer'
    const colonPrefix = colonFormat + ':'
    const dotPrefix = dotFormat + '.'
    if (userPerms.some(p => p.startsWith(colonPrefix) || p.startsWith(dotPrefix))) return true
    return false
  }
```

**保留的前缀匹配**（正向）：如果菜单要求 `order`（父模块），用户有 `order.list.view`（子权限），能正确匹配，保证父菜单可见。

**移除的父模块匹配**（反向）：菜单要求 `order:audit`，用户只有 `order`，不再错误匹配。

### 修复2：`src/utils/menu.ts` - `hasMenuPermission` 对所有角色强制权限检查

```diff
  if (hasExplicitPermission) return true

+ // 菜单定义了权限要求但用户没有匹配的权限 → 直接拒绝（所有角色）
+ if (menuItem.permissions && menuItem.permissions.length > 0) {
+   return false
+ }

- // 只对系统角色做角色检查（自定义角色跳过）
- const systemRoles = [...]
- if (menuItem.roles && ... && isSystemRole && ...) {
-   if (!menuItem.roles.includes(effectiveRole)) return false
- }
- return true

+ // 没有 permissions 定义的菜单，使用角色白名单（对所有角色生效）
+ if (menuItem.roles && menuItem.roles.length > 0) {
+   return menuItem.roles.includes(effectiveRole)
+ }
+ return true
```

### 修复3：后端增加权限加载详细日志

**`backend/src/controllers/UserController.ts`**：
- 登录时日志输出权限来源（租户克隆角色 / 共享角色 / 无）
- 输出权限数量，便于快速定位异常

**`backend/src/controllers/RoleController.ts`**：
- 克隆角色时输出源角色权限数量和目标角色ID
- 更新权限时输出新权限数量

## 四、修复后的权限检查流程

```
hasMenuPermission(menuItem, userRole, userPermissions):
  1. 菜单隐藏 → false
  2. super_admin 或 权限含 '*' → true
  3. 菜单有 permissions 定义：
     a. 精确匹配（含格式转换: 冒号↔点号） → true
     b. 前缀匹配（用户子权限 → 父菜单可见） → true
     c. 无匹配 → false（不再回退到 roles 或 return true）
  4. 菜单无 permissions，有 roles 定义：
     a. 用户角色在白名单中 → true
     b. 不在白名单中 → false
  5. 菜单无 permissions 无 roles → true
```

## 六、验证方式

### 1. 权限精确匹配验证
- 销售员角色有 `order`, `order.list`, `order.add`
- 应看到：订单列表、新增订单
- 不应看到：订单审核（需 `order:audit`）、取消代收审核（需 `finance:cod`）

### 2. 自定义角色验证
- 自定义角色（如 TONGYONG）只有 37 个权限
- 应只看到这 37 个权限对应的菜单项
- 不应看到管理员专属菜单（系统管理、商品管理等）

### 3. 后端日志验证
登录后检查后端日志，应输出：
```
[Login] 开始加载角色权限: roleCode=TONGYONG, tenantId=xxx
[Login] ✅ 找到租户克隆角色: id=TONGYONG_abcd1234, code=TONGYONG
[Login] 角色权限加载完成: source=tenant_clone(id=TONGYONG_abcd1234), code=TONGYONG, 权限数=39
```

### 4. 租户隔离验证
- 租户A修改角色权限 → 只影响租户A的克隆角色
- 租户B登录 → 使用租户B自己的克隆角色或共享角色，不受租户A影响
