## 一、通话系统架构：一张网，连接所有通话场景

在深入功能之前，我们先看看这套系统的"骨架"。

```
                    ┌─────────────┐
                    │   PBX/SIP   │
                    │  FreePBX    │
                    │  Asterisk   │
                    └──────┬──────┘
                           │ Webhook
                           ▼
┌──────────┐    ┌──────────────────┐    ┌──────────────┐
│ APP 端   │◀──▶│   CRM 后端       │◀──▶│ 云呼叫平台   │
│ Android  │ WS │  (Express.js)    │API │ 阿里云/腾讯  │
│ iOS      │    │                  │    │ 华为云       │
└──────────┘    └───────┬──────────┘    └──────────────┘
                        │ WebSocket
                        ▼
               ┌──────────────────┐
               │  CRM 前端 (Vue3) │
               │  通话管理页面     │
               └──────────────────┘
```

**通信链路解析：**

- **外呼**：CRM前端点击呼叫 → 后端API → 判断线路类型 → 如果是工作手机则通过WebSocket推送给APP拨号 / 如果是VoIP则调用云通信API发起呼叫
- **呼入（SIP/PBX）**：客户拨打企业号码 → PBX触发Webhook → 后端匹配客户信息 → 通过WebSocket推送来电弹窗到CRM前端
- **呼入（工作手机）**：客户拨打销售手机 → APP原生检测到来电 → 通过WebSocket上报后端 → 后端匹配客户 → 推送到CRM前端弹窗
- **录音**：通话结束 → APP自动扫描手机录音文件 → 上传至服务器 / 云通信服务端自动录音 → 回调同步

**一条关键设计：** 录音播放路由（`/recordings/stream/*`）独立于全局鉴权中间件之前，支持URL携带 `?token=` 参数，使得 `<audio>` 标签能直接播放录音，而不会因为header鉴权问题被401拒绝。

## 二、外呼管理：五种方式，覆盖所有业务场景

电销的第一件事就是打电话。但"怎么打"，决定了你的效率和成本。

### 2.1 工作手机外呼（最受团队欢迎）

销售员登录CRM，点击客户号码，系统通过WebSocket给手机APP下发拨号指令——**手机自动拨号，全程无需手动按号码。**

**背后的技术链路：**

```
CRM端点击"呼叫"
       ↓
后端创建 call_records 记录（callType: 'outbound', callStatus: 'pending'）
       ↓
后端通过 WebSocket 发送 DIAL_REQUEST 消息到 APP
       ↓
APP 的 CallStateService 接收指令
       ↓
APP 调用系统 Intent.ACTION_CALL 拨号
       ↓
APP 监听 PhoneStateListener 获取通话状态（dialing → ringing → connected → ended）
       ↓
状态实时上报 → 后端更新 call_records → WebSocket 推送到 CRM 前端更新界面
       ↓
通话结束 → APP 的 RecordingService 扫描手机录音 → 自动上传至服务器
```

**核心优势：**
- 使用真实手机号码，客户接听率比座机号高出30%以上
- 通话质量稳定，不受网络环境影响
- 自动录音，支持小米、华为、OPPO、VIVO、三星等20+品牌的录音路径自动扫描
- 销售离职，客户数据留在系统，一个都带不走

> **真实场景：**
>
> 某保健品电销公司，销售小王每天要打150-200通电话。以前拿着纸质名单一个一个拨号，手都按酸了，还经常拨错，一天光拨号就浪费了将近1小时。
>
> 现在用云客CRM，点一下客户号码，手机就自动拨出去了。通话结束，CRM自动弹出跟进窗口，记备注、打标签、设置下次跟进时间——**一气呵成**。
>
> **单通电话节省30秒，一天200通=节省1.6小时，相当于多打40通电话。**

### 2.2 网络电话VoIP（适合大批量外呼）

如果你的团队每天外呼量上千通，工作手机不够用怎么办？VoIP线路来顶上。

云客CRM已经对接了**阿里云通信、腾讯云通信、华为云通信**三大主流服务商。管理员在后台配置好线路，销售一键切换即可使用。

**VoIP的技术实现：**

- 后端 `AliyunCallService.ts` 封装了与阿里云VMS/CCC API的完整对接
- 发起外呼 → 调用云通信API → 服务商发起呼叫 → 状态回调 `POST /webhook/aliyun/status` → 自动更新 `call_records`
- 录音回调 `POST /webhook/aliyun/recording` → 自动关联录音到通话记录

**VoIP对比工作手机：**

| 维度 | 工作手机外呼 | VoIP网络电话 |
|:----|:----------|:-----------|
| 号码显示 | 真实手机号 | 固话/400号码 |
| 客户信任度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 通话质量 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐（依赖网络） |
| 成本 | 手机套餐费 | 按分钟计费，约0.05-0.15元/分钟 |
| 并发能力 | 1路/手机 | 10-100路可配置 |
| 录音方式 | APP自动扫描上传 | 服务端自动录音 |

### 2.3 SIP线路（对接已有电话系统）

如果企业已经有自己的IP-PBX（如FreePBX、Asterisk等），或者有传统的固话中继线路，云客CRM同样支持对接。

**支持的中继类型：**
- **模拟中继（FXO）**：连接传统电话线路，简单稳定，适合小型团队
- **数字中继（E1/T1）**：E1支持30路通话，适合中等规模呼叫中心
- **SIP中继**：通过IP网络连接，灵活扩展通道数
- **PRI中继**：基于ISDN的数字中继，信令稳定可靠

> 已有投资不用废，老设备直接复用。Webhook密钥验证支持 `crypto.timingSafeEqual` 防时序攻击，安全可靠。

### 2.4 PSTN语音网关

通过传统电话网关拨号，适合使用传统固话线路的企业。支持模拟/数字中继，兼容性强。

### 2.5 手动拨号记录

针对线下面对面沟通、微信语音等场景，提供手动录入通话记录的功能，确保数据完整。

### 2.6 智能线路优先级

系统自动判断呼叫线路，销售不用选，系统帮你选最优方案：

```
工作手机已绑定 + 在线 → 优先使用工作手机
工作手机离线 + 有VoIP线路 → 自动切换VoIP
工作手机离线 + 无VoIP线路 → 提示销售手动拨打
```

同时，管理员可以在"呼出配置"中为不同员工分配不同的可用线路，实现精细化管理。

## 三、呼入管理：客户主动上门，第一时间识别

电销不只是打出去，还有打进来的。客户看到广告来电咨询、老客户回电谈复购——**每一个呼入电话，都是一次成交机会。**

### 3.1 场景一：SIP/PBX呼入（企业固话）

客户拨打企业400电话或固话号码：
```
客户拨打企业号码
    ↓
企业PBX/SIP网关收到来电
    ↓ POST /api/v1/calls/webhook/sip/incoming
CRM后端（callWebhook.ts）
    ├─ 1. 根据主叫号码查找客户信息（phone/mobile字段）
    ├─ 2. 根据被叫号码/DID匹配负责人
    ├─ 3. 创建 inbound 类型 call_records 记录
    └─ 4. WebSocket推送 CALL_INCOMING 到CRM前端
         ↓
CRM前端（CallManagement.vue）
    ├─ 弹出来电提醒弹窗
    ├─ 显示客户信息：姓名、号码、等级、历史订单、跟进记录
    └─ 操作按钮：接听确认 / 拒绝 / 查看详情 / 快速跟进
```

**销售不用问"您是哪位"，拿起电话就能直呼其名：**
> "李总您好，上次的保健品用完了是吧？这次给您推荐新款……"

**专业感，从来电的第一秒开始。**

### 3.2 场景二：工作手机呼入

如果客户打的是销售的个人手机号呢？

APP在后台运行（前台服务保活），通过 `PhoneStateListener`（Android）/ `CXCallObserver`（iOS）检测到来电状态变化：

```
客户拨打销售手机号
    ↓
手机系统响铃
    ↓ APP原生检测（PhoneStateListener）
APP端
    ├─ 检测到来电状态（RINGING）
    ├─ 获取来电号码
    └─ 通过 WebSocket 发送 INCOMING_CALL_DETECTED 消息
         ↓
CRM后端（MobileWebSocketService）
    ├─ 1. 解析来电号码
    ├─ 2. 查找匹配客户信息
    ├─ 3. 创建 inbound 类型 call_records（status: ringing）
    └─ 4. 通过 WebSocket 推送 CALL_INCOMING 到CRM前端
         ↓
CRM前端
    └─ 弹出来电提醒弹窗（与SIP呼入一致）
```

**无论客户打哪个号码，CRM都能识别出来。**

### 3.3 呼叫路由与分配

后端实现了完整的呼入路由分配逻辑，优先级从低到高：
1. `inbound_routes` 表 DID 号码路由匹配
2. `user_line_assignments` 表线路分配匹配
3. 客户专属销售匹配（客户表的 salesPersonId）
4. 坐席状态检查（agent_status，仅就绪状态的坐席接收分配）
5. 默认分配给部门主管或随机分配

### 3.4 Webhook 安全验证

所有呼入Webhook请求都经过密钥验证：
```
环境变量: SIP_WEBHOOK_SECRET
支持 Header: X-Webhook-Secret 或 Body: secret 字段
使用 crypto.timingSafeEqual 防止时序攻击
未配置密钥时跳过验证（开发模式）
```

## 四、通话记录管理：每一通电话都可追溯

"我没说过这个价格。"
"客户说收到货了，但没付款。"
"那个客户说考虑一下，是谁跟进的？"

**没有记录，这些都是扯皮。有记录，就是铁证。**

### 4.1 完整的数据模型

每一条通话记录包含完整的字段信息：

```
call_records 表字段：
├── 基本信息：id, tenantId, customerId, customerName, customerPhone
├── 通话类型：callType（outbound / inbound）
├── 通话状态：callStatus（connected / missed / busy / failed / rejected / pending / calling）
├── 时间信息：startTime, endTime
├── 通话时长：duration（秒）
├── 录音信息：recordingUrl, hasRecording, recordingSize
├── 跟进信息：notes（通话备注）, callTags（通话标签JSON）, followUpRequired
├── 线路信息：callMethod（system/mobile/voip/sip/manual）, lineId, callerNumber, providerCallId
├── 挂断原因：hangupCause
├── 人员信息：userId, userName, department
└── 时间戳：createdAt, updatedAt（自动管理）
```

### 4.2 多维筛选与搜索

通话记录支持丰富的筛选维度：
- **时间范围**：任意日期区间
- **通话状态**：已接通 / 未接通 / 忙线 / 失败 / 拒接
- **通话类型**：呼入 / 呼出
- **通话方式**：工作手机 / VoIP / SIP
- **客户搜索**：按客户姓名或号码模糊搜索
- **人员筛选**：管理员可查看全团队，部门经理看本部门，销售只看自己

### 4.3 通话详情与关联

点击一条通话记录，可以查看完整详情：
- 通话基本信息（时间、时长、状态、方式）
- 关联客户信息（点击可直接跳转到客户详情页）
- 该客户的订单历史、售后记录
- 通话录音播放/下载
- 关联的跟进记录列表

### 4.4 数据安全与权限隔离

**角色权限严格控制数据可见范围：**

| 角色 | 可查看范围 |
|:----|:----------|
| 超级管理员 | 全租户所有通话记录 |
| 管理员 | 全团队数据，支持按人员筛选 |
| 部门经理 | 本部门数据，支持按成员筛选 |
| 普通销售 | 仅自己数据 |

## 五、录音管理：完整的录音生命周期

没有录音的电销管理，就像没有监控的银行——出了问题谁也说不清。

云客CRM构建了从**录音采集 → 上传 → 存储 → 播放 → 下载 → 清理**的完整链路。

### 5.1 多场景录音支持

| 外呼方式 | 录音方式 | 可靠度 | 自动上传 |
|:-------|:-------|:------|:--------|
| **工作手机外呼** | 手机系统原生通话录音 | ⭐⭐⭐⭐ | APP自动扫描上传 |
| **VoIP网络电话** | 云服务商服务端自动录音 | ⭐⭐⭐⭐⭐ | 回调自动同步 |
| **SIP线路** | SIP服务器/PBX录音模块 | ⭐⭐⭐⭐⭐ | API/FTP自动同步 |
| **PSTN网关** | 网关/PBX内置录音 | ⭐⭐⭐⭐ | FTP/API定期同步 |

### 5.2 APP录音自动上传（最有特色的功能）

APP的 `recordingService.ts` 实现了非常强大的录音自动扫描上传机制：

**支持30+品牌手机的通话录音路径：**

| 品牌 | 录音存储路径 |
|:----|:-----------|
| 小米/红米/POCO | `/MIUI/sound_recorder/call_rec/` |
| 华为/荣耀 | `/Sounds/CallRecord/`, `/PhoneRecord/` |
| OPPO/Realme/一加 | `/Recordings/Call Recordings/`, `/ColorOS/Recordings/` |
| VIVO/iQOO | `/Record/Call/`, `/VIVORecorder/` |
| 三星 | `/Call/`, `/Recordings/Call recordings/` |
| Google Pixel | `/Recordings/Call recordings/` |
| Nothing Phone | `/Recordings/Call Recordings/` |
| 魅族 | `/Recorder/Call/` |
| 联想/摩托罗拉 | `/Record/PhoneRecord/` |
| TECNO/Infinix/itel | `/Record/Call/`, `/CallRecorder/` |

**录音文件名智能匹配规则：**
- 文件名包含11位手机号模式（`/1[3-9]\d{9}/`） → 匹配
- 文件名包含 "call" 关键词 → 匹配
- 文件名包含中文 "通话" → 匹配
- 文件名包含 "record" + 日期数字模式 → 匹配
- 文件名是纯数字时间戳格式（`20260417143022.amr`）→ 匹配
- Google Dialer格式 "Recording_电话号码" → 匹配

**上传流程：**

```
通话结束（APP检测到IDLE状态）
    ↓ 等待2秒（确保录音文件已写入完成）
APP扫描所有已知录音目录
    ↓ 按品牌优先级排序路径
查找匹配当前通话的录音文件
    ↓ 按通话时间匹配
找到录音文件
    ↓ POST /api/v1/mobile/recording/upload
上传至服务器
    ↓ 上传成功
更新 call_records 表的 has_recording=1, recording_url
    ↓ 自动删除本地录音（节省手机空间）
触发 `recording:uploaded` 事件通知通话记录列表刷新
```

### 5.3 录音播放与转码

后端 `RecordingStorageService` 实现了智能播放：
- **在线流播放**：支持 `Accept-Ranges` 范围请求，拖拽进度条
- **自动转码**：amr/3gp 格式自动通过 ffmpeg 转码为 mp3，浏览器兼容无忧
- **Token鉴权**：URL支持 `?token=` 参数，`<audio>` 标签直接播放

### 5.4 录音存储管理

支持两种存储方式：
- **本地存储**：文件存储在服务器 recordings 目录，适合私有化部署
- **云存储**：支持阿里云OSS、腾讯云COS，适合大规模录音存储

提供存储统计功能，可查看：
- 总录音数
- 总存储空间
- 总录音时长
- 按存储类型分布统计

### 5.5 录音定期清理

支持设置录音过期时间，系统自动清理过期录音，节省存储空间。

### 5.6 一个真实的法律案例

> 某金融电销公司，客户投诉销售承诺"保本保息"。
>
> 销售说没说过，客户说有自己的录音——但客户的录音只有中间一段，被截取拼接，无法反映完整对话。
>
> 公司调出CRM中的完整通话录音，从开场白到挂机，一整段清晰可辨。最终证明销售从未说过"保本保息"，为客户断章取义。
>
> **录音不只是管理工具，更是保护公司和销售的法律武器。**

## 六、外呼名单导入与管理：从Excel到拨号，只需3步

电销的第一步是"有名单"。但现实是名单在Excel里、在微信文件里、在各个渠道的表单里——**销售需要一个一个复制粘贴去打电话。**

### 6.1 双通道导入模式

#### ✅ 单个录入（适合零星添加）

```
打开"导入外呼名单"弹窗 → 选择"单个录入"标签
    ├─ 输入客户姓名（必填）
    ├─ 输入手机号（必填）→ 失焦自动检测
    ├─ 选填：性别、收货地址、备注、标签
    └─ 点击"确认录入"→ 系统最终验证 → 录入成功
```

**核心细节：** 手机号输入后失焦即触发验证，同时在`外呼名单表`和`客户表`中检查是否存在，防止重复导入。这是从源头杜绝撞单的关键设计。

#### ✅ 批量导入（适合大批量名单）

**三步操作，流畅体验：**

**Step 1：上传文件**
- 下载系统标准模板（xlsx格式）
- 按模板格式填入客户数据
- 支持 `.xlsx / .xls / .csv` 格式，单次最多5000条
- 拖拽或点击上传

**Step 2：数据预览**
- 系统自动解析并展示所有数据
- 智能列名映射：支持"客户姓名→name"、"手机号→phone"等多达15种列名自动匹配

## 七、跟进管理：通话结束，服务才开始

很多电销团队的问题不是"不打"，而是"打了之后没有然后"。

### 7.1 通话结束即跟进（关键设计）

通话结束后，CRM自动弹出跟进窗口，**引导而不是强迫**销售完成跟进动作——这是一个很重要的产品设计哲学：减少销售的操作负担，而不是增加负担。

```
通话已结束 (00:03:25)
━━━━━━━━━━━━━━━━━━━━━
☐ 意向    ☐ 无意向    ☐ 再联系
☐ 成交    ☐ 需报价    ☐ 已成交

跟进备注：客户对产品有兴趣，但想对比一下价格
          已发送产品资料到微信

下次跟进时间：2026-06-15 10:00  [📅 选择日期]

[ 保存跟进 ]  [ 稍后再说 ]
```

### 7.2 完整的跟进数据模型

每条跟进记录包含：
- **跟进类型**：call（电话）/ visit（拜访）/ email（邮件）/ message（消息）
- **跟进内容**：详细的沟通记录
- **客户意向**：high（高意向）/ medium（中意向）/ low（低意向）/ none（无意向）
- **通话标签**：意向 / 无意向 / 再联系 / 成交 / 需报价 / 已成交（数组，支持多标签）
- **下次跟进时间**：精确到分钟
- **优先级**：urgent（紧急）/ high（高）/ medium（中）/ low（低）
- **状态**：pending（待处理）/ completed（已完成）/ cancelled（已取消）

### 7.3 自动创建跟进提醒

当销售设置了"下次跟进时间"后，系统自动在 `notifications` 表创建一条跟进提醒通知：

```
通知内容示例：
"您有一条待跟进任务，计划跟进时间：2026-06-15 10:00。
客户：张三，跟进内容：客户对产品有兴趣，但想对比一下价格"
```

提醒方式：
- **站内消息通知**：登录CRM后右上角小红点提示
- **弹窗提醒**：到时间自动弹出跟进提醒
- **通知列表**：所有待跟进任务在"待跟进"列表中集中展示

### 7.4 跟进看板

管理者可以在系统中查看团队成员的跟进情况：
- 今日跟进数
- 待跟进任务（pending状态）
- 逾期未跟进（超过设定的跟进时间仍未完成）
- 紧急跟进任务（priority=urgent）

系统计算 `pendingFollowUps` 和 `urgentFollowUps` 两个关键指标，帮助管理者快速掌握团队跟进状态。

> **真实案例：**
>
> 某金融电销公司，以前销售每天打100通电话，记了七八个"再联系"的客户在本子上，到第二天就忘了。
>
> 用了云客CRM的跟进管理后，系统每天上午10点自动弹出待跟进名单，销售按名单一个一个回访。一个月后，跟进完成率从35%提升到82%，销售额涨了40%。

### 8.6 远程拨号（最核心功能）

这是最受电销团队欢迎的功能。

**操作流程：**

```
销售打开PC端CRM
    → 在通话管理/客户详情中点击客户号码
    → 选择"工作手机拨号"
    → 后端通过 WebSocket 发送 DIAL_REQUEST 到绑定的APP
    → APP的 CallStateService 接收指令
    → APP调用系统拨号 Intent
    → 手机自动拨出客户号码
    → 销售在PC端通过耳麦与客户通话
```

整个过程：**销售不需要碰手机，手机放在桌上，自动拨号、自动挂断、自动上传录音。**

**WebSocket消息格式：**
```json
{
  "type": "DIAL_REQUEST",
  "messageId": "msg_xxxx",
  "timestamp": 1689583200000,
  "data": {
    "callId": "call_1689583200_xxxx",
    "phoneNumber": "13800138000",
    "customerName": "张三",
    "customerId": "cust_xxxx",
    "source": "crm_web",
    "operatorId": "user_xxxx",
    "operatorName": "销售小王"
  }
}
```

### 8.7 通话状态实时同步

APP通过 `CallStateService` 实时监听通话状态，并通过WebSocket上报：

```
状态变化链路：
DIALING（拨号中） → RINGING（响铃中） → CONNECTED（已接通） → ENDED（已结束）
       ↑                ↑                   ↑                   ↑
    上报后端         上报后端              上报后端            上报后端
```

后端收到状态更新后：
1. 更新 `call_records` 表中的 `callStatus` 字段
2. 通过WebSocket推送到CRM前端
3. CRM前端实时更新通话状态界面

### 8.8 来电检测与服务

APP在后台运行（前台服务保活），通过原生API监听来电状态：

```
Android：PhoneStateListener + BroadcastReceiver
iOS：CXCallObserver

检测到来电 → RINGING状态
    → 获取来电号码
    → 通过WebSocket上报 INCOMING_CALL_DETECTED
    → 后端匹配客户信息
    → PC端弹出：来电弹窗
```

**保活机制：**
- 前台服务（Foreground Service）保持APP在后台不被系统杀死
- 30秒心跳检测维持WebSocket连接
- 断线自动重连，最多尝试10次
- 外呼状态超过5分钟未清理时强制清理，防止状态残留

### 8.9 统计页面

APP端提供多维度的个人通话统计：

**核心指标卡片：**
- 总通话数 + 接通率环形图
- 已接通数 / 未接通数 分类卡片
- 总通话时长 / 平均通话时长

**周期筛选：**
- 今日
- 本周
- 本月

**详细趋势：**
- 日/周/月的通话趋势变化
- 个人业绩排行榜（团队版）

### 8.10 设置页面

- **账号信息**：用户信息、设备绑定状态
- **录音设置**：自动录音开关、录音引导（根据手机品牌智能显示操作指引）
- **通知设置**：来电提醒、振动开关

## 十二、多行业应用场景
