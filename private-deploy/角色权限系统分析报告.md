> 生成时间：2026-04-28  
> 分析范围：角色权限树、权限ID、授予回收、数据库存储、数据范围、租户隔离

## 一、当前系统架构

### 1.1 权限定义（权限树结构）

| 来源 | 文件 | 格式 | 说明 |
|------|------|------|------|
| 主权限树 | `src/services/permissionService.ts` → `PERMISSION_TREE` | 点号 (`wecom.customer.view`) | PermissionSettingsDialog 直接调用 |
| 类型权限树 | `src/types/permissions.ts` → `PERMISSION_TREE` | 冒号 (`wecom:customer:view`) | 类型定义，部分组件引用 |
| 降级数据 | `src/views/System/Role/permissionFallbackData.ts` | 点号 | API失败时的回退 |
| 角色默认权限 | `src/config/defaultRolePermissions.ts` | 点号 | 新角色/恢复默认时使用 |
| 菜单权限映射 | `src/config/menu.ts` → `rolePermissions` | 冒号 | 控制菜单可见性 |

**⚠️ 核心问题：权限树定义是硬编码在前端代码中的**

- `PermissionSettingsDialog.vue` 第282行：`permissionService.getAllPermissions()` 直接返回硬编码的 `PERMISSION_TREE`
- 并未调用后端API `GET /api/v1/permissions/tree`
- 修改权限树必须修改代码并重新部署，无法在线配置

### 1.2 权限存储与生效流程

```
[权限设置弹窗] → el-tree 展示硬编码的 PERMISSION_TREE
       ↓ 用户勾选后点"保存"
[roleApiService.updateRolePermissions(id, permissions)]
       ↓ PUT /api/v1/roles/:id/permissions
[RoleController] → UPDATE roles SET permissions = JSON数组 WHERE id = ?
       ↓ 
[数据库 roles 表] → permissions 字段 (JSON数组存储权限ID字符串)
       ↓ 用户登录
[roleApiService.getRolePermissions(id)] → 从数据库读取角色权限
       ↓
[PermissionSettingsDialog] → setCheckedKeys(validPermissions)
```

**角色的权限赋值（哪些权限被勾选）→ 已走数据库 ✅**  
**权限树本身（有哪些可勾选的权限节点）→ 仍是硬编码 ❌**

## 二、数据范围权限分析

### 2.1 数据范围（data_scope）

在 `PermissionSettingsDialog.vue` 中，每个角色有三种数据范围：

| 数据范围 | 值 | 说明 | 适用角色 |
|----------|-----|------|----------|
| 全部数据 | `all` | 可查看系统中所有数据 | 超级管理员、管理员 |
| 部门数据 | `department` | 仅可查看本部门及下属部门的数据 | 部门经理 |
| 个人数据 | `self` | 仅可查看自己创建的数据 | 销售员、客服 |

### 2.2 数据范围存储

- 存储在 `roles` 表的 `data_scope` 字段
- 通过 `roleApiService.updateRole({ id, dataScope })` 保存到数据库
- 支持租户隔离（每个租户可为自己的角色设定不同的数据范围）

### 2.3 数据范围与权限的关系

- 数据范围是**独立于**功能权限的维度
- 功能权限控制"能做什么"（如查看、编辑、删除）
- 数据范围控制"能看到哪些数据"（如全部、部门、个人）
- 两者在 `PermissionSettingsDialog.vue` 中通过两个Tab页分开设置

### 2.4 各角色默认数据范围

| 角色 | 默认数据范围 | 说明 |
|------|-------------|------|
| 超级管理员 (super_admin) | `all` | 全部数据 |
| 管理员 (admin) | `all` | 全部数据 |
| 部门经理 (department_manager) | `department` | 本部门及下属部门 |
| 销售员 (sales_staff) | `self` | 仅个人数据 |
| 客服 (customer_service) | `all` | 全部数据（处理全公司工单） |

## 三、租户隔离分析

### 3.1 已有的租户隔离机制

| 组件 | 隔离状态 | 说明 |
|------|---------|------|
| `roles` 表 | ✅ 已有 `tenant_id` | 每个租户可创建自己的角色 |
| `permissions` 表 | ✅ 已有 `tenant_id` | 支持租户级权限定义 |
| `tenantRepo.ts` | ✅ Proxy 自动注入 | `roles` 和 `permissions` 已在 TENANT_ENTITIES 列表中 |
| `RoleController` | ✅ 手动租户过滤 | 使用 `tenantRawSQL()` / `getCurrentTenantIdSafe()` |
| `PermissionController` | ✅ 手动租户过滤 | 使用 `getCurrentTenantIdSafe()` |
| `TenantContextManager` | ✅ AsyncLocalStorage | 基于请求的租户上下文管理 |

### 3.2 租户隔离结论

**后端租户隔离框架完善 ✅**

- 角色表和权限表均已有 `tenant_id` 字段
- `getTenantRepo()` 和 `tenantRawSQL()` 工具已覆盖所有 CRUD 操作
- 每个租户可以独立创建角色、设定权限，互不影响

**但权限树是硬编码的，所有租户看到相同的权限树 ❌**

- 当前前端直接使用 `permissionService.getAllPermissions()` 返回硬编码树
- 未调用 `GET /api/v1/permissions/tree` API
- 导致：即使数据库支持按租户隔离权限，前端也不会使用

## 四、问题清单

### 4.1 核心问题

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | **权限树硬编码** | 🔴 高 | `PermissionSettingsDialog` 使用 `permissionService.getAllPermissions()` 而非API |
| 2 | **permissions表为空** | 🔴 高 | 数据库有 `permissions` 表但从未写入数据 |
| 3 | **前端未调用权限API** | 🔴 高 | `permissionApiService.getPermissionTree()` 已存在但未被调用 |
| 4 | **所有租户共享同一权限树** | 🟡 中 | 因硬编码导致，数据库驱动后自动解决 |

### 4.2 影响分析

- **不灵活**：修改权限树必须修改代码重新部署
- **不一致**：前端5处权限定义可能出现不同步
- **租户无法自定义**：所有租户看到完全相同的权限选项

---

## 五、解决方案

### 5.1 核心思路

**将硬编码的 PERMISSION_TREE 数据迁移到 `permissions` 表，前端改为从API加载权限树**

### 5.2 改造步骤

1. **数据库迁移**：将现有 PERMISSION_TREE 的全部权限节点 INSERT 到 `permissions` 表
2. **后端验证**：确认 `PermissionController.getPermissionTree()` 能正确返回树结构
3. **前端改造**：`PermissionSettingsDialog.vue` 的 `loadPermissionTree()` 改为调用 API
4. **保留降级**：API失败时回退到硬编码数据（已有 fallback 机制）

### 5.3 预期效果

- 权限设置弹窗 → 从数据库加载权限树 → 勾选后保存到数据库 → 立即生效
- 每个租户可以有自己独立的权限树（通过 tenant_id 隔离）
- 在系统管理中随时配置，无需修改代码
