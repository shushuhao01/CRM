## 🔴 一、JSON 转义与数据处理规范 (最高优先级)

### 1.1 严禁在 JSON 字符串中出现非法转义

```
❌ 禁止：在 JSON 字符串值中使用未转义的反斜杠、换行符、制表符
❌ 禁止：手动拼接 JSON 字符串（如 '{"key":"' + value + '"}'）
✅ 必须：始终使用 JSON.stringify() 生成 JSON 字符串
✅ 必须：始终使用 JSON.parse() 解析 JSON 字符串（包裹 try-catch）
```

### 1.2 JSON 安全编码模板

**后端（Node.js / TypeScript）：**
```typescript
// ✅ 正确：安全序列化
const jsonStr = JSON.stringify(dataObject);

// ✅ 正确：安全反序列化
function safeJsonParse<T>(str: string, fallback: T): T {
  try {
    return JSON.parse(str) as T;
  } catch {
    return fallback;
  }
}

// ✅ 正确：数据库 JSON 字段存储
entity.tagIds = JSON.stringify(tagArray);        // 存
const tags = safeJsonParse(entity.tagIds, []);   // 取

// ❌ 禁止：手工拼接
const bad = '{"ids":[' + ids.join(',') + ']}';
```

**前端（Vue / TypeScript）：**
```typescript
// ✅ 正确：API 请求体
await request.post('/api/endpoint', dataObject);
// axios/request 内部会自动 JSON.stringify，不要手动序列化

// ✅ 正确：localStorage 存取
localStorage.setItem('key', JSON.stringify(value));
const data = safeJsonParse(localStorage.getItem('key') || '', defaultValue);

// ❌ 禁止：模板字符串拼 JSON
const bad = `{"name":"${userName}"}`;  // userName 含引号或换行会炸
```

### 1.3 数据库 JSON 字段规范

```typescript
// TypeORM 实体中 JSON 字段定义
@Column({ type: 'json', nullable: true })
features: Record<string, any> | null;

// 或使用 text + 手动序列化
@Column({ type: 'text', nullable: true, comment: '标签ID列表(JSON)' })
tagIds: string | null;

// 读取时必须 safeJsonParse
const ids: string[] = safeJsonParse(record.tagIds, []);
```

### 1.4 API 响应规范

```typescript
// ✅ 统一响应格式 — Express 的 res.json() 自动处理序列化
res.json({ success: true, data: result });
res.json({ success: false, message: '错误描述' });

// ❌ 禁止：手动 res.send(JSON.stringify(...))，除非有特殊 Content-Type 需求
```

## 🟠 二、代码风格与结构规范

### 2.1 文件编码与格式
- 所有源文件使用 **UTF-8** 编码
- 缩进使用 **2 个空格**（不用 Tab）
- 行尾统一 **LF**（不用 CRLF）
- 文件末尾保留一个空行
- 单行不超过 **100** 字符

### 2.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| Vue 组件文件 | PascalCase.vue | `WecomConfig.vue` |
| TypeScript 文件 | camelCase.ts | `wecomApiService.ts` |
| 实体类 | PascalCase | `WecomConfig` |
| 数据库表名 | snake_case | `wecom_configs` |
| 数据库列名 | snake_case | `corp_id` |
| TypeScript 属性 | camelCase | `corpId` |
| API 路由 | kebab-case | `/wecom/acquisition-links` |
| CSS 类名 | kebab-case | `.wecom-config` |
| 常量 | UPPER_SNAKE_CASE | `WECOM_API_BASE` |

### 2.3 TypeScript 规范
- 严格模式：不使用 `any`（已有代码例外，新代码必须明确类型）
- 优先使用 `interface` 定义对象类型，`type` 用于联合类型和工具类型
- 函数参数超过 3 个时使用对象参数 `(options: { ... })`
- 异步函数统一使用 `async/await`，禁止裸 `.then().catch()`

## 🟡 三、项目架构规范

### 3.1 后端分层（backend/src/）

```
entities/          → TypeORM 实体定义（数据库映射）
services/          → 业务逻辑服务层
routes/            → Express 路由定义（薄控制器）
controllers/       → 复杂业务的控制器
middleware/        → 中间件（认证、限流、日志等）
config/            → 配置文件（数据库、日志、部署等）
utils/             → 工具函数
```

### 3.2 前端分层（src/）

```
api/               → API 请求封装（一个模块一个文件）
views/             → 页面组件（按模块分文件夹）
components/        → 可复用组件
stores/            → Pinia 状态管理
config/            → 菜单、常量等配置
services/          → 前端服务（模块状态等）
utils/             → 工具函数
router/            → 路由配置
```

### 3.3 新增模块检查清单

添加新功能模块时，必须同步完成以下步骤：

- [ ] **后端实体** → `backend/src/entities/XxxEntity.ts`
- [ ] **注册实体** → `backend/src/config/database.ts` 的 entities 数组
- [ ] **数据库迁移** → `backend/database-migrations/` 添加 SQL 脚本
- [ ] **后端路由** → `backend/src/routes/xxx.ts` + 在 `app.ts` 注册
- [ ] **前端 API** → `src/api/xxx.ts`
- [ ] **前端页面** → `src/views/Xxx/*.vue`
- [ ] **前端路由** → `src/router/index.ts` 添加路由
- [ ] **菜单配置** → `src/config/menu.ts` 添加菜单项
- [ ] **权限配置** → `src/config/menu.ts` 的 `rolePermissions` 添加权限
- [ ] **模块注册** → `src/services/moduleStatusService.ts` 的 DEFAULT_MODULES

## 🟢 四、安全规范

### 4.1 敏感信息处理
- API 返回中**必须**脱敏：Secret、密码、密钥 → 返回 `'******'`
- 更新接口中，值为 `'******'` 时**跳过更新**（表示未修改）
- 日志中**禁止**输出完整密钥，最多输出前10位 + `...`

### 4.2 多租户隔离
- CRM 端 API 必须使用 `getTenantRepo()` 而非直接 `AppDataSource.getRepository()`
- 租户回调类接口（无 JWT 上下文）使用 `AppDataSource.getRepository()` 并遍历验证
- 管理后台 API 使用 `adminAuthMiddleware` 认证

### 4.3 输入验证
- 所有用户输入必须验证（类型、长度、格式）
- SQL 参数化查询，禁止拼接 SQL
- 前端表单必须配合 Element Plus 的 `rules` 做校验

## 🔵 五、Git 提交规范

```
feat: 新功能      fix: 修复bug      docs: 文档更新
style: 格式调整    refactor: 重构     perf: 性能优化
test: 测试        chore: 构建/工具    ci: CI/CD
```

格式：`<type>(<scope>): <description>`

示例：
```
feat(wecom): 新增企微获客助手页面
fix(wecom): 修复会话存档分页查询参数错误
docs(wecom): 更新企微管理模块开发方案
```
