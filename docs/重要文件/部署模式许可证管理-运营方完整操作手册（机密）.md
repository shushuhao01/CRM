## 一、先搞清楚：系统里有哪些"许可证"

系统里有 **3 种不同的"许可证/授权码"**，很多人搞混，这里一次讲清楚：

| 序号 | 名称 | 格式示例 | 谁签发 | 给谁用 | 作用 |
|------|------|----------|--------|--------|------|
| ① | **SaaS 平台许可证** | `SAAS_LICENSE_TOKEN=eyJhbG...`（很长的JWT） | 你自己用脚本生成 | **只给你自己的服务器** | 证明"这台服务器有权运行 SaaS 多租户模式" |
| ② | **租户授权码** | `TENANT-A1B2-C3D4-E5F6-G7H8` | 管理后台自动生成 | 给你的 SaaS 租户客户 | 租户登录 CRM 时输入，验证租户身份 |
| ③ | **私有部署授权码** | `PRIV-X1Y2-Z3W4-Q5R6` | 管理后台"私有客户"页面生成 | 给私有部署客户 | 客户激活私有部署的 CRM 系统 |

**关系图**：

```
你（平台运营方）
  │
  ├── 你自己的 SaaS 服务器
  │   ├── ① SaaS 平台许可证（SAAS_LICENSE_TOKEN） ← 你自己生成，配在.env里
  │   ├── 运行 SaaS 多租户模式
  │   └── 租户 A、B、C... 
  │       └── ② 租户授权码 ← 管理后台创建租户时自动生成
  │
  └── 卖给客户张三的私有部署
      ├── 没有 ① SaaS 平台许可证（也不需要）
      ├── 运行 private 私有模式
      └── ③ 私有部署授权码 ← 管理后台"私有客户"页面生成
```

## 二、你自己部署 SaaS 版本的完整步骤

### 第 1 步：生成 SaaS 平台许可证

这个只需要做**一次**（或者每年续期时重新做一次）。

**在你的开发电脑上操作**（不是在服务器上）：

```bash
# 进入后端目录
cd backend

# 生成许可证（指定你的实际域名和有效期）
node scripts/generate-saas-license.js --domain crm.yourdomain.com --expires 365
```

屏幕上会输出类似这样的内容：

```
🔐 CRM SaaS 平台许可证生成工具
==================================================

📋 许可证参数：
   绑定域名: crm.yourdomain.com
   最大租户数: 无限制
   有效期: 365 天

✅ 许可证已生成！
   许可证ID: dcb259ce-38a1-4d3b-b1f3-046497dabfce
   过期时间: 2027-04-02

==================================================
📝 请将以下内容添加到 .env 文件中：
==================================================

SAAS_LICENSE_TOKEN=eyJhbGciOiJSUzI1NiI......（很长一串）
```

**复制那个 `SAAS_LICENSE_TOKEN=eyJ...` 整行。**

### 第 2 步：配置服务器 .env

SSH 登录到你的 SaaS 服务器，编辑后端 `.env` 文件：

```bash
# SSH 到服务器
ssh root@your-server

# 编辑后端配置文件
cd /www/wwwroot/your-crm-project/backend
nano .env    # 或者 vim .env
```

确保以下两行存在（第一行可能已经有了，第二行是新加的）：

```env
# 这行设置为 saas
DEPLOY_MODE=saas

# 这行粘贴你刚才生成的 Token（注意是一整行，没有换行）
SAAS_LICENSE_TOKEN=eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIs......
```

### 第 3 步：重启后端服务

```bash
# 如果用 PM2
pm2 restart crm-backend

# 如果用宝塔面板的 Node.js 管理
# 在宝塔面板中重启 Node.js 项目
```

### 第 4 步：检查启动日志确认 SaaS 模式正常

```bash
pm2 logs crm-backend --lines 30
```

你应该看到这样的日志（表示 SaaS 验证通过）：

```
[SaaSGuard] 检测到 DEPLOY_MODE=saas，开始验证 SaaS 平台许可...
[SaaSGuard] ✅ SaaS 平台许可验证通过！
[SaaSGuard]   许可证ID: dcb259ce-38a1-4d3b-b1f3-046497dabfce
[SaaSGuard]   绑定域名: crm.yourdomain.com
[SaaSGuard]   最大租户数: 无限制
[SaaSGuard]   过期时间: 2027-04-02T06:32:37.000Z
==================================================
CRM系统部署配置
==================================================
配置的部署模式: saas
实际生效模式: saas          ← 这里必须是 saas！
是否SaaS模式: true          ← 这里必须是 true！
==================================================
```

**如果看到 `实际生效模式: private`，说明 Token 有问题，请检查：**
- Token 有没有复制完整（不能有换行或空格）
- Token 有没有过期
- .env 文件有没有保存

### 第 5 步：现在你的 SaaS 系统正常运行

之后就是正常的 SaaS 运营流程了：
1. 登录**管理后台** (`admin` 项目) 
2. 在"租户客户"中创建新租户 → 系统自动生成租户授权码
3. 把租户授权码发给客户 → 客户打开你的 CRM 网址输入授权码就能登录使用

## 三、卖给客户私有部署版本的完整步骤

### 第 1 步：在管理后台为客户生成私有授权码

1. 打开你的**管理后台** （`admin` 项目，通常地址类似 `https://admin.yourdomain.com`）
2. 左侧菜单点 **「私有客户」** → **「客户列表」**
3. 点右上角 **「新增客户」** 按钮
4. 填写客户信息：
   - 客户名称：张三公司
   - 联系人：张三
   - 授权类型：年度授权 / 永久授权
   - 用户数量：根据客户购买的数量
   - 功能模块：勾选客户购买的功能
5. 点 **「生成授权码」**
6. 系统会生成一个类似 `PRIV-A1B2-C3D4-E5F6-G7H8` 的授权码
7. **复制这个授权码，发给客户**

### 第 2 步：准备交付给客户的代码包

给客户的代码包中，`.env` 文件应该这样配置：

```env
# ⚠️ 客户的配置文件
DEPLOY_MODE=private

# 不需要配置 SAAS_LICENSE_TOKEN（客户根本不需要这个）
# SAAS_LICENSE_TOKEN=          ← 留空或直接删掉这行

# 其他配置（数据库等根据客户环境填写）
DB_HOST=localhost
DB_PORT=3306
DB_DATABASE=crm
DB_USERNAME=crm_user
DB_PASSWORD=客户自己设的密码
# ... 其他配置
```

### 第 3 步：客户部署并激活

客户部署好系统后：
1. 打开 CRM 网址
2. 系统会弹出"激活"对话框
3. 客户选择 **「私有部署」** 模式
4. 输入你发给他的授权码 `PRIV-A1B2-C3D4-E5F6-G7H8`
5. 点"激活" → 系统激活成功 → 客户可以正常使用

### ⚠️ 交付给客户时绝对不能包含的文件

| 文件 | 原因 |
|------|------|
| `backend/scripts/generate-saas-license.js` | 含 RSA 私钥，能签发 SaaS 许可证 |
| 任何含 RSA 私钥的文件 | 泄露等于放弃 SaaS 保护 |
| 本文档 | 描述了完整的安全机制和操作方法 |

### 如果客户偷偷改 DEPLOY_MODE=saas 会怎样？

**什么都不会发生！** 系统日志会显示：

```
[SaaSGuard] 检测到 DEPLOY_MODE=saas，开始验证 SaaS 平台许可...
[SaaSGuard] ⚠️ SaaS 许可验证失败，自动降级为私有部署模式
[SaaSGuard] 原因: 未提供 SAAS_LICENSE_TOKEN，SaaS 功能不可用
==================================================
配置的部署模式: saas
实际生效模式: private       ← 自动降级了！
⚠️ SaaS 许可验证未通过，已自动降级为私有部署模式
==================================================
```

客户的系统会继续以私有部署模式正常运行，**多租户功能完全不可用**。

| 你想做什么 | 去哪里操作 |
|------------|-----------|
| 给新的 SaaS 租户开账号 | 管理后台 → 租户客户 → 客户列表 → 新增 |
| 给私有部署客户生成授权码 | 管理后台 → 私有客户 → 客户列表 → 新增 |
| 查看某个租户的使用情况 | 管理后台 → 租户客户 → 客户列表 → 点击详情 |
| 暂停/恢复某个租户 | 管理后台 → 租户客户 → 客户列表 → 操作列 |
| 给租户续费/续期 | 管理后台 → 租户客户 → 客户列表 → 操作列 → 续期 |
| 吊销私有客户的授权码 | 管理后台 → 私有客户 → 客户列表 → 操作列 → 吊销 |
| 管理套餐价格和功能 | 管理后台 → 租户客户 → 套餐管理 |
| 生成 SaaS 平台许可证 | **没有界面，用命令行**（见下方说明） |

### 为什么"SaaS 平台许可证"没有界面？

因为这是**你自己服务器的身份证**，只需要做一次，而且涉及 RSA 私钥安全，放在命令行脚本里比放在管理后台里更安全。如果放在管理后台，就意味着管理后台也有私钥，万一管理后台被攻击，私钥就泄露了。

**记住：SaaS 平台许可证只需要在以下情况重新生成：**
- 首次部署 SaaS 服务器时（一次性）
- 许可证快要过期时（按你设的有效期来，建议设 365 天，每年续一次）
- 更换服务器域名时

## 五、常见问题 FAQ

### Q1：私钥存在哪里？

私钥存在 `backend/scripts/generate-saas-license.js` 这个脚本文件里。这个文件：
- 已被 `.gitignore` 排除，不会上传到 Git
- **你自己的开发电脑上有**
- **绝不发给客户**
- 建议同时在安全的地方备份一份（比如加密U盘、密码管理器）

### Q2：现有的 SaaS 租户会受影响吗？

**完全不受影响。** 只要你的 SaaS 服务器 `.env` 里正确配置了 `SAAS_LICENSE_TOKEN`，一切照常运行。租户的授权码验证流程没有任何变化。

### Q3：如果我忘了给 .env 加 SAAS_LICENSE_TOKEN 会怎样？

系统不会崩溃，会自动降级为私有部署模式运行。你的 SaaS 租户暂时无法使用多租户功能。补上 Token 然后重启就恢复了。

### Q4：Token 过期了怎么办？

重新运行生成脚本，得到新 Token，替换服务器 .env 里的旧 Token，重启服务即可。

```bash
cd backend
node scripts/generate-saas-license.js --domain crm.yourdomain.com --expires 365
# 复制新 Token，替换服务器 .env 里的旧 Token
# 重启: pm2 restart crm-backend
```

### Q5：能不能给客户不同的 SaaS 限制？

可以。生成时指定参数：

```bash
# 最多100个租户，绑定特定域名
node scripts/generate-saas-license.js --domain partner.example.com --max-tenants 100 --expires 365
```

### Q6：本地开发怎么测 SaaS 模式？

本地的 `backend/.env.local` 已经配好了一个 10 年有效期的 Token，直接用就行。

### Q7：generate-saas-license.js 脚本有什么可选参数？

```bash
node scripts/generate-saas-license.js --help

# 可选参数：
#   --domain <域名>          绑定域名（多个用逗号分隔）
#   --max-tenants <数量>     最大租户数（0=无限制，默认: 0）
#   --expires <天数>         有效期天数（默认: 365）
#   --generate-keys          重新生成 RSA 密钥对（私钥泄露时使用）
```

### Q8：如果 RSA 私钥泄露了怎么办？

1. 立即重新生成密钥对：
   ```bash
   cd backend
   node scripts/generate-saas-license.js --generate-keys
   ```
2. 将输出的**新公钥**更新到 `backend/src/services/SaaSGuardService.ts` 的 `SAAS_PUBLIC_KEY` 常量
3. 将**新私钥**更新到 `scripts/generate-saas-license.js` 脚本中
4. 用新私钥重新签发你自己的 SaaS 平台许可证
5. 更新服务器 `.env` 中的 `SAAS_LICENSE_TOKEN`
6. 重新部署后端代码（因为公钥改了）
7. 重启服务

旧的 Token 会立即失效（因为公钥变了，旧签名验证不通过）。

## 六、安全机制原理（简要）

```
                    你持有                              代码里硬编码
               ┌──────────────┐                    ┌──────────────┐
               │   RSA 私钥   │ ──── 签名 ────→   │   RSA 公钥   │
               │  (能签发)    │                    │  (只能验证)  │
               └──────────────┘                    └──────────────┘
                      │                                    │
                      ▼                                    ▼
              生成 JWT Token                      验证 JWT Token 签名
         (SAAS_LICENSE_TOKEN)                   (系统启动时自动执行)
                      │                                    │
                      ▼                                    ▼
              配置到 .env                           ✅ 通过 → SaaS 模式
                                                   ❌ 失败 → 强制 private

客户的情况：
  - 客户有代码 → 能看到公钥 → 但公钥只能验证，不能签发
  - 客户没有私钥 → 无法生成有效的 Token
  - 客户伪造的 Token → RSA 签名验证失败 → 系统降级为 private
```

**一句话总结：客户没有你的私钥，他造不出来有效的 Token，改什么环境变量都没用。**

## 七、应急处理

### 场景 1：SaaS 服务器重启后租户无法登录

**排查步骤：**
1. `pm2 logs crm-backend --lines 50` 查看启动日志
2. 搜索 `SaaSGuard` 关键字
3. 如果看到 `SaaS 许可验证失败`，检查 `.env` 中的 `SAAS_LICENSE_TOKEN` 是否完整
4. 如果 Token 过期，重新生成一个

### 场景 2：需要紧急关停某个租户

1. 登录管理后台
2. 租户客户 → 客户列表
3. 找到该租户 → 操作列 → 暂停
4. 租户将立即无法登录

### 场景 3：私有客户声称系统无法激活

1. 登录管理后台
2. 私有客户 → 客户列表
3. 找到该客户的授权码，检查状态是否正常
4. 如果授权码正常，让客户检查网络连接
5. 如果需要，可以重新生成授权码发给客户
