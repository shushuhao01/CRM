## 一、现有普通商品/订单完整生命周期

### 1.1 普通商品生命周期

```
新增商品 -> 上架(active) -> 被订单引用 -> 库存扣减 -> 下架/删除
```

### 1.2 普通订单完整状态流

```
draft(草稿)
  |
  v
pending_transfer(待流转) -- markType: normal/reserved
  |
  | [预留单停在这里，不自动流转]
  | [正常单自动或手动提审]
  v
pending_audit(待审核)
  |
  |-- 审核拒绝 --> audit_rejected --> 可修改重新提审
  |-- 审核通过 --> pending_shipment(待发货)
                      |
                      v
                  shipped(已发货) -- 填写快递公司+单号
                      |
                      v
                  delivered(已签收) -- 物流同步自动更新
                      |
                      |-- 正常完结
                      |-- package_exception(包裹异常)
                      |-- rejected(拒收) --> rejected_returned(拒收已退回)
                      |-- logistics_returned(物流退回)
                      |-- after_sales_created(已建售后)
                      |
                      v
                  cancelled(已取消) / pending_cancel / cancel_failed
```

### 1.3 关联模块触发点

| 节点 | 触发的关联模块 |
|------|--------------|
| pending_shipment | 进入发货列表(Shipping.vue)待发货标签 |
| shipped | 进入已发货列表、物流追踪(List.vue)、状态更新(StatusUpdate.vue) |
| shipped | 进入代收管理(CodCollection.vue) |
| delivered/completed | 绩效统计、增值管理计算 |
| markType变更 | 影响提审逻辑（预留单不自动流转） |

## 二、虚拟商品方案 - 全流程对照审计

### 2.1 虚拟商品生命周期

```
新增虚拟商品 -> 配置发货方式(none/card_key/resource_link)
  |
  |-- none(无需发货): 直接可用，无库存概念
  |-- card_key: 需配置卡密库存(CardKeyManage)
  |-- resource_link: 需配置资源库存(ResourceManage)
  |
  v
上架(active) -> 被订单引用
  |
  |-- card_key: 不扣商品stock，发货时扣卡密库存
  |-- resource_link: 不扣商品stock，发货时扣资源库存
  |-- none: 无需扣库存
```

**审计结果：** 逻辑正确。

**优化建议1：** ~~卡密/资源型商品在下单时应检查可用库存数量~~ **已修正**：下单时即预占卡密/资源，库存递减。库存不足时阻断下单或给出警告。发货时自动匹配预占库存，无库存时支持手动填写兜底。

---

### 2.2 虚拟订单状态流对照

```
                        普通订单                    虚拟订单(无需发货)        虚拟订单(卡密/资源)
                        --------                    ------------------      ------------------
创建                    draft/pending_transfer       draft/pending_transfer   draft/pending_transfer
                              |                            |                        |
标记                    normal/reserved               virtual                  virtual
                              |                            |                        |
提审                    pending_audit                 pending_audit            pending_audit
                              |                            |                        |
审核通过                pending_shipment              ★ completed(直接完成)    ★ virtual_pending(虚拟待发货)
                              |                                                     |
发货                    shipped(填单号)               --                       ★ completed(选卡密/资源后完成)
                              |
物流追踪                delivered/异常/拒收            --                       --
                              |
代收管理                进入代收列表                   ★ 不进入                  ★ 不进入
绩效统计                正常计算                      正常计算                  正常计算
```

**审计发现1 - 状态值冲突问题：**

现有状态枚举中没有 `virtual_pending` 和 `completed`。

- `completed` 在现有代码 statusPriority 中已有（优先级=4，等同delivered），**可复用** ✅
- `virtual_pending` 是新状态，需要新增到以下位置：
  - `Order/List.vue` 的 allOrderStatuses、getStatusText、getStatusType
  - `Order/Detail.vue` 的 statusFlow、statusPriority
  - `Order/Audit.vue` 的审核通过后状态设置逻辑

**审计发现2 - 预留单 + 虚拟的冲突：**

方案中设计虚拟商品订单不能选预留单，这是正确的。但需确认：**编辑已有订单时，如果把普通商品全部删除只剩虚拟商品，预留单标记需自动切换为虚拟发货**。方案中已有watch逻辑处理此场景 ✅。

### 2.3 审核环节审计

| 审计点 | 现有普通流程 | 虚拟方案 | 审计结果 |
|--------|------------|---------|---------|
| 审核方式 | 审核员审核通过/拒绝 | 相同 | ✅ 正确 |
| 审核通过后状态 | pending_shipment | none型:completed, 卡密/资源型:virtual_pending | ✅ 正确 |
| 审核拒绝 | audit_rejected | 相同，可修改重新提审 | ✅ 正确 |
| 撤销审核 | 从pending_shipment撤回 | 从virtual_pending/completed撤回 | ⚠️ 需补充 |
| 批量审核 | 支持 | 需根据订单类型分别处理 | ⚠️ 需补充 |

**优化建议2：** 撤销审核逻辑需要补充：
- 虚拟订单（无需发货）审核通过后状态直接变completed，**撤销审核时需将completed回退到pending_audit**
- 需增加判断：仅当completed是由审核直接导致（非发货导致）时才可撤销
- 建议在订单中增加字段 `completion_source`（值为 `audit_auto_complete` / `virtual_delivery` / `logistics_delivery`），用于区分完成来源

**优化建议3：** 批量审核时，如果勾选的订单中同时包含普通订单和虚拟订单（无需发货），提示用户：「本次批量审核包含 X 个虚拟商品订单（无需发货），审核通过后将直接完成，是否继续？」
