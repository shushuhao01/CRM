## 一、功能概述

在现有CRM系统中新增「虚拟商品」类型，支持无实物发货的商品（如会员卡、激活码、网盘资源等）。虚拟商品与普通商品在录入、下单、审核、发货、财务、绩效等全链路中有差异化处理。

### 2.2 订单管理 - 新增/编辑订单

#### 2.2.1 发货标记扩展

**位置**：`src/views/Order/Edit.vue` 第462-499行的「订单标记」区域

**现有选项**：正常发货单、预留单

**新增选项**：虚拟发货

```
订单类型：
  ○ 正常发货单 [绿色tag]    ← 现有
  ○ 预留单 [黄色tag]        ← 现有
  ○ 虚拟发货 [紫色tag]      ← 新增
```

**互斥逻辑**（核心）：

| 选择的商品组合 | 可选标记 | 不可选标记 | 说明 |
|---------------|---------|-----------|------|
| 仅普通商品 | 正常发货单、预留单 | 虚拟发货（禁用） | 有实物必须走物流 |
| 普通+虚拟混合 | 正常发货单、预留单 | 虚拟发货（禁用） | 含实物按实物订单处理 |
| 仅虚拟商品 | 虚拟发货 | 正常发货单、预留单（禁用） | 纯虚拟走虚拟流程 |

**自动切换逻辑**：
- 当商品列表变化时（添加/删除商品），自动检测商品类型组合
- 如果从「仅虚拟」变为「含普通」，自动切换标记为「正常发货单」
- 如果从「含普通」变为「仅虚拟」，自动切换标记为「虚拟发货」
- 给出 `ElMessage.info` 提示用户标记已自动切换

**实现思路**：
```typescript
// 新增计算属性
const hasPhysicalProduct = computed(() => 
  orderForm.products.some(p => p.productType === 'physical')
)
const hasVirtualProduct = computed(() => 
  orderForm.products.some(p => p.productType === 'virtual')
)
const onlyVirtualProducts = computed(() => 
  orderForm.products.length > 0 && !hasPhysicalProduct.value && hasVirtualProduct.value
)

// watch 商品列表变化，自动调整markType
watch(() => orderForm.products, () => {
  if (onlyVirtualProducts.value && orderForm.markType !== 'virtual') {
    orderForm.markType = 'virtual'
    ElMessage.info('当前仅含虚拟商品，已自动切换为虚拟发货')
  } else if (hasPhysicalProduct.value && orderForm.markType === 'virtual') {
    orderForm.markType = 'normal'
    ElMessage.info('包含实物商品，已自动切换为正常发货单')
  }
}, { deep: true })
```

### 3.1 订单审核流程

#### 3.1.1 审核方式

**位置**：`src/views/Order/Audit.vue`

- 虚拟订单与普通订单使用 **相同的审核流程**，审核方式保持不变

#### 3.1.2 审核通过后流转差异

| 订单类型 | 虚拟发货方式 | 审核通过后去向 | 订单状态 |
|---------|------------|--------------|---------|
| 普通订单 / 混合订单 | — | 物流管理 → 普通待发货 | 待发货 |
| 虚拟订单 | `none`（无需发货） | **直接完成** | 已完成 |
| 虚拟订单 | `card_key` / `resource_link` | 物流管理 → 虚拟待发货 | 虚拟待发货 |

### 3.6 数据库设计（阶段2补充）

#### 3.6.1 virtual_delivery_records 虚拟发货记录表

```sql
CREATE TABLE virtual_delivery_records (
  id VARCHAR(36) PRIMARY KEY,
  order_id VARCHAR(36) NOT NULL COMMENT '订单ID',
  delivery_type VARCHAR(20) NOT NULL COMMENT '发货类型: none/card_key/resource_link',
  card_key_content TEXT DEFAULT NULL COMMENT '卡密内容（加密存储）',
  resource_link VARCHAR(500) DEFAULT NULL COMMENT '资源链接',
  resource_password VARCHAR(100) DEFAULT NULL COMMENT '提取码',
  remark TEXT DEFAULT NULL COMMENT '备注说明',
  operator_id VARCHAR(36) NOT NULL COMMENT '操作人ID',
  operator_name VARCHAR(50) DEFAULT NULL COMMENT '操作人姓名',
  delivered_at DATETIME NOT NULL COMMENT '发货时间',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_order_id (order_id),
  FOREIGN KEY (order_id) REFERENCES orders(id)
) COMMENT '虚拟商品发货记录表';
```

## 五、需求补充（2026-04-19 第二轮）

### 5.1 虚拟商品创建流程细化

#### 5.1.1 无需发货型（服务方案类）

- 选择「无需发货」后，表单只需填写：基本信息 + 价格 + 状态，即可直接提交
- 无需配置卡密、资源链接、库存等字段
- 适用场景：服务方案、咨询服务等，仅做下单记录和业绩统计
- 此类商品无库存概念，可无限下单

#### 5.1.2 卡密发货型 - 卡密库存管理

- 选择「卡密发货」后，保存商品基本信息后，弹出卡密配置对话框（也可后续在虚拟商品管理中配置）
- **卡密唯一性**：每个卡密是唯一编码，系统自动校验不可重复
- **卡密状态**：`未使用`（可用）/ `已使用`（已关联订单发货）/ `已过期`（手动或自动标记）
- **一旦卡密关联发货了订单，该卡密状态变为「已使用」，不再可用**
- **批量导入**：支持下载Excel模板，按模板一行一个卡密批量上传
- **使用说明**：每个商品可配置统一的卡密使用说明（富文本），发货时展示给客户

#### 5.1.3 网盘资源型 - 资源库存管理

- 选择「网盘资源」后，保存商品基本信息后，配置资源链接和提取码
- **资源条目**：每条资源包含链接 + 提取码 + 使用说明，同样是唯一性
- 同卡密逻辑：已关联发货则标记为「已使用」
- 支持批量导入

## 六、补充3 - 虚拟发货流程深化（2026-04-20）

### 6.1 库存递减时机变更

**原方案**：发货时扣减卡密/资源库存
**修正为**：**下单时即递减库存**，取消订单时恢复

| 时机 | 操作 | 库存变化 |
|------|------|---------|
| 下单提交 | 系统自动预占一个可用卡密/资源 | 可用数 -1，预占数 +1 |
| 取消订单 | 释放预占的卡密/资源 | 可用数 +1，预占数 -1 |
| 发货确认 | 预占转为已使用 | 预占数 -1，已使用 +1 |
| 售后退款（未领取） | 回退为可用 | 已使用 -1，可用数 +1 |
| 售后退款（已领取） | 标记作废 | 已使用 -1，已作废 +1 |

卡密状态扩展为：`unused`(未使用) / `reserved`(已预占) / `used`(已使用-已发货) / `claimed`(已领取) / `expired`(已过期) / `voided`(已作废)

### 6.2 虚拟发货 - 自动匹配 + 手动兜底

#### 6.2.1 发货流程

```
点击[发货]按钮
    |
    v
系统自动匹配该商品的预占卡密/资源（下单时已预占的那条）
    |
    |-- 有预占记录 --> 自动填充卡密/资源到发货弹窗
    |                    |
    |                    v
    |               支持修改（换一个其他可用卡密）
    |                    |
    |                    v
    |               点击[确认发货]
    |
    |-- 无预占记录（异常/旧数据） --> 两种处理方式：
    |       |
    |       |-- 库存有可用卡密 --> 自动分配一个，填充到弹窗
    |       |-- 库存无可用卡密 --> 显示手动填写输入框
    |                               用户手动输入卡密/链接
    |                               系统将手动输入的卡密存入库存表（标记为已使用）
    |
    v
确认发货后：
  1. 卡密状态: reserved -> used
  2. 订单状态: virtual_pending -> completed
  3. completion_source: 'virtual_delivery'
  4. 弹出领取链接弹窗（见6.3）
  5. 发送系统消息通知
```

- 部分无预占：提示「X个订单库存不足，需手动处理」，仅发货有库存的
- 批量发货后，弹出汇总弹窗显示每个订单的领取链接，支持一键复制全部

### 6.9 邮件发送功能设计

### 8.2 邮件配置完善 - 双邮箱来源

#### 8.2.1 邮箱来源选择
