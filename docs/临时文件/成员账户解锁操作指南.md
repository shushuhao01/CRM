# 成员账户解锁操作指南

## 问题说明
当成员登录失败次数超过5次时,账户会被自动锁定,状态变为 `locked`。

## 数据库字段说明

在 `users` 表中,与账户锁定相关的字段有:

- `status`: 账户状态 (`active`/`inactive`/`resigned`/`locked`)
- `login_fail_count`: 登录失败次数
- `locked_at`: 账户锁定时间
- `tenant_id`: 租户ID (多租户系统)

## 解锁方法

### 方法一: 通过SQL直接解锁 (推荐)

```sql
-- 1. 查询被锁定的用户
SELECT id, username, name, status, login_fail_count, locked_at, tenant_id
FROM users
WHERE status = 'locked';

-- 2. 解锁指定用户 (根据用户名)
UPDATE users
SET status = 'active',
    login_fail_count = 0,
    locked_at = NULL
WHERE username = '用户名';

-- 3. 解锁指定用户 (根据用户ID)
UPDATE users
SET status = 'active',
    login_fail_count = 0,
    locked_at = NULL
WHERE id = '用户ID';

-- 4. 批量解锁所有被锁定的用户
UPDATE users
SET status = 'active',
    login_fail_count = 0,
    locked_at = NULL
WHERE status = 'locked';

-- 5. 解锁指定租户下的用户
UPDATE users
SET status = 'active',
    login_fail_count = 0,
    locked_at = NULL
WHERE username = '用户名'
  AND tenant_id = '租户ID';
```

### 方法二: 通过宝塔面板操作

1. 登录宝塔面板
2. 进入 **数据库** 管理
3. 找到对应的数据库,点击 **管理**
4. 点击 **SQL** 标签
5. 执行上述SQL语句

### 方法三: 通过Navicat等数据库工具

1. 连接到数据库
2. 找到 `users` 表
3. 筛选 `status = 'locked'` 的记录
4. 直接修改字段:
   - `status` 改为 `active`
   - `login_fail_count` 改为 `0`
   - `locked_at` 改为 `NULL`

## 验证解锁结果

```sql
-- 查询用户当前状态
SELECT id, username, name, status, login_fail_count, locked_at
FROM users
WHERE username = '用户名';
```

预期结果:
- `status` = `active`
- `login_fail_count` = `0`
- `locked_at` = `NULL`

## 注意事项

1. **多租户系统**: 如果是多租户系统,需要指定 `tenant_id` 避免误操作其他租户的用户
2. **备份数据**: 执行UPDATE操作前建议先备份数据或记录原始值
3. **通知用户**: 解锁后建议通知用户修改密码,避免再次被锁定
4. **查看日志**: 可以查看 `locked_at` 字段了解锁定时间

## 预防措施

1. 提醒用户妥善保管密码
2. 考虑调整登录失败次数限制 (当前为5次)
3. 可以在系统设置中配置锁定策略

## 相关代码位置

## 常见问题

**Q: 解锁后用户还是无法登录?**
A: 检查以下几点:
- 确认密码是否正确
- 检查 `status` 是否确实为 `active`
- 清除浏览器缓存和Cookie
- 检查是否有IP限制 (`authorized_ips` 字段)

**Q: 如何查看所有被锁定的用户?**
A: 
```sql
SELECT id, username, name, phone, email, locked_at, tenant_id
FROM users
WHERE status = 'locked'
ORDER BY locked_at DESC;
```

**Q: 如何临时禁用账户锁定功能?**
A: 需要修改后端代码 `backend/src/controllers/UserController.ts` 中的登录逻辑,不建议在生产环境这样做。
