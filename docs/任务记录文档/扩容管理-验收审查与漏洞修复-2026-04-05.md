## 二、发现问题与修复

### 🔴 严重问题 (3个)

#### 问题 #1: `activateCapacity()` 竞态条件

**描述**: 使用 SELECT + UPDATE 两步操作，并发支付回调可能导致重复激活扩容额度。

**位置**: `CapacityService.ts` → `activateCapacity()`

**修复**: 改为原子化 CAS（Compare-And-Swap）操作：
```typescript
// 之前: SELECT ... WHERE status='pending' → UPDATE ... WHERE id=?
// 修复: UPDATE ... WHERE order_no=? AND status='pending' → 检查 affectedRows
const updateResult = await AppDataSource.query(
  `UPDATE capacity_orders SET status = 'paid', paid_at = NOW() WHERE order_no = ? AND status = 'pending'`,
  [orderNo]
);
if (!updateResult?.affectedRows || updateResult.affectedRows === 0) {
  return { success: false, message: '订单不存在或已处理' };
}
```

---

#### 问题 #2: 自动关闭逻辑误杀扩容订单

**描述**: 扩容订单支付后，会自动关闭同租户**所有**其他待支付的扩容订单。但用户数扩容和存储空间扩容是**独立购买项**，不应互相关闭。

**场景**:
1. 租户创建「扩容用户数 x5」订单（CAP...001）
2. 租户创建「扩容存储空间 10GB」订单（CAP...002）
3. 支付 CAP...001 → ❌ 自动关闭了 CAP...002（存储扩容订单被误杀）

**位置**: `PaymentService.ts` → `updateOrderStatus()` 自动关闭逻辑

**修复**: 扩容订单支付后不再自动关闭其他扩容订单，仅套餐订单才自动关闭同类：
```typescript
if (!isCapacityOrder) {
  // 仅套餐订单：关闭同租户其他待支付的非扩容订单
  await AppDataSource.query(
    `UPDATE payment_orders SET status = 'closed' WHERE tenant_id = ? AND status = 'pending' AND order_no != ? AND order_no NOT LIKE 'CAP%'`,
    [tenantId, orderNo]
  );
}
// 扩容订单不自动关闭（用户数和存储空间独立购买）
```

---

#### 问题 #3: 扩容类型与价格配置不匹配校验缺失

**描述**: `createOrder()` 未验证 `data.type` 是否与价格配置的 `config.type` 一致。恶意请求可以用存储空间的价格购买用户数扩容（或反之）。

**位置**: `CapacityService.ts` → `createOrder()`

**修复**: 添加类型匹配校验：
```typescript
if (data.type !== config.type) {
  return { success: false, message: '扩容类型与价格配置不匹配' };
}
```

---

### 🟡 中等问题 (4个)

#### 问题 #4: 退款状态语义不一致

**描述**: 扩容退款后 `capacity_orders.status` 设为 `'closed'`，但 `payment_orders.status` 为 `'refunded'`，语义混淆。

**修复**:
- `capacity_orders` ENUM 新增 `'refunded'` 值
- `ensureTables()` 添加自动迁移 `ALTER TABLE ... MODIFY COLUMN`
- 退款路由中 `capacity_orders` 状态改为 `'refunded'`
- 前端 `Capacity.vue` 增加退款状态显示和样式

---

#### 问题 #5: 导出CSV缺少订单类型列

**描述**: CSV导出无法区分扩容订单和套餐订单。

**修复**: CSV header 和数据行中新增"订单类型"列，根据 `order_no.startsWith('CAP')` 判断。

---

#### 问题 #6: 价格配置缺少业务校验

**描述**: `savePriceConfig()` 未校验 `unit_price > 0` 和 `min_qty <= max_qty`。

**修复**: 添加三项校验：
- 扩容类型必须为 `'user'` 或 `'storage'`
- 单价必须大于 0
- 最小购买量不能大于最大购买量

---

#### 问题 #7: 未校验租户状态

**描述**: 已暂停/已过期的租户仍可创建扩容订单并付款。

**修复**: `createOrder()` 添加前置校验：
- 租户不能是 suspended/disabled/deleted 状态
- 套餐不能已过期（expire_date < 当前时间）

## 五、完整功能流程验证清单

```
1. 管理端价格配置
   Admin → 套餐管理 → 扩容管理 → 新增/编辑/删除/启禁用配置 → ✅

2. 会员端浏览价格
   官网 → 会员中心 → 扩容管理 → 查看可用价格列表 → ✅

3. 创建扩容订单
   选择类型 → 选价格 → 输数量 → 选支付方式 → 创建订单
   → 校验租户状态 → 校验类型匹配 → 双表写入 → ✅

4. 支付流程
   repay生成二维码 → 扫码支付 → 回调 → CAP识别 →
   activateCapacity(原子CAS) → extra_users/extra_storage_gb累加 →
   max_users/max_storage_gb同步 → ✅

5. 支付后权限同步
   CRM前端检查 max_users → 允许新增用户 → ✅
   会员中心 profile → 显示含扩容额度 → ✅

6. 管理端订单管理
   支付列表 → 类型筛选 → 扩容/套餐Tag → 详情查看 →
   退款(扩容:扣额度, 套餐:暂停授权) → 关闭(双表同步) → ✅

7. CSV导出
   包含订单类型列 → 区分扩容/套餐 → ✅

8. 续费场景
   套餐续费 → max_users = 新套餐基础 + COALESCE(extra_users, 0) → ✅
```
