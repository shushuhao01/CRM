## 🔴 二、数据隔离安全审计 - 问题清单

### 问题 P1：wecom.ts（企业微信）完全未做租户隔离 【严重】

**风险等级**: 🔴 严重  
**文件**: `backend/src/routes/wecom.ts`  
**问题描述**:
- 整个企微路由文件（814行）完全没有任何租户过滤
- 使用 `AppDataSource.getRepository(WecomConfig)` 而非 `getTenantRepo`
- `configRepo.find({ where: { isEnabled: true } })` 会返回所有租户的配置
- 13处 `findOne` 调用均未添加 `tenant_id` 过滤
- 企微用户绑定、客户数据、聊天记录全部暴露

**影响**: 租户A可能看到/操作租户B的企业微信配置和客户数据

**修复方案**:
```typescript
// 将所有 AppDataSource.getRepository(WecomConfig) 替换为：
import { getTenantRepo } from '../utils/tenantRepo';
const configRepo = getTenantRepo(WecomConfig);
// 或使用 BaseRepository
```

### 问题 P2：app.ts 定时任务消息清理未做租户隔离 【中等】

**风险等级**: 🟡 中等  
**文件**: `backend/src/app.ts` (第352-386行)  
**问题描述**:
```typescript
// 当前代码 - 会删除所有租户的30天前消息
const result = await messageRepo
  .createQueryBuilder()
  .delete()
  .where('created_at < :date', { date: thirtyDaysAgo })
  .execute();
```
- 清理任务不区分租户，一次删除所有租户的过期消息
- SaaS模式下，这实际上是正常行为（统一清理老数据）
- 但如果不同租户有不同的消息保留策略，则存在问题

**修复方案**: 在私有部署模式下此问题不影响（只有一个租户），可标记为低优先级。SaaS模式下建议按tenant分别配置保留策略。

### 问题 P3：orders.ts 部分查询使用 AppDataSource.getRepository 【中等】

**风险等级**: 🟡 中等  
**文件**: `backend/src/routes/orders.ts`  
**问题描述**:
- 订单路由中，在获取客户信息时使用了 `AppDataSource.getRepository(Customer)` 而非租户仓储
- `customerRepository.findByIds(customerIds)` 不会自动添加租户过滤
- 这可能导致通过客户ID跨租户获取客户信息

**具体位置**:
```typescript
// 多处出现（约4处）:
const customerRepository = AppDataSource.getRepository(Customer);
const customers = customerIds.length > 0
  ? await customerRepository.findByIds(customerIds)  // ⚠️ 无租户过滤
  : [];
```

**修复方案**:
```typescript
const customerRepository = getTenantRepo(Customer);
// 或使用带租户过滤的查询
```

### 问题 P4：license.ts 用户数统计未添加租户过滤 【低】

**风险等级**: 🟢 低  
**文件**: `backend/src/routes/license.ts` (第183行)  
**问题描述**:
```typescript
const userCountResult = await AppDataSource.query(
  `SELECT COUNT(*) as count FROM users WHERE status = 'active'`
);
```
- 统计活跃用户数时未过滤`tenant_id`，会统计所有租户的用户

**修复方案**: 私有部署下影响不大（只有一个租户），SaaS模式下应添加`AND tenant_id IS NULL`或按当前租户过滤。

### 问题 P5：create-admin-user.js 查找admin用户未加租户过滤 【低】

**风险等级**: 🟢 低  
**文件**: `backend/create-admin-user.js` (第21行)  
**问题描述**:
```javascript
const existingAdmin = await userRepo.findOne({
  where: { username: 'admin' }
});
```
- 若多租户下不同租户都有`admin`用户名，此脚本可能找到错误的用户

**修复方案**: 添加 `tenantId: null` 条件，确保只查找非租户的admin用户。

### 问题 P6：数据库唯一约束缺少租户维度 【重要】

**风险等级**: 🟠 重要  
**文件**: `database/schema.sql`  
**问题描述**:
- `roles`表的`code`字段设置了`UNIQUE`，但未加`tenant_id`复合唯一索引
  - 不同租户不能有相同角色代码，这是设计缺陷
- `users`表没有 `(tenant_id, username)` 复合唯一索引
  - 当前依赖业务逻辑（login时按tenant+username查找），但数据库层面未约束
  - 可能导致同一租户下创建重复用户名

**修复方案**:
```sql
-- 角色表：改为租户级唯一
ALTER TABLE roles DROP INDEX code;
ALTER TABLE roles ADD UNIQUE INDEX uk_tenant_role_code (tenant_id, code);

-- 用户表：添加租户级用户名唯一
ALTER TABLE users ADD UNIQUE INDEX uk_tenant_username (tenant_id, username);
```

### 5.3 数据库唯一约束修复SQL

```sql
-- 1. 用户表：添加租户级用户名唯一约束
-- 先检查是否存在旧的唯一索引
SET @index_exists = (SELECT COUNT(*) FROM information_schema.STATISTICS 
  WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 'users' AND INDEX_NAME = 'uk_tenant_username');
SET @sql = IF(@index_exists = 0, 
  'ALTER TABLE users ADD UNIQUE INDEX uk_tenant_username (tenant_id, username)', 
  'SELECT 1');
PREPARE stmt FROM @sql;
EXECUTE stmt;
DEALLOCATE PREPARE stmt;

-- 2. 角色表：修改code唯一约束为租户级
ALTER TABLE roles DROP INDEX IF EXISTS code;
ALTER TABLE roles ADD UNIQUE INDEX uk_tenant_role_code (tenant_id, code);
```

## 📌 八、宝塔面板部署要点

### 8.1 架构图
```
用户浏览器 → Nginx(宝塔) → 前端静态文件(dist/)
                        → 反向代理 → Node.js Backend(:3000)
                        → MySQL 8.0
```

### 8.2 关键Nginx配置
```nginx
server {
    listen 80;
    server_name your-domain.com;
    
    # 前端静态文件
    location / {
        root /www/wwwroot/crm/dist;
        try_files $uri $uri/ /index.html;
    }
    
    # API反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
    
    # WebSocket代理
    location /socket.io/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
    
    # 文件上传
    location /uploads/ {
        alias /www/wwwroot/crm/backend/uploads/;
    }
}
```
