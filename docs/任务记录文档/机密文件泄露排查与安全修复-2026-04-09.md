## 四、远程仓库历史清理方案

### ⚠️ 重要：以下操作需要你确认后执行

远程仓库（backup: https://github.com/shushuhao01/backups.git）的 Git 历史记录中仍然包含所有已提交的机密文件。即使在新的提交中删除了这些文件，之前的提交历史仍然可以查看到完整内容。

### 方案一：使用 `git filter-branch` 或 `BFG Repo-Cleaner` 清理历史（推荐）

**步骤：**
1. 安装 BFG Repo-Cleaner（比 filter-branch 快10-720倍）
2. 创建一个需要清除的敏感文件列表
3. 执行 BFG 清理历史
4. 强制推送到远程仓库
5. 通知所有协作者重新 clone

**命令示例：**
```bash
# 1. 下载 BFG（Java jar包）
# https://rtyley.github.io/bfg-repo-cleaner/

# 2. 备份当前仓库
git clone --mirror https://github.com/shushuhao01/backups.git backups-mirror

# 3. 清除特定文件
java -jar bfg.jar --delete-files .env.production.server backups-mirror
java -jar bfg.jar --delete-files generate-saas-license.js backups-mirror

# 4. 清除包含密钥的文本
# 创建 passwords.txt 文件，每行一个要替换的密钥值
java -jar bfg.jar --replace-text passwords.txt backups-mirror

# 5. 清理和推送
cd backups-mirror
git reflog expire --expire=now --all
git gc --prune=now --aggressive
git push --force
```

### 方案二：删除远程仓库并重建（最简单但丢失历史）

**步骤：**
1. 在 GitHub 上删除 shushuhao01/backups 仓库
2. 重新创建同名空仓库
3. 本地清理敏感文件后重新推送
4. 确保 .gitignore 已更新

### 方案三：将备份仓库改为私有（临时应急）

**步骤：**
1. 在 GitHub Settings 中将仓库设为 Private
2. 后续再用方案一或二彻底清理

## 五、密钥更换优先级清单

| 优先级 | 密钥 | 操作 | 影响 |
|--------|------|------|------|
| P0 | 数据库密码 | 服务器MySQL: `ALTER USER 'abc789'@'localhost' IDENTIFIED BY '新密码';` | 需更新 .env |
| P0 | JWT Secret | 生成新密钥: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` | 所有用户需重新登录 |
| P0 | JWT Refresh Secret | 同上 | 所有刷新令牌失效 |
| P1 | RSA 私钥 | 运行: `node backend/scripts/generate-saas-license.js --generate-keys` | 需重新签发所有SaaS许可证 |
| P2 | 更换仓库地址 | 如有必要，迁移到新的私有仓库 | 需更新所有 remote 配置 |

## 六、预防措施建议

1. **pre-commit hook**: 添加 Git hook 检测 .env 文件和私钥文件的提交
2. **GitHub Secret Scanning**: 启用 GitHub 的密钥扫描功能
3. **仓库权限**: 确保备份仓库设为 Private
4. **定期审计**: 每月检查一次 .gitignore 是否覆盖所有敏感文件
5. **大模型操作规范**: 指定不允许 AI 模型提交机密文件的规则清单
