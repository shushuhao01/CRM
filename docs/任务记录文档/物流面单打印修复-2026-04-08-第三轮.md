# 物流面单打印修复 - 2026-04-08 第三轮

## 修复内容

### 问题1：LODOP已安装连接但打单默认显示浏览器打印
**现象：** LODOP控件已安装并成功连接，但在单个打单和批量打单对话框中，打印机选择默认显示"浏览器打印"，没有加载用户设置的默认打印方式。

**修复方案：**
1. **`printService.ts`** 新增打印方式管理函数：
   - `getDefaultPrintMode()` - 获取默认打印方式（browser/lodop/usb）
   - `setDefaultPrintMode()` - 设置默认打印方式
   - `getDefaultLodopPrinter()` - 获取默认LODOP打印机名称
   - `setDefaultLodopPrinter()` - 保存默认LODOP打印机名称

2. **`PrinterConfigDialog.vue`** 配置界面改进：
   - 每种打印方式卡片下方添加 **"设为默认"** 和 **"取消默认"** 按钮
   - 默认打印方式卡片显示金色边框和"默认"标签
   - LODOP区域增加 **"保存为默认LODOP打印机"** 按钮和默认打印机显示
   - 切换模式时自动同步保存

3. **`PrintLabelDialog.vue` / `BatchPrintLabelDialog.vue`** 打单界面适配：
   - 初始化时读取默认打印方式（`getDefaultPrintMode()`）
   - 如果默认方式是LODOP，显示LODOP标签和打印机名称
   - 如果默认方式是USB，显示USB标签
   - 如果是浏览器打印，显示原有的打印机下拉选择
   - 打印机配置弹窗关闭后自动刷新打印方式和打印机

### 问题2：全屏预览打印后确认发货弹窗不触发
**现象：** 在全屏预览窗口点击"打印全部面单"后，浏览器打印对话框打开并执行打印，但返回系统时没有弹出确认发货的对话框。

**根因分析：** 原来只依赖 `BroadcastChannel` 和 `window.opener.postMessage` 两个通道来检测打印完成。部分浏览器/环境下这两个通道可能不可靠。

**修复方案：** 增加4通道检测机制：
1. **BroadcastChannel** - 最可靠的跨窗口通信
2. **postMessage** - opener.postMessage 备用
3. **窗口关闭轮询** - `setInterval` 每秒检测 `previewWin.closed`，窗口关闭后500ms触发确认
4. **主窗口 focus 事件** - 用户切回主窗口时，如果预览窗口已关闭，触发确认

**适用范围：** `PrintLabelDialog.vue` 和 `BatchPrintLabelDialog.vue` 的 `handlePreview` / `handlePreviewAll` 方法。

### 问题3：面单模板选了小面单但全屏预览和打印还是旧模板
**现象：** 在面单模板管理中选择了小面单（如76×130mm），对话框内预览正确，但全屏预览和打印输出的纸张尺寸仍然是旧的默认大小（如100×180mm）。

**根因分析：** `previewLabels()` 和 `printLabels()` 接收的 `config` 参数来自打印机配置（PrinterConfig），其纸张尺寸与模板的纸张尺寸是独立的。调用时没有用模板的纸张尺寸覆盖打印机配置。

**修复方案：**
- 新增 `getConfigWithTemplatePaper()` 辅助函数
- 在调用 `previewLabels` / `printLabels` 前，用当前选中模板的 `paperSize` 覆盖打印机配置的纸张参数
- 单个打印：使用 `currentTemplate.value?.paperSize`
- 批量打印：使用第一个可打印订单的模板纸张尺寸

### 问题4：物流公司和运单号不匹配没有提示
**现象：** 订单选择了物流公司A，但填写了物流公司B的运单号（如选了中通但填了SF开头的单号），系统没有任何提示即可预览和打单。

**修复方案：**
1. **`printService.ts`** 新增 `validateTrackingCompanyMatch()` 函数：
   - 内置物流公司运单号前缀规则表（顺丰SF、圆通YT、京东JD、EMS E*、极兔JT等）
   - 检查单号是否明确属于其他物流公司（跨公司前缀匹配）
   - 检查当前物流公司的单号前缀规则是否符合
   - 返回 `{ valid, warning }` 结果

2. **`PrintLabelDialog.vue`** 单个打单校验：
   - 运单号/物流公司变化时实时校验，显示 `el-alert` 警告
   - 全屏预览和打印前执行校验，不匹配时弹出确认对话框让用户选择"仍然继续"或"返回修改"
   - 新增 `trackingValidationWarning` 响应式变量

3. **`BatchPrintLabelDialog.vue`** 批量打单校验：
   - 订单列表状态栏增加"不匹配"状态显示
   - 批量预览/打印前检查所有订单，汇总不匹配项
   - 弹出确认对话框列出不匹配订单详情

4. **LODOP打印支持：**
   - `handlePrint` / `handleBatchPrint` 中，根据 `currentPrintMode` 选择打印方式
   - LODOP模式下使用 `lodopPrintHTML()` 直接发送到LODOP打印机
   - 浏览器模式下使用原有的 `printLabels()` 流程

## 确认发货弹窗检测4通道机制

```
通道1: BroadcastChannel ('crm-label-print')
  ↓ 收到 label-print-done 消息
通道2: window.addEventListener('message')
  ↓ 收到 postMessage 消消息
通道3: setInterval 轮询 previewWin.closed
  ↓ 检测到预览窗口关闭，延迟500ms
通道4: window.addEventListener('focus')
  ↓ 用户切回主窗口且预览窗口已关闭
  ↓
handlePrintDone() → ElMessageBox.confirm("确认打印结果")
```

任何一个通道触发即弹出确认对话框，`handled` 标志防止重复触发。


# 物流面单打印修复 - 2026-04-08 第四轮

## 修复内容

### 问题1：全屏预览窗口打印方式不跟随默认设置
**现象：** 用户在打单对话框选了LODOP打印方式，但全屏预览窗口的"打印全部面单"按钮始终使用浏览器`window.print()`。

**修复方案：**
1. **`printService.ts` `generatePrintPageHTML`** 新增 `defaultPrintMode` 参数：
   - 预览工具栏增加打印方式下拉选择（`<select>`）：浏览器打印 / LODOP打印
   - 默认选中与打单对话框一致的打印方式
   - 浏览器模式：使用 `window.print()`
   - LODOP模式：通过 `BroadcastChannel` + `postMessage` 发送 `{ type: 'lodop-print-request' }` 给父窗口
   - 监听父窗口回复的 `lodop-print-result` 消息，显示成功/失败状态
   - LODOP模式按钮为绿色风格，浏览器模式为蓝色

2. **`previewLabels` 函数** 新增 `printMode` 参数，传递给 `generatePrintPageHTML`

3. **`PrintLabelDialog.vue` / `BatchPrintLabelDialog.vue`**：
   - `handlePreview` / `handlePreviewAll` 传入 `currentPrintMode.value`
   - 新增 `handleLodopPrintRequest` 处理函数：收到预览窗口的 LODOP 打印请求后，调用 `lodopPrintHTML` 执行 LODOP 打印，并回复结果

### 问题2：LODOP打印排版问题（CSS缺失）
**现象：** LODOP打印出来没有排版，纯文本。

**根因：** `lodopPrintHTML` 传入的 `htmlContent` 只是面单HTML片段（来自 `generateLabelHTML`），没有包含 CSS 样式（`getLabelCSS`）。

**修复方案：**
1. **新增 `generateLodopFullHTML` 函数**（`printService.ts`）：
   - 将 `getLabelCSS()` 生成的完整CSS包裹在 `<style>` 标签中
   - 与面单HTML合并为完整 `<!DOCTYPE html>` 文档
   - 确保 `.sl-tag-recv` / `.sl-tag-send` 的背景色使用 `!important` + `print-color-adjust: exact`

2. **`PrintLabelDialog.vue` 和 `BatchPrintLabelDialog.vue`**：
   - `handlePrint` / `handleBatchPrint` 的 LODOP 分支使用 `generateLodopFullHTML(htmlLabels, config)` 替代裸HTML拼接
   - 预览窗口的 LODOP 打印请求处理同样使用 `generateLodopFullHTML`

### 问题3：发货列表增加"确认发货"功能
**场景：** 用户打印了面单后，确认发货弹窗被不小心关闭，但实际已经发货了，没有地方可以重新确认。

**修复方案：**
1. **"更多"下拉菜单** 新增"确认发货"选项（`Shipping.vue`）：
   - 位于"导出"之前
   - 只有有运单号（`expressNo`）且未发货的订单才可点击
   - 点击后弹出确认框，显示订单号和运单号
   - 确认后调用 `orderApi.update` 标记为已发货

2. **头部操作栏** 新增"批量确认发货"按钮：
   - 计算属性 `confirmableSelectedOrders`：从已选订单中筛选有运单号且未发货的
   - 按钮显示可确认数量：`批量确认发货 (N)`
   - 点击后循环更新所有符合条件的订单状态

3. **`handleCommand`** 新增 `confirmShip` case

### 问题4：API 500 错误（logistics/api-configs/SF）
**现象：** 点击"获取单号"时，`GET /api/v1/logistics/api-configs/SF` 返回 500。

**根因：** 后端 `getTenantRepo(LogisticsApiConfig)` 在数据库表不存在时抛出异常，被 catch 后统一返回 500，前端 axios 对 500 状态码直接抛异常。

**修复方案：**
1. **后端** `logisticsStatus.ts`：
   - `/api-configs/:companyCode` 路由：配置不存在时改用 `200 + { success: false, data: null }` 替代 `404`
   - catch 块中检测表不存在错误（`no such table` / `ER_NO_SUCH_TABLE` / `SQLITE_ERROR`），返回 `200 + { success: false }` 替代 500
   - `/api-configs` 列表接口做同样处理，表不存在时返回空数组

2. **前端** `logistics.ts` `checkCreateOrderSupport`：
   - 在 `getApiConfig` 调用外包裹独立 try-catch
   - HTTP 404 / 500 视为"未配置"，返回 `supported: false` + 友好提示
   - 避免错误向上传播导致用户看到"网络错误"
