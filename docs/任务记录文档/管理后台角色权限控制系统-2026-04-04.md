# 管理后台角色权限控制系统 - 任务记录

> 完成日期：2026-04-04
> 任务类型：功能开发
> 影响范围：管理后台(admin) + 后端(backend)

---

## 一、任务背景

管理后台(admin)已基本开发完成，所有登录用户可以看到全部菜单和功能按钮，缺少角色和权限控制机制。运营方后续需要让不同角色（如销售员、技术支持人员等）登录管理后台时，只能看到和操作其权限范围内的功能。

### 原有状况
- AdminUser 实体仅有 `role` 枚举字段（`super_admin` / `admin` / `operator`），无法自定义角色
- 前端无菜单过滤、无按钮权限控制
- 所有登录用户可以访问全部9个一级菜单和所有二级菜单
- 后端仅在少数接口做了 `super_admin` 角色检查

---

## 二、实现方案

### 2.1 权限体系设计

采用 **RBAC（基于角色的访问控制）** 模型：

```
管理员用户(AdminUser) → 关联角色(AdminRole) → 角色包含权限码列表(permissions)
```

权限码采用三段式命名：`模块:子模块:操作`

- 菜单权限示例：`customer-management:all:view`（查看所有客户）
- 按钮权限示例：`settings:admins:create`（新建管理员）

### 2.2 内置角色（4个）

| 角色 | 标识码 | 权限范围 |
|---|---|---|
| 超级管理员 | `super_admin` | 全部权限 `["*"]` |
| 运营管理员 | `operation_admin` | 客户管理、支付管理、版本查看、操作日志 |
| 销售员 | `sales` | 仅客户查看权限 |
| 技术支持 | `tech_support` | 模块管理、版本管理、接口管理 |

### 2.3 权限码总览（60+项）

| 一级菜单 | 二级菜单 | 操作权限 |
|---|---|---|
| 仪表盘 | dashboard:view | - |
| 客户管理 | all:view, renewal:view | all:edit, all:followup |
| 私有客户 | list:view, detail:view | list:create/edit/delete, detail:renew/revoke/reset-password |
| 租户客户 | list:view, packages:view, detail:view | list:create/edit/delete, packages:create/edit/delete, detail:renew/suspend/reset-password |
| 模块服务 | list/config/distribute/message:view | list:create/edit/delete, config:edit, distribute:edit, message:edit |
| 支付管理 | list/config/reports:view | list:edit, config:edit, reports:export |
| 版本发布 | list/upload/changelog:view | list:edit/delete, upload:edit, changelog:edit |
| 接口管理 | list:view | list:create/edit/delete |
| 系统设置 | admins/roles/basic/notification-templates/notification-service/operation-logs/recycle-bin:view | admins:create/edit/delete, roles:create/edit/delete, basic:edit, notification-templates:edit, notification-service:edit, recycle-bin:restore/delete |

## 四、使用指南

### 4.1 部署步骤

1. **执行数据库迁移**：运行 `backend/database-migrations/20260404-admin-role-permission-system.sql`
2. **重启后端服务**：使新的实体和路由生效
3. **重新构建前端**：管理后台前端需要重新 build

### 4.2 管理角色

进入 **系统设置 → 角色管理**：
- 可新建自定义角色，通过权限树勾选分配权限
- 系统内置角色（超级管理员、运营管理员、销售员、技术支持）不可删除
- 超级管理员角色的权限不可修改

### 4.3 分配角色给用户

进入 **系统设置 → 管理员账号**：
- 新建/编辑管理员时，选择"用户类型"和"分配角色"
- 超级管理员类型自动拥有全部权限，无需分配角色
- 管理员和操作员类型需要分配角色才有访问权限

### 4.4 权限控制效果

- **菜单过滤**：用户只能看到有权限的一级菜单和二级菜单
- **按钮控制**：无权限的操作按钮自动隐藏
- **路由守卫**：直接访问无权限的URL会被重定向到首页
- **仪表盘**：所有用户默认可见（即使无 dashboard:view 权限也允许进入）

### 4.5 在其他页面使用权限控制

**按钮级权限控制（使用 v-permission 指令）：**
```vue
<!-- 单个权限 -->
<el-button v-permission="'private-customers:list:create'">新建客户</el-button>

<!-- 多个权限（任一满足即显示） -->
<el-button v-permission="['payment:list:edit', 'payment:config:edit']">操作</el-button>
```

**在 JS 中判断权限：**
```typescript
import { useUserStore } from '@/stores/user'

const userStore = useUserStore()

if (userStore.hasPermission('private-customers:list:delete')) {
  // 执行删除操作
}
```

**权限码常量引用：**
```typescript
import { PERMISSIONS } from '@/utils/permissions'

if (userStore.hasPermission(PERMISSIONS.PRIVATE_LIST_DELETE)) {
  // ...
}
```

## 五、API 接口文档

### 5.1 角色管理接口

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/v1/admin/roles/permission-tree` | 获取权限树 |
| GET | `/api/v1/admin/roles` | 获取角色列表（支持 keyword/status 查询） |
| GET | `/api/v1/admin/roles/:id` | 获取角色详情 |
| POST | `/api/v1/admin/roles` | 创建角色 |
| PUT | `/api/v1/admin/roles/:id` | 更新角色 |
| DELETE | `/api/v1/admin/roles/:id` | 删除角色（仅超级管理员） |

### 5.2 登录/Profile 接口变更

登录和 Profile 接口新增返回字段：
```json
{
  "data": {
    "id": "xxx",
    "username": "admin",
    "role": "operator",
    "roleId": "角色ID",
    "permissions": ["dashboard:view", "customer-management:all:view", "..."]
  }
}
```

## 七、注意事项

1. **向下兼容**：原有的 super_admin 用户无需任何改动，自动拥有全部权限
2. **无角色用户**：admin/operator 类型用户如果没有分配角色，则没有任何菜单访问权限
3. **权限树扩展**：如需新增权限码，需同步修改 `backend/src/routes/admin/roles.ts` 中的 `PERMISSION_TREE` 和 `admin/src/utils/permissions.ts` 中的常量定义
4. **内置角色保护**：系统内置角色（is_system=1）不可删除，超级管理员角色的权限不可修改
