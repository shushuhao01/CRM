## 一、数据库设计变更

### 1.1 修改 `tenant_packages` 表（新增3个字段）

```sql
-- 在 tenant_packages 表添加订阅相关字段
ALTER TABLE tenant_packages 
  ADD COLUMN subscription_enabled TINYINT(1) DEFAULT 0 
    COMMENT '是否支持订阅模式：0-不支持 1-支持' 
    AFTER yearly_price;

ALTER TABLE tenant_packages 
  ADD COLUMN subscription_channels VARCHAR(50) DEFAULT 'all' 
    COMMENT '订阅渠道：wechat-仅微信 alipay-仅支付宝 all-全部支持' 
    AFTER subscription_enabled;

ALTER TABLE tenant_packages 
  ADD COLUMN subscription_discount_rate DECIMAL(5,2) DEFAULT 0.00 
    COMMENT '订阅优惠折扣率（百分比，如10表示9折）' 
    AFTER subscription_channels;
```

**字段说明：**

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `subscription_enabled` | TINYINT(1) | 0 | 管理后台勾选"支持订阅"时为1 |
| `subscription_channels` | VARCHAR(50) | 'all' | 可选值：`wechat` / `alipay` / `all` |
| `subscription_discount_rate` | DECIMAL(5,2) | 0.00 | 如10=9折，15=8.5折，0=无优惠 |

### 1.3 新建 `subscriptions` 表（订阅记录主表）

```sql
CREATE TABLE IF NOT EXISTS subscriptions (
  id VARCHAR(36) NOT NULL PRIMARY KEY COMMENT '订阅ID（UUID）',
  tenant_id VARCHAR(36) NOT NULL COMMENT '关联租户ID',
  package_id INT NOT NULL COMMENT '关联套餐ID',
  
  -- 订阅状态
  status ENUM('signing', 'active', 'paused', 'cancelled', 'expired') 
    DEFAULT 'signing' COMMENT '订阅状态',
  channel ENUM('wechat', 'alipay') NOT NULL COMMENT '订阅渠道',
  
  -- 第三方签约信息
  wechat_contract_id VARCHAR(100) DEFAULT NULL COMMENT '微信委托代扣签约协议号',
  wechat_plan_id VARCHAR(100) DEFAULT NULL COMMENT '微信代扣计划编号',
  alipay_agreement_no VARCHAR(100) DEFAULT NULL COMMENT '支付宝周期扣款协议号',
  
  -- 扣款信息
  amount DECIMAL(10,2) NOT NULL DEFAULT 0 COMMENT '每期扣款金额',
  billing_cycle ENUM('monthly', 'yearly') DEFAULT 'monthly' COMMENT '扣款周期',
  
  -- 时间管理
  next_deduct_date DATE DEFAULT NULL COMMENT '下次扣款日期',
  last_deduct_date DATE DEFAULT NULL COMMENT '最近一次扣款日期',
  sign_date DATETIME DEFAULT NULL COMMENT '签约成功时间',
  cancel_date DATETIME DEFAULT NULL COMMENT '取消时间',
  
  -- 扣款统计
  total_deducted DECIMAL(10,2) DEFAULT 0 COMMENT '累计扣款金额',
  deduct_count INT DEFAULT 0 COMMENT '已扣款次数',
  fail_count INT DEFAULT 0 COMMENT '连续失败次数',
  
  -- 来源信息
  source ENUM('register', 'renew', 'upgrade') DEFAULT 'register' 
    COMMENT '订阅来源：注册时/续费时/升级时',
  
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  
  INDEX idx_tenant_id (tenant_id),
  INDEX idx_status (status),
  INDEX idx_next_deduct (next_deduct_date),
  INDEX idx_channel (channel)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订阅记录表';
```

**订阅状态流转图：**

```
                  签约请求
                    |
                    v
              +-----------+
              | signing   |  签约中（等待用户授权）
              +-----------+
                    |
          签约成功  |  签约失败/超时
                    |      |
                    v      v
              +-----------+     (删除记录)
              |  active   |  生效中（自动扣款）
              +-----------+
               /    |     \
  用户取消   /     |扣款   \  过期
            /      |失败    \
           v       v        v
    +-----------+ +---------+ +-----------+
    | cancelled | | paused  | |  expired  |
    +-----------+ +---------+ +-----------+
                       |
                   重试成功
                       |
                       v
                 +-----------+
                 |  active   |
                 +-----------+
```

### 2.6 定时任务设计

**任务1：自动扣款 (`subscription-auto-deduct`)**

```
执行时间：每天凌晨 01:00
执行逻辑：
1. 查询 subscriptions 表中 status='active' 且 next_deduct_date <= TODAY 的记录
2. 对每条记录：
   a. 调用对应渠道（微信/支付宝）扣款接口
   b. 创建 subscription_deduct_logs 记录
   c. 扣款成功 → 更新 tenants.expire_date，更新 next_deduct_date
   d. 扣款失败 → 记录失败，设置 next_retry_at，fail_count++
3. fail_count >= 3 → 状态改为 paused，发送短信/邮件提醒
```

**任务2：扣款重试 (`subscription-deduct-retry`)**

```
执行时间：每天 14:00
执行逻辑：
1. 查询 subscription_deduct_logs 中 status='failed' 且 retry_count < 3 
   且 next_retry_at <= NOW 的记录
2. 重试扣款
3. 重试成功 → 更新状态为 success，重置 fail_count
4. 重试失败 → retry_count++，设置下次重试时间

宽限期策略：
- 扣款失败后第1天：第1次重试
- 扣款失败后第3天：第2次重试
- 扣款失败后第5天：第3次重试
- 第7天仍失败：subscription 状态改为 expired，停止服务
```

## 三、Member JWT 认证设计

### 3.1 Token 结构

```javascript
// Member JWT Payload
{
  "type": "member",           // 区分CRM用户Token和会员Token
  "tenantId": "uuid...",
  "tenantCode": "ABC123",
  "phone": "13800138000",
  "iat": 1712188800,
  "exp": 1712793600           // 7天有效期
}
```

### 3.2 中间件设计

**新增 `memberAuth` 中间件：**

```
文件位置：backend/src/middleware/memberAuth.ts
功能：
1. 从 Authorization Header 提取 Bearer Token
2. 验证 JWT 签名和有效期
3. 检查 payload.type === 'member'
4. 查询 tenants 表确认租户存在且非禁用
5. 将 tenantId 注入 req.memberTenantId
```

**与现有认证体系的隔离：**

```
现有认证体系：
├── authenticateToken  → CRM用户登录 → req.user
├── adminAuth          → Admin管理员登录 → req.adminUser
└── memberAuth (新增)  → 会员中心登录 → req.memberTenantId

三套体系完全独立，互不干扰。
```
