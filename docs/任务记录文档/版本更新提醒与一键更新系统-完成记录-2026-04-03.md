## 一、任务描述

实现运营发布新版本后，私有客户和SaaS租户能收到更新提醒：
- 管理后台和CRM前端顶部添加版本更新图标（铃铛旁边）
- 有更新显示红点，无更新灰色
- 点击查看历史版本记录、最新版本、待更新版本
- SaaS租户只能查看更新历史和新增功能（只读）
- 私有客户支持一键更新（自动备份→拉取代码→安装依赖→构建→数据库迁移→重启服务）
- 管理后台版本发布支持上传压缩包、Git仓库、URL三种更新源
- 更新过程失败自动回滚，确保安全

## 三、数据库变更

### versions 表新增字段

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `source_type` | VARCHAR(20) | 'url' | 更新源类型: url/upload/git |
| `git_repo_url` | VARCHAR(500) | NULL | Git仓库地址 |
| `git_branch` | VARCHAR(100) | 'main' | Git分支 |
| `git_tag` | VARCHAR(100) | NULL | Git标签 |
| `package_path` | VARCHAR(500) | NULL | 上传包服务器路径 |
| `target_audience` | VARCHAR(20) | 'all' | 目标受众: all/saas/private |
| `release_notes_html` | TEXT | NULL | 富文本更新说明 |

### 新增 update_tasks 表

记录每次更新任务的完整生命周期：状态、进度、日志、备份路径、版本号、操作人等。

### 新增 migration_history 表

追踪数据库迁移历史：文件名、校验值、执行耗时、是否成功。

## 五、一键更新流程（核心）

```
预检查 (5%) → 备份数据 (20%) → 获取新版本 (40%) → 安装依赖 (55%)
→ 构建项目 (75%) → 数据库迁移 (88%) → 重启服务 (95%) → 完成 (100%)
```

### 安全机制
1. **预检查**: 磁盘空间检查、更新源可用性验证
2. **全量备份**: 后端dist + 前端dist + package.json + 数据库dump
3. **自动回滚**: 任何步骤失败 → 自动恢复备份 → 重启服务
4. **PM2平滑重启**: 使用 `pm2 reload` 而非 `restart`，减少中断时间
5. **迁移幂等**: 数据库迁移忽略 "already exists" 类错误，可安全重复执行

### 支持的更新源
| 类型 | 说明 | 适用场景 |
|------|------|----------|
| URL | 从下载链接获取压缩包 | CDN/OSS分发 |
| Upload | 管理后台直接上传压缩包 | 内网/离线环境 |
| Git | 从Git仓库拉取代码 | 标准开发流程 |

## 七、生产环境部署说明

### 数据库迁移
在phpMyAdmin中执行：`backend/database-migrations/production-20260403-version-update-system.sql`

该脚本：
- 使用 PREPARE/EXECUTE 安全添加列（列已存在则跳过）
- 使用 `CREATE TABLE IF NOT EXISTS` 建表
- 可多次执行不会报错
- 兼容 MySQL 5.7+ 和 8.0+

### 代码部署
1. 拉取最新代码
2. 后端: `cd backend && npm install && npm run build`
3. 前端: `npm install && npm run build`
4. 管理后台: `cd admin && npm install && npm run build`
5. PM2重启: `pm2 reload ecosystem.config.js`
