# SaaS 模式 RSA 签名保护机制 — 防止私有客户擅自启用多租户

> 📅 完成日期：2026-04-02  
> 📋 类型：安全架构  
> 🏷️ 标签：SaaS守卫、RSA签名、部署模式保护、许可证

---

## 📌 问题背景

### 原始问题
CRM 系统支持两种部署模式：
- **私有部署 (`private`)**：卖给客户的独立部署版本
- **SaaS 多租户 (`saas`)**：平台运营方自己运营的多租户版本

原来仅通过 `.env` 文件中 `DEPLOY_MODE=private|saas` 环境变量控制。

**安全漏洞**：私有部署客户只需修改 `DEPLOY_MODE=saas`，就能把系统变成多租户 SaaS 平台，自己收费做租户运营，严重侵害平台运营方利益。

### 核心需求
即使客户修改了 `DEPLOY_MODE=saas`，也无法真正激活 SaaS 多租户功能。

---

## 🔐 解决方案：RSA 签名许可验证（三层防护）

### 设计原理

```
私有部署客户                           平台运营方（你）
    │                                      │
    ├── 修改 DEPLOY_MODE=saas              ├── 持有 RSA 私钥
    ├── 没有 SAAS_LICENSE_TOKEN            ├── 用私钥签发 SAAS_LICENSE_TOKEN
    ├── 无法伪造签名                        ├── 配置 Token 到自己的服务器
    │                                      │
    ▼                                      ▼
  验证失败 → 自动降级为 private          验证通过 → SaaS 正常运行
  系统正常运行，但无多租户能力            多租户、租户管理全部可用
```

### 三层防护机制

| 层级 | 防护点 | 说明 |
|------|--------|------|
| **第1层** | 启动时验证 | 服务启动时用硬编码的 RSA 公钥验证 `SAAS_LICENSE_TOKEN` |
| **第2层** | 运行时持续校验 | 每小时重新验证 Token 有效性，过期自动降级 |
| **第3层** | 路由级守卫 | SaaS 专属路由（租户管理等）增加 `requireSaaSMode` 中间件 |

### 为什么安全？

1. **RSA 非对称加密**：公钥验签、私钥签发，即使客户看到源码中的公钥也无法伪造 Token
2. **私钥不随代码交付**：签发脚本 `generate-saas-license.js` 被 `.gitignore` 排除
3. **JWT 自带过期机制**：Token 有有效期，过期后自动失效
4. **静默降级**：验证失败不会导致系统崩溃，只是多租户功能不可用

## 🔧 技术实现细节

### 1. RSA 密钥对

- **算法**：RSA 2048-bit + RS256 签名
- **公钥位置**：硬编码在 `backend/src/services/SaaSGuardService.ts`（安全：仅能验证，不能签发）
- **私钥位置**：仅存于 `backend/scripts/generate-saas-license.js`（不随代码交付）

### 2. SAAS_LICENSE_TOKEN（JWT 格式）

```json
{
  "type": "saas_platform_license",
  "mode": "saas",
  "allowedDomains": ["localhost"],
  "maxTenants": 0,
  "iat": 1775111557,
  "exp": 2090471557,
  "licenseId": "dcb259ce-38a1-4d3b-b1f3-046497dabfce",
  "version": 1
}
```

- `type`: 固定为 `saas_platform_license`，防止用其他 JWT 冒充
- `allowedDomains`: 可绑定域名，限制 Token 使用范围
- `maxTenants`: 最大租户数限制（0=无限制）
- `exp`: 过期时间（Unix 时间戳）

### 3. 部署模式判断流程

```
系统启动
  │
  ├── 读取 DEPLOY_MODE 环境变量
  │   ├── private → 直接以私有模式运行（无需验证）
  │   └── saas → 进入 SaaS 许可验证
  │
  ├── SaaS 许可验证
  │   ├── 读取 SAAS_LICENSE_TOKEN 环境变量
  │   ├── 用 RSA 公钥验证 JWT 签名
  │   ├── 检查 Token 是否过期
  │   ├── 检查 payload 字段正确性
  │   │
  │   ├── ✅ 验证通过 → effectiveMode = 'saas'
  │   │   └── 启动定时器（每小时重新验证）
  │   │
  │   └── ❌ 验证失败 → effectiveMode = 'private'（降级）
  │       └── 输出警告日志
  │
  └── 所有代码通过 deployConfig.isSaaS() 判断
      └── 基于 effectiveMode，而非原始 DEPLOY_MODE
```

### 4. deploy.ts 改造要点

```typescript
// 原来（不安全）
isSaaS: () => deployConfig.mode === 'saas'

// 现在（安全）
isSaaS: () => deployConfig.effectiveMode === 'saas'
// effectiveMode 由 SaaSGuardService 验证通过后才会保持为 'saas'
// 验证失败则被强制设为 'private'
```

这意味着全系统 20+ 处调用 `deployConfig.isSaaS()` 的代码**无需修改**，自动继承保护。

## 📋 操作指南

### 平台运营方（你自己）

```bash
# 1. 生成 SaaS 许可证
cd backend
node scripts/generate-saas-license.js --domain crm.yourdomain.com --expires 365

# 2. 将输出的 Token 配置到服务器 .env 中
DEPLOY_MODE=saas
SAAS_LICENSE_TOKEN=eyJhbGciOiJSUzI1NiI...

# 3. 重启服务即可
```

### 交付给私有部署客户

```bash
# 客户的 .env 配置（正常情况）
DEPLOY_MODE=private
# 不需要 SAAS_LICENSE_TOKEN

# 即使客户改成这样也没用：
DEPLOY_MODE=saas
SAAS_LICENSE_TOKEN=     # 没有有效 Token，无法伪造
# → 系统自动降级为 private 模式
```

### 重新生成密钥对（如私钥泄露）

```bash
cd backend
node scripts/generate-saas-license.js --generate-keys

# 将输出的公钥更新到 SaaSGuardService.ts 中
# 将私钥更新到 generate-saas-license.js 中
# 重新签发所有 SaaS 许可证
```

## ⚠️ 重要安全提醒

### 绝不可交付给客户的文件
- `backend/scripts/generate-saas-license.js`（含 RSA 私钥）
- 任何包含 RSA 私钥的文件

### 可以安全交付的文件
- `backend/src/services/SaaSGuardService.ts`（仅含公钥，看到也无法伪造）
- `backend/src/middleware/saasGuard.ts`（中间件）
- `backend/src/config/deploy.ts`（部署配置）

### 交付私有版本的标准配置
```env
DEPLOY_MODE=private
# 无需配置 SAAS_LICENSE_TOKEN
```
