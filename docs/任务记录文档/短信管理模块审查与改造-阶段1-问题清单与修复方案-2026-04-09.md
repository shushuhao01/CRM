## 二、发现的问题清单（共18项）

### 🔴 严重问题（P0）

#### 问题 1：CRM端存在模板审核权，违反合规要求
- **位置**: `SmsManagement.vue` 第86-248行，模板审核Tab
- **描述**: 当前CRM端租户管理员可直接审核短信模板（通过/拒绝），这意味着租户自己审核自己的模板内容，存在**合规风险**。运营方无法对短信内容进行把关。
- **影响**: 不合规短信可能直接发出，无法统一管控。

#### 问题 2：CRM端可直接创建短信模板，缺少运营方管控
- **位置**: `SmsManagement.vue` 第14行 "创建短信模板"按钮、第10行 "模板管理"按钮
- **描述**: 租户和私有客户可以自由创建短信模板，缺少运营方审核环节。
- **影响**: 不合规内容的模板可能被创建并使用。

#### 问题 3：管理后台缺少短信管理模块
- **位置**: Admin后台
- **描述**: 管理后台（admin）没有短信管理一级菜单，无法查看所有租户的短信发送记录，也无法审核租户提交的模板申请。
- **影响**: 运营方无法做短信内容合规审查，无法追溯问题短信。

#### 问题 4：后端模板审核API缺少运营方权限校验
- **位置**: `backend/src/routes/sms.ts` 第69行 `/templates/:id/approve`
- **描述**: 审核API使用 `requireAdmin` 中间件，但这是CRM端的管理员权限（租户管理员），不是运营方管理后台管理员。租户管理员不应该有模板审核权。
- **影响**: 任意租户管理员可自行审核自己的模板，绕过运营方管控。

#### 问题 5：前端API Mock数据使用localStorage无租户隔离
- **位置**: `src/api/sms.ts` 第330-357行
- **描述**: `generateMockTemplates()`、`generateMockSmsRequests()`、`generateMockSendRecords()` 使用固定的 `localStorage` key（如 `crm_sms_templates`），不同租户在开发环境会共享同一份Mock数据。
- **影响**: 开发测试阶段不同租户数据可能互串。

### 🟠 重要问题（P1）

#### 问题 6：SmsManagement.vue 接口类型重复声明
- **位置**: `SmsManagement.vue` 第638行 `import type { SmsTemplate ... }` 和 第641-706行本地接口定义
- **描述**: 同时从 `@/api/sms` 导入接口类型，又在本地重复定义 `SmsTemplate`、`SmsRequest`、`SendRecord` 等接口，导致类型冲突。
- **影响**: TypeScript编译可能产生类型冲突警告。

#### 问题 7：模板审核/短信审核只是前端Mock操作，未调用后端API
- **位置**: `SmsManagement.vue` 第845-927行
- **描述**: `handleApproveTemplate`、`handleRejectTemplate`、`handleApproveSms`、`handleRejectSms` 函数只是在前端数组中操作数据（`splice/unshift`），没有调用 `smsApi.approveTemplate()` 或 `smsApi.approveSms()` 等后端API。
- **影响**: 审核操作不会持久化到数据库，刷新页面后审核状态丢失。

#### 问题 8：发送短信API缺少审批流程
- **位置**: `backend/src/routes/sms.ts` 第124-156行
- **描述**: `/send` 接口直接创建发送记录并标记 `status: 'completed'`，没有走审核流程。按需求，用户成员发短信需要租户管理员审核。
- **影响**: 任意用户可直接发送短信，无需审批。

#### 问题 9：后端sms路由的createQueryBuilder未额外处理tenant_id
- **位置**: `backend/src/routes/sms.ts` 第14-37行
- **描述**: `getTenantRepo(SmsTemplate).createQueryBuilder('template')` — 虽然 `getTenantRepo` 的 Proxy 拦截了 `createQueryBuilder` 并自动注入 `tenant_id`，但后续的 `.andWhere()` 使用的是自定义条件。需确认 Proxy 的 `.where()` 拦截是否在 `.andWhere()` 之后仍生效。
- **说明**: 经过审查 `tenantRepo.ts` 源码，Proxy 只拦截了 `.where()` 方法重新追加 `tenant_id`。如果查询只用 `.andWhere()` 而不调用 `.where()`，租户隔离条件会在 `createQueryBuilder` 返回时通过 `.andWhere()` 注入初始条件，这是安全的。**但如果后续手动调用 `.where()` 会重新追加**。此处无 `.where()` 调用，应该安全，但需验证。

#### 问题 10：统计数据API未隔离pendingSms计数
- **位置**: `backend/src/routes/sms.ts` 第158-197行
- **描述**: 统计API中 `pendingSms` 硬编码为 `0`，没有实际查询待审核短信数量。
- **影响**: CRM端显示的"待审核短信"数量始终为0。

#### 问题 11：SmsRecord实体缺少发件人手机号字段
- **位置**: `backend/src/entities/SmsRecord.ts`
- **描述**: 记录中有 `recipients`（收件人JSON）但没有发件人的手机号/渠道信息。管理后台需要知道是通过哪个渠道/号码发送的。
- **影响**: 管理后台短信记录缺少发送渠道信息。

### 🟡 一般问题（P2）

#### 问题 12：loadApprovedTemplates使用硬编码Mock数据
- **位置**: `SmsManagement.vue` 第1021-1135行
- **描述**: `loadApprovedTemplates` 函数使用硬编码的12个模板Mock数据，没有调用API获取已审核通过的模板。
- **影响**: 生产环境无法获取真实的已审核模板。

#### 问题 13：发送记录详情使用硬编码Mock
- **位置**: `SmsManagement.vue` 第939-952行
- **描述**: `handleViewRecord` 函数使用硬编码的5条发送详情数据。
- **影响**: 无法查看真实的发送详情。

#### 问题 14：SmsTemplate实体缺少审核拒绝原因字段
- **位置**: `backend/src/entities/SmsTemplate.ts`
- **描述**: 只有 `approvedBy`、`approvedAt`，没有 `rejectReason`（拒绝原因）字段。
- **影响**: 拒绝模板时无法记录拒绝原因，租户无法了解为何被拒。

#### 问题 15：前端SmsStatistics接口与后端API字段不一致
- **位置**: `src/api/sms.ts` 第109-114行 vs 第740-775行
- **描述**: 有两套统计接口定义 `SmsStatistics` 和 `getSmsStatistics()`，字段不一致（一个4字段，一个6字段）。
- **影响**: 组件使用时可能取到空值。

#### 问题 16：CRM端SmsSettings.vue嵌入了旧版SmsManagement组件
- **位置**: `src/views/System/Settings/SmsSettings.vue` 第100行
- **描述**: 系统设置-短信设置中嵌入了 `src/components/SmsManagement.vue`（旧版管理组件），与 `ServiceManagement/SmsManagement.vue` 功能重叠。
- **影响**: 两处入口的短信管理功能可能不一致。

#### 问题 17：后端模板创建用requireAdmin但应允许普通用户申请
- **位置**: `backend/src/routes/sms.ts` 第41行
- **描述**: `router.post('/templates', requireAdmin, ...)` 要求管理员权限才能创建/申请模板，但按业务需求普通销售员也应能申请模板。
- **影响**: 非管理员用户无法提交模板申请。

#### 问题 18：组件SmsManagement.vue（旧版）也有创建模板和发送短信功能
- **位置**: `src/components/SmsManagement.vue`
- **描述**: 与 `views/ServiceManagement/SmsManagement.vue` 功能高度重叠，维护成本高。
- **影响**: 代码重复，修改一处另一处不同步。

## 三、修复方案设计

### 3.1 架构改造总览

```
改造前:
  CRM端: 模板审核 + 短信审核 + 发送记录 + 创建模板 + 模板管理
  管理后台: 无短信模块

改造后:
  CRM端: 短信审核(用户发短信需管理员审批) + 发送记录(只读) + 申请模板(提交到管理后台审核) + 发送短信
  管理后台: [新增一级菜单"短信管理"] → 短信记录(全部租户收发记录) + 模板审核(审核租户申请的模板)
```

### 3.2 CRM端改造方案

#### 3.2.1 SmsManagement.vue 改造
1. **移除/注释**「模板管理」按钮 → 不再跳转到模板管理页
2. **移除/注释**「创建短信模板」按钮 → 改为「申请模板」，提交申请到管理后台审核
3. **移除/注释** 模板审核Tab → 将整个"模板审核"tab-pane注释掉
4. **保留** 短信审核Tab（租户管理员审核成员的发送申请）
5. **保留** 发送记录Tab
6. **保留**「发送短信」按钮
7. **新增** 「申请模板」按钮，调用后端接口提交模板申请到管理后台
8. 统计卡片：移除"待审核模板"，改为"已有模板数"

#### 3.2.2 API层改造（src/api/sms.ts）
1. 新增 `applyTemplate()` 函数 → 提交模板申请到管理后台
2. Mock数据的localStorage key加上租户标识

#### 3.2.3 旧版组件处理
1. `src/components/SmsManagement.vue` — 同步注释创建模板按钮
2. `src/views/System/SmsTemplates.vue` — 保留只读查看已通过模板的功能，注释创建/编辑

### 3.3 管理后台改造方案

#### 3.3.1 新增一级菜单「短信管理」
- 路由: `/sms-management`
- 二级菜单:
  - **短信记录** `/sms-management/records` — 查看所有租户/私有客户的短信发送记录
  - **模板审核** `/sms-management/template-review` — 审核租户提交的模板申请

#### 3.3.2 短信记录页面设计
- 显示字段: 租户名称、发送人、收件人（脱敏）、短信内容、模板名称、发送时间、状态（成功/失败）
- 支持筛选: 按租户、时间范围、状态筛选
- 支持搜索: 搜索内容、收件人
- 数据来源: `sms_records` 表，跨租户查询（管理后台不受 tenant_id 过滤限制）

#### 3.3.3 模板审核页面设计
- 待审核列表: 显示所有租户提交的模板申请
- 审核操作: 通过 / 拒绝（需填写拒绝原因）
- 已审核列表: 查看历史审核记录
- 数据来源: `sms_templates` 表，筛选 `status = 'pending'`

#### 3.3.4 权限配置
- 新增权限树节点: `sms-management` → `sms-management:records:view`、`sms-management:template-review:view`、`sms-management:template-review:approve`

### 3.4 后端API改造方案

#### 3.4.1 CRM端短信路由改造 (`backend/src/routes/sms.ts`)
1. **修改** `POST /templates` — 移除 `requireAdmin`，允许任意认证用户提交模板申请（status固定为pending）
2. **移除/禁用** `POST /templates/:id/approve` — CRM端不再有模板审核权
3. **新增** `GET /templates/approved` — CRM端获取本租户已审核通过的模板列表
4. **新增** `POST /sms/request` — 用户提交发送短信申请（需走审批流程）
5. **修改** `POST /send` — 改为只有审批通过的短信才能执行发送

#### 3.4.2 管理后台短信路由（新建 `backend/src/routes/admin/sms-management.ts`）
1. `GET /records` — 获取所有租户的短信发送记录（支持tenant_id筛选、分页）
2. `GET /records/:id` — 查看短信记录详情
3. `GET /templates/pending` — 获取所有待审核的模板列表
4. `GET /templates/reviewed` — 获取已审核的模板列表
5. `POST /templates/:id/approve` — 审核模板（通过/拒绝）
6. `GET /statistics` — 短信管理统计概览

#### 3.4.3 数据库实体改造
1. `SmsTemplate` 新增字段:
   - `reject_reason` TEXT — 拒绝原因
   - `reviewed_by_admin` VARCHAR(50) — 管理后台审核人
   - `reviewed_at` TIMESTAMP — 管理后台审核时间
2. `SmsRecord` 新增字段:
   - `sender_phone` VARCHAR(20) — 发送号码/渠道

## 六、方案修订记录

### v2 修订（2026-04-10）

根据进一步讨论，方案做了重要修订：

1. **区分两套短信**：管理后台系统短信（验证码/通知等运营方自用）和CRM业务短信（租户给客户发的营销/通知短信）是完全不同的两套体系，不能混淆
2. **模板审核流程增加服务商环节**：租户申请 → 管理员初审 → 提交服务商审核 → 获取模板CODE → 管理员最终激活
3. **管理后台短信管理菜单扩展为4个二级菜单**：短信记录、模板审核、CRM短信配置（从配置下发迁出）、统计总览
4. **配置下发的短信设置卡片**改为引导跳转到新的CRM短信配置页面
5. **SmsTemplate实体新增7个字段**支持服务商审核流程

详见: `短信管理模块审查与改造-方案修订v2-2026-04-10.md`
