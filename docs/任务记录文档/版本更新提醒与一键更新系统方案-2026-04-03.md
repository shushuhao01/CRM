## 一、需求概述

运营发布新版本后，系统各角色需要收到更新提醒并根据部署类型执行不同操作：

| 角色 | 能力 | 操作 |
|------|------|------|
| 管理员（Admin后台） | 发布版本、上传更新包、查看版本分布 | 顶部更新图标提醒 + 管理版本发布 |
| SaaS租户（CRM前端） | 查看更新历史、新增功能 | 只读查看，无更新按钮 |
| 私有客户（CRM前端） | 查看更新 + 一键更新 | 收到通知后可触发自动更新 |

## 三、技术可行性分析

### 3.1 一键更新可行性：✅ 可行

**核心原理：** 后端内置一个 `UpdateService`，相当于一个"更新代理"，在私有部署的服务器上执行以下自动化流程：

```
备份 → 获取新代码 → 安装依赖 → 构建 → 数据库迁移 → 重启服务
```

**技术依据：**
- 项目已使用 PM2 管理进程（`ecosystem.config.js`），可通过 `pm2 reload` 实现平滑重启
- TypeScript 后端使用 `tsc` 编译，构建命令明确（`npm run build`）
- 前端使用 Vite 构建，命令明确（`npm run build`）
- 数据库迁移可通过扫描 `database-migrations/` 目录实现
- Node.js 的 `child_process` 可执行所有系统命令

**支持的更新源：**
| 更新源 | 方式 | 适用场景 |
|--------|------|----------|
| 上传压缩包 | Admin后台上传 ZIP → 私有客户下载解压 | 无Git环境的客户 |
| GitHub仓库 | `git pull` 或 `git clone` | 有Git环境的客户 |
| 下载链接 | HTTP下载 ZIP/TAR.GZ | OSS/CDN分发 |

### 3.2 宝塔面板兼容性：✅ 可行

- 宝塔部署的 Node.js 项目本质上就是 PM2 管理的进程
- 一键更新通过内部 API 调用系统命令，不依赖宝塔面板
- 更新完成后 PM2 自动重启，宝塔面板中进程状态自动更新

### 3.3 本地部署兼容性：✅ 可行

- 本地部署同样基于 PM2 或直接 `node` 运行
- 更新脚本检测运行环境自适应

### 3.4 桌面端应用：⚠️ 后续规划

- 桌面端（Electron）有独立的更新机制（`electron-updater`）
- 建议后续单独处理，当前方案预留扩展接口

### 3.5 潜在风险与应对

| 风险 | 等级 | 应对方案 |
|------|------|----------|
| 更新过程中服务中断 | 中 | 使用 `pm2 reload`（graceful reload），中断仅几秒 |
| 构建失败 | 中 | 每步骤前备份，失败自动回滚 |
| 数据库迁移失败 | 高 | 迁移前 `mysqldump` 备份，失败恢复 |
| 磁盘空间不足 | 低 | 更新前检查磁盘空间（预检） |
| 网络中断（Git/下载） | 中 | 重试机制 + 超时中断 + 清理临时文件 |
| 私有客户权限不足 | 低 | 文件权限检查、以当前进程用户执行 |

## 四、数据库变更

### 4.1 扩展 `versions` 表

```sql
ALTER TABLE versions
  ADD COLUMN source_type ENUM('url', 'upload', 'git') DEFAULT 'url' COMMENT '更新源类型' AFTER download_url,
  ADD COLUMN git_repo_url VARCHAR(500) NULL COMMENT 'Git仓库地址' AFTER source_type,
  ADD COLUMN git_branch VARCHAR(100) DEFAULT 'main' COMMENT 'Git分支' AFTER git_repo_url,
  ADD COLUMN git_tag VARCHAR(100) NULL COMMENT 'Git标签' AFTER git_branch,
  ADD COLUMN package_path VARCHAR(500) NULL COMMENT '上传包服务器路径' AFTER git_tag,
  ADD COLUMN target_audience ENUM('all', 'saas', 'private') DEFAULT 'all' COMMENT '目标受众' AFTER package_path,
  ADD COLUMN release_notes_html TEXT NULL COMMENT '富文本更新说明(HTML)' AFTER changelog;
```

### 4.2 新增 `update_tasks` 表

```sql
CREATE TABLE update_tasks (
  id VARCHAR(36) PRIMARY KEY,
  version_id VARCHAR(36) NOT NULL COMMENT '目标版本ID',
  customer_id VARCHAR(36) NULL COMMENT '私有客户ID（null=管理后台自身）',
  status ENUM('pending', 'backing_up', 'downloading', 'installing', 'building', 'migrating', 'restarting', 'success', 'failed', 'rolled_back') DEFAULT 'pending' COMMENT '更新状态',
  current_step VARCHAR(50) NULL COMMENT '当前执行步骤',
  progress INT DEFAULT 0 COMMENT '进度百分比(0-100)',
  logs LONGTEXT NULL COMMENT '执行日志(JSON数组)',
  backup_path VARCHAR(500) NULL COMMENT '备份目录路径',
  error_message TEXT NULL COMMENT '错误信息',
  from_version VARCHAR(20) NULL COMMENT '更新前版本号',
  to_version VARCHAR(20) NULL COMMENT '目标版本号',
  triggered_by VARCHAR(36) NULL COMMENT '操作人ID',
  started_at DATETIME NULL COMMENT '开始时间',
  completed_at DATETIME NULL COMMENT '完成时间',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_version_id (version_id),
  INDEX idx_customer_id (customer_id),
  INDEX idx_status (status),
  FOREIGN KEY (version_id) REFERENCES versions(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='版本更新任务记录';
```

### 4.3 新增 `migration_history` 表（迁移追踪）

```sql
CREATE TABLE migration_history (
  id INT AUTO_INCREMENT PRIMARY KEY,
  filename VARCHAR(200) NOT NULL UNIQUE COMMENT '迁移文件名',
  checksum VARCHAR(64) NULL COMMENT '文件校验值',
  executed_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '执行时间',
  execution_time INT NULL COMMENT '执行耗时(ms)',
  success TINYINT DEFAULT 1 COMMENT '是否成功',
  error_message TEXT NULL COMMENT '错误信息'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='数据库迁移历史';
```

**逻辑：**
- 定时轮询（60秒）`GET /versions/latest` 获取最新已发布版本
- 比较当前系统版本（从 `system-config` 获取）与最新版本的 `versionCode`
- `versionCode` 大于当前 → 显示红点 + 新版本信息
- 相等 → 灰色图标，"已是最新版本"

### 6.2 一键更新服务 `UpdateService.ts`

```
执行流程（7步）：

 Step 1: 预检查
    ├── 检查磁盘空间 ≥ 500MB
    ├── 检查当前无正在执行的更新任务
    ├── 检查目标版本有效且已发布
    └── 记录当前版本号

 Step 2: 备份 (backing_up)
    ├── 创建备份目录 backups/update-{timestamp}/
    ├── 复制当前 backend/dist/ 目录
    ├── 复制当前前端 dist/ 目录
    ├── mysqldump 导出数据库
    └── 记录 backupPath

 Step 3: 获取新代码 (downloading)
    ├── [Git模式] git fetch && git checkout {tag/branch}
    ├── [Upload模式] 下载压缩包 → 解压到临时目录 → 覆盖项目文件
    └── [URL模式] HTTP下载 → 解压 → 覆盖

 Step 4: 安装依赖 (installing)
    ├── cd backend && npm install --production
    └── cd frontend && npm install

 Step 5: 构建项目 (building)
    ├── cd backend && npm run build  (tsc编译)
    ├── cd frontend && npm run build (vite构建)
    └── cd admin && npm run build (如果需要)

 Step 6: 数据库迁移 (migrating)
    ├── 扫描 database-migrations/ 目录
    ├── 比对 migration_history 表，找出未执行的SQL
    ├── 按文件名排序依次执行
    └── 记录执行结果到 migration_history

 Step 7: 重启服务 (restarting)
    ├── pm2 reload ecosystem.config.js (平滑重启)
    ├── 等待服务健康检查通过
    └── 更新系统配置中的版本号

 ❌ 任何步骤失败 → 自动回滚
    ├── 恢复备份的代码文件
    ├── 恢复数据库（mysql < backup.sql）
    ├── pm2 restart（重启到旧版本）
    └── 标记任务为 rolled_back
```

### 6.3 实时日志推送

**方式：SSE (Server-Sent Events)**

```
GET /api/v1/admin/update-tasks/:id/logs  (SSE)

客户端接收的数据流：
data: {"step":"backing_up","message":"正在备份数据库...","progress":15,"time":"10:30:01"}
data: {"step":"backing_up","message":"数据库备份完成 (23MB)","progress":20,"time":"10:30:05"}
data: {"step":"downloading","message":"正在从Git拉取代码...","progress":30,"time":"10:30:06"}
data: {"step":"downloading","message":"代码拉取完成","progress":45,"time":"10:30:20"}
data: {"step":"installing","message":"正在安装后端依赖...","progress":50,"time":"10:30:21"}
...
data: {"step":"success","message":"更新完成！系统已升级到 v1.9.0","progress":100,"time":"10:32:15"}
```

选择 SSE 而非 WebSocket 的原因：
- 更新日志是单向推送，不需要双向通信
- 项目已有 Socket.IO，但 SSE 更轻量且足够
- SSE 自动重连，更适合长时间运行的任务监控

### 6.4 SaaS vs 私有客户区分

**CRM前端判断逻辑：**
```typescript
// 判断方式1：通过环境变量
const isPrivateDeployment = import.meta.env.VITE_DEPLOY_MODE === 'private'

// 判断方式2：通过授权验证API返回的部署类型
const { deploymentType } = await licenseApi.verify()
const isPrivateDeployment = deploymentType === 'on-premise'
```

**私有客户更新流程：**
1. CRM前端轮询公开API检查新版本
2. 发现新版本 → 显示更新提示
3. 管理员点击"一键更新" → 调用后端 `POST /update/execute`
4. 后端 UpdateService 执行完整更新流程
5. 前端通过 SSE 显示实时进度
6. 更新完成后页面自动刷新

**SaaS租户更新流程：**
1. 运营在管理后台发布新版本
2. SaaS租户前端轮询检查到新版本
3. 显示更新提示面板，展示新功能和修复
4. 无更新按钮，仅提示"系统已自动更新至最新版本"

## 九、关键技术决策需确认

### ❓ 决策1：更新执行方式
- **方案A（推荐）：后端内置更新代理** - UpdateService 直接在后端执行系统命令
  - 优点：不需要额外安装，私有客户开箱即用
  - 缺点：后端自己重启自己，需要特殊处理（PM2 会自动拉起）
- **方案B：独立更新守护进程** - 单独的 Node.js 脚本作为更新代理
  - 优点：更新时不影响主服务
  - 缺点：私有客户需要额外配置

**建议选择方案A**：PM2 管理的进程在 `pm2 reload` 后会自动拉起新进程，执行更新的最后一步调用 `pm2 reload` 后，旧进程退出，新进程启动加载新代码，整个过程是安全的。

### ❓ 决策2：私有客户更新触发方式
- **方案A：私有客户端手动触发** - 管理员在CRM前端点击"一键更新"
- **方案B：管理后台远程推送** - 管理后台直接向私有客户服务器发送更新命令
- **方案C（推荐）：A+B结合** - 私有客户可自行检查更新并手动触发，管理后台也可推送更新通知

**建议选择方案C**：最灵活，私有客户有主动权，管理后台也能推送提醒。

### ❓ 决策3：Git仓库访问方式
- **方案A：公开仓库** - 不需要认证
- **方案B：私有仓库 + HTTPS Token** - 管理后台配置 Personal Access Token
- **方案C：私有仓库 + SSH Key** - 私有客户服务器配置 SSH Key

**建议先支持方案A+B**，SSH Key 后续扩展。

### ❓ 决策4：前端构建是否包含在更新流程中
- **方案A（推荐）：包含前后端完整构建** - 更新时同时构建前端和后端
- **方案B：只发布构建产物** - 上传包中直接包含编译后的 dist 文件，无需构建

**建议方案A为主，方案B为辅**：Git方式需要构建，上传压缩包可以直接包含构建产物。

## 十、安全考虑

1. **更新包完整性**：使用 SHA256 校验下载的文件完整性
2. **授权验证**：公开版本检查API需验证 licenseKey
3. **操作权限**：仅超级管理员可触发一键更新
4. **备份保留**：默认保留最近3次更新备份，超出自动清理
5. **并发控制**：同时只能有一个更新任务在执行
6. **超时机制**：单个步骤超时15分钟自动中断并回滚
