> 审查日期：2026-04-08
> 审查范围：CRM前端、后端API、管理后台(Admin)、官网(Website)、会员中心、移动APP(crmAPP)
> 重点：多租户数据隔离安全、代码质量、部署配置

## 一、审查总览

### 1.1 项目架构概述

| 子项目 | 技术栈 | 说明 |
|--------|--------|------|
| CRM前端 | Vue3 + TypeScript + Vite | 主业务系统前端 |
| 后端API | Node.js + Express + TypeORM + MySQL | 统一后端服务 |
| Admin管理后台 | Vue3 + TypeScript + Vite | 平台管理后台 |
| 官网(Website) | Vue3 + TypeScript + Vite | 产品官网 |
| 移动APP(crmAPP) | uni-app + Vue3 + TypeScript | 外呼助手APP |

### 1.2 多租户架构分析

项目采用 **共享数据库+行级隔离** 方案：
- 实体表通过 `tenant_id` 字段区分租户
- `BaseRepository` / `getTenantRepo` 自动注入租户过滤
- `TenantContextManager`（AsyncLocalStorage）管理请求级租户上下文
- 中间件链：`tenantContextMiddleware` → `authenticateToken` / `tenantAuth` → 路由处理

## 二、发现的问题清单

### 🔴 严重级别（P0 - 安全隐患）

#### 问题 1：WebSocket 房间未包含租户隔离
- **文件**: `backend/src/services/WebSocketService.ts`
- **位置**: 第 136-144 行
- **描述**: WebSocket 连接时，用户加入的房间 `role_{roleName}` 和 `department_{departmentId}` 未包含 `tenantId` 前缀。不同租户中可能存在相同的角色名和部门ID，导致跨租户消息泄露。
- **影响**: 不同租户的相同角色/部门用户会收到彼此的实时推送消息
- **修复方案**: 将房间名改为 `tenant_{tenantId}_role_{roleName}` 和 `tenant_{tenantId}_department_{departmentId}` 格式，所有 `sendToRole`、`sendToDepartment`、`broadcast` 方法也需要对应修改

#### 问题 2：WebSocket broadcast 方法无租户范围限制
- **文件**: `backend/src/services/WebSocketService.ts`
- **位置**: 第 294-303 行
- **描述**: `broadcast` 方法使用 `this.io.emit()` 向所有连接用户发送消息，不区分租户。虽然 `pushSystemMessage` 已注释了无目标时的广播，但 `publishAnnouncement`（MessageController 第 830 行）仍直接调用 `broadcast`。
- **影响**: 全员广播的公告会发送给所有租户的在线用户
- **修复方案**: 添加 `broadcastToTenant(tenantId, event, data)` 方法，让用户在连接时加入 `tenant_{tenantId}` 房间

#### 问题 3：静态文件服务无访问控制
- **文件**: `backend/src/app.ts`
- **位置**: 第 186-187 行
- **描述**: `app.use('/uploads', express.static(...))` 和 `app.use('/recordings', express.static(...))` 未做任何认证和租户访问控制。虽然 SaaS 模式下文件按租户编码存储在子目录中（如 `uploads/T260303A1B2/products/`），但任何人只要知道 URL 就可以直接访问其他租户的文件。
- **影响**: 跨租户文件访问漏洞
- **修复方案**: 保持静态文件服务但接受当前状态（图片类资源通常不含高度敏感数据），或添加中间件验证 `req.tenantId` 与请求路径中的租户编码一致性（此项视为低优先级可接受风险）

#### 问题 4：MessageController 部门订阅配置查询未添加租户过滤
- **文件**: `backend/src/controllers/MessageController.ts`
- **位置**: 第 157-161 行、第 277-280 行
- **描述**: `departmentConfigRepo.find({ relations: ['department'] })` 未添加任何租户过滤。`DepartmentSubscriptionConfig` 实体本身没有 `tenant_id` 字段，但其关联的 `Department` 实体有 `tenant_id`。在 SaaS 模式下，查询可能返回其他租户的部门配置。
- **影响**: 跨租户数据泄露风险
- **修复方案**: 通过 Department 关联进行租户过滤，使用 `getTenantRepo(Department)` 先获取本租户部门ID列表，然后筛选配置；或在查询时通过 `where: { department: { tenantId } }` 做嵌套过滤

### 🟡 中等级别（P1 - 潜在风险）

#### 问题 5：MessageController 使用原始 getRepository 而非 getTenantRepo
- **文件**: `backend/src/controllers/MessageController.ts`
- **位置**: 多处（第 143/207/275/301/487/514 行等）
- **描述**: MessageController 大量使用 `dataSource.getRepository()` 而非 `getTenantRepo()`。虽然代码中大部分手动添加了 `tenantId` 条件，但部分查询遗漏了租户过滤（如问题4）。
- **影响**: 增加遗漏租户过滤的风险
- **修复方案**: 将 `getRepository` 调用替换为 `getTenantRepo`（需注意：部分查询已手动添加了 tenantId，替换后不要重复过滤）

#### 问题 6：WebSocket getTenantRepo 可能缺失租户上下文
- **文件**: `backend/src/services/WebSocketService.ts`
- **位置**: 第 101/201/220/239 行
- **描述**: WebSocket 事件处理在长连接中运行，`TenantContextManager`（AsyncLocalStorage）的上下文可能已丢失。`getTenantRepo` 依赖 `TenantContextManager.getTenantId()` 获取租户ID，但 WebSocket 事件回调并非在 HTTP 请求的 async context 中执行。
- **影响**: WebSocket 中的数据查询/更新可能缺少租户隔离
- **修复方案**: 在 WebSocket 连接时将 `tenantId` 保存到 `socket.tenantId`，在事件处理中手动设置 `TenantContextManager.setContext()`

#### 问题 7：handleMarkAllRead 的 QueryBuilder 可能缺少租户过滤
- **文件**: `backend/src/services/WebSocketService.ts`
- **位置**: 第 239-246 行
- **描述**: `messageRepo.createQueryBuilder().update()...` 使用了 `getTenantRepo` 的 `createQueryBuilder`，但由于问题6中的上下文丢失问题，租户过滤可能不会自动注入。
- **影响**: 可能影响到其他租户的消息已读状态
- **修复方案**: 结合问题6的修复，确保上下文存在

#### 问题 8：公告发布 WebSocket 推送缺少租户范围
- **文件**: `backend/src/controllers/MessageController.ts`
- **位置**: 第 822-832 行
- **描述**: `publishAnnouncement` 中按部门推送时调用 `sendToDepartment(deptId, ...)` 和全员广播 `broadcast(...)`，都没有租户范围限制。
- **影响**: 公告推送可能被其他租户的同部门ID用户接收
- **修复方案**: 需要结合问题1/2的修复

### 🟢 低级别（P2 - 优化建议）

#### 问题 9：部分实体可能缺少 tenant_id 字段检查
- **观察**: `DepartmentSubscriptionConfig` 和 `AnnouncementRead` 等关联实体需确认是否有 `tenant_id` 字段
- **建议**: 审查所有实体确保需要隔离的表都有 `tenant_id` 字段

#### 问题 10：UserController 登录时用户名唯一性依赖租户隔离
- **文件**: `backend/src/controllers/UserController.ts`
- **位置**: 第 84-87 行
- **描述**: 登录查询 `{ username, tenantId }` 正确实现了隔离。但如果 SaaS 模式下客户端未传 `tenantId`，且存在不同租户的同名用户，会返回错误的用户。
- **现状**: 已有 `deployConfig.isSaaS() && !tenantId` 的校验（第77行），确保 SaaS 模式必须传租户编码。✅ 已正确处理

#### 问题 11：mobile 登录路由使用原始 SQL 需确认隔离完整性
- **文件**: `backend/src/routes/mobile/auth.ts`
- **位置**: 第 41-49 行
- **描述**: 移动端登录使用原始 SQL `SELECT ... FROM users WHERE username = ?`，有条件地添加 `AND tenant_id = ?`。但仅在 `tenantId` 有值时才添加，SaaS 模式下如果 APP 端未传 tenantId 会返回错误用户。
- **修复方案**: 建议在 SaaS 模式下强制要求 APP 端传入 tenantId（与 CRM 登录一致）

#### 问题 12：验证码存储使用内存 Map（register 路由）
- **文件**: `backend/src/routes/public/register.ts`
- **位置**: 第 34 行
- **描述**: `const verificationCodes: Map<string, ...>` 存储在进程内存中，多实例部署时不共享。
- **现状**: 注释中已说明"生产环境应使用 Redis"，当前单实例部署可接受

## 四、各子项目审查结论

### 4.1 CRM 前端（src/）
- ✅ API 调用均通过统一的 axios 拦截器，自动携带 JWT Token
- ✅ Token 中包含 tenantId，后端通过中间件自动解析
- ✅ 前端不直接处理租户隔离逻辑，由后端保证
- ⚠️ localStorage 中存储的用户信息无租户隔离（同一浏览器切换租户需要重新登录）

### 4.2 后端 API（backend/src/）
- ✅ BaseRepository 和 getTenantRepo 提供可靠的自动租户过滤
- ✅ 核心业务路由（客户/订单/产品/部门/角色等）均使用 getTenantRepo
- ✅ 登录认证正确实现了租户隔离（SaaS 模式强制 tenantId）
- ✅ 文件上传按租户编码隔离目录
- ⚠️ MessageController 和 WebSocket 存在上述安全问题
- ⚠️ 部分使用原始 SQL 的地方需确认 tenant_id 过滤

### 4.3 Admin 管理后台（admin/src/）
- ✅ 使用独立的 adminAuth 中间件（验证 isAdmin 标记）
- ✅ Admin 操作的是全局数据（租户/授权/套餐），不受 tenant_id 限制（设计正确）
- ✅ 后端 admin 路由统一使用 adminAuthMiddleware

### 4.4 官网（website/src/）
- ✅ 仅展示公开信息和注册流程
- ✅ 注册时创建独立租户，不涉及跨租户数据
- ✅ 会员登录通过 memberAuth 中间件，正确验证租户状态

### 4.5 移动APP（crmAPP/src/）
- ✅ 使用与 CRM 相同的后端 API 和认证体系
- ⚠️ 登录时 SaaS 模式下需确保传入 tenantId（问题11）
