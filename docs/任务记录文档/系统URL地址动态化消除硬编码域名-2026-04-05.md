 # 系统URL地址动态化 — 消除硬编码域名

**日期**: 2026-04-05  
**状态**: ✅ 已完成  
**涉及模块**: 后端API / 官网前端 / Admin前端 / CRM前端  
**类型**: 修复 + 架构优化（硬编码域名 → 动态配置）

---

## 一、问题背景

系统多处代码中硬编码了旧域名 `https://app.yunke-crm.com`（CRM系统地址）和 `https://www.yunke-crm.com`（官网地址），导致：

1. 当运营方使用不同域名部署（如 `crm.yunkes.com`）时，这些链接指向错误地址
2. 不同开发者/运营方部署在自定义域名时，需要手动修改大量源代码
3. 违反了"单一配置源"原则

**正确的架构**：所有系统URL都应从后端 `SITE_CONFIG`（`backend/src/config/sites.ts`）集中配置，通过 API 动态下发到各前端。

---

## 二、后端集中配置源

**文件**: `backend/src/config/sites.ts`

```typescript
export const SITE_CONFIG = {
  CRM_URL: process.env.CRM_URL || 'https://crm.yunkes.com',
  WEBSITE_URL: process.env.WEBSITE_URL || 'https://yunkes.com',
  API_URL: process.env.API_URL || 'https://api.yunkes.com',
  ADMIN_URL: process.env.ADMIN_URL || 'https://admin.yunkes.com',
  RENEW_URL: process.env.RENEW_URL || 'https://yunkes.com/pricing'
};
```

**特点**:
- 默认值使用运营方实际域名
- 支持通过环境变量覆盖（不同部署环境用不同域名）
- 已被 `NotificationTemplateService`、`PaymentService`、`AlipayService` 等正确使用

## 五、部署说明

### 不同运营方自定义域名

只需修改 **一个文件**（或设置环境变量）：

**方式一：修改 `backend/src/config/sites.ts` 默认值**
```typescript
CRM_URL: process.env.CRM_URL || 'https://crm.你的域名.com',
WEBSITE_URL: process.env.WEBSITE_URL || 'https://你的域名.com',
```

**方式二：设置环境变量（推荐，无需改代码）**
```bash
CRM_URL=https://crm.你的域名.com
WEBSITE_URL=https://你的域名.com
API_URL=https://api.你的域名.com
ADMIN_URL=https://admin.你的域名.com
```

所有前端（官网/CRM/Admin）将自动获取正确的地址。

## 六、注意事项

1. `website/public/sitemap.xml` 中也有旧域名，但 sitemap 是 SEO 静态文件，需要部署时根据实际域名手动更新
2. `.md` 文档中的旧域名仅作为历史记录，不影响功能
3. `backend/src/middleware/memberAuth.ts` 中的 `yunke-crm-member` 是 JWT issuer 标识，不是域名，不需要修改
4. 官网前端使用了 `getWebsiteConfig()` 的内存缓存机制，多个组件共享同一个API请求，不会造成重复请求
