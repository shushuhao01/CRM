## 一、任务背景

CRM系统物流管理模块支持9家主流快递公司的API对接（顺丰、中通、圆通、申通、韵达、极兔、邮政EMS、京东物流、德邦快递）。其中顺丰速运(SF)的API配置已验证正确，其余8家公司的API配置在初始开发时存在以下类型的问题：

1. **签名算法错误** — 与物流公司官方API文档不一致
2. **API URL不一致** — 前端配置与后端测试函数使用不同的URL
3. **字段映射混乱** — 部分公司的表单字段标签与实际API参数不匹配
4. **缺少配置引导** — 大部分公司缺少setupSteps操作步骤指引
5. **签名编码格式错误** — 部分使用hex编码应为base64

### 2. 中通快递 (ZTO) — 签名算法修正
**问题**: 签名使用MD5，应为HMAC-SHA256+Base64  
**修复**:
- **前端**: 更新API URL为新版端点 `traceInterfaceNewTraces`，字段提示增加签名说明，添加setupSteps
- **后端**: 签名算法从 `MD5(appKey+timestamp+data+appSecret).hex.toUpperCase` 改为 `HMAC-SHA256(data, appSecret).base64`

- **认证方式**: 请求头 `x-companyid`(企业ID) + `x-appkey`(应用密钥) + `x-datadigest`(HMAC-SHA256签名) + `x-timestamp`(时间戳)

### 3. 圆通速递 (YTO) — 字段映射简化 + 请求结构修正

- 前端URL (`track_query_adapter`) 与后端URL (`track_query`) 不一致  
**修复**:
- **前端**: `showAppKey`改为`false`(隐藏多余字段)，简化为：appId→AppKey, appSecret→SecretKey, customerId→客户编码
- **后端switch**: 参数传递从 `(appId, appKey, appSecret)` 改为 `(appId, appSecret, customerId)`
- **后端函数**: 请求体key从`data`改为`param`，JSON结构改为`{Number, OrderType}`格式
- **API URL统一**: `https://openapi.yto.net.cn/open/track_query/v1/query`

## 五、部署使用说明

修复完成后，各物流公司API配置流程：

1. 在CRM系统中进入 **物流管理 → 物流公司管理**
2. 选择要配置的物流公司，点击 **API配置**
3. 按照对话框中显示的 **配置步骤** 引导，登录对应物流公司的开放平台
4. 获取API密钥信息，填入配置表单
5. 选择环境（测试/生产），点击 **测试连接** 验证
6. 测试通过后保存配置，即可投入生产使用

### 6.2 快递100响应解析Bug修复

**问题**: `ExpressAPIService.ts`中`queryByKuaidi100`方法的响应判断 `response.data.returnCode === '200'` 永远为false

**原因**: 快递100成功响应格式为 `{message: 'ok', state: '0', status: '200', data: [...]}`，不包含`returnCode`字段。`returnCode`仅在错误响应中出现。

**修复**: 
- 成功判断改为 `response.data.message === 'ok' || (data && Array.isArray(data))`
- `state`字段在根对象`response.data.state`上，而非`response.data.data.state`
- 轨迹数组直接在`response.data.data`，而非`response.data.data.data`

| 修复前 | 修复后 |
|--------|--------|
| `returnCode === '200'` | `message === 'ok'` |
| `data.state`（嵌套错误） | `resData.state`（根对象） |
| `data.data?.[0]?.context`（双重嵌套） | `resData.data?.[0]?.context` |
| `data.data?.map(...)`（双重嵌套） | `resData.data?.map(...)` |

## 七、查询优先级逻辑实现

### 7.1 需求

- 官方API已配置 → 优先使用，失败则降级快递100（如果已配置）
- 官方API未配置，仅配置快递100 → 直接使用快递100
- 两者都未配置 → 返回配置提示

### 7.2 修改文件

`backend/src/services/LogisticsTraceService.ts` — `queryTrace()` 方法重构

### 7.3 原有逻辑（有缺陷）

```
获取官方API配置
→ 未找到配置 → 直接返回"API未配置"错误（❌ 不尝试快递100）
→ 配置已禁用 → 直接返回"API已禁用"错误（❌ 不尝试快递100）  
→ 密钥不完整 → 直接返回"密钥不完整"错误（❌ 不尝试快递100）
→ 调用官方API → 成功 → 返回结果
                → 失败 → 尝试快递100降级
```

### 7.4 新逻辑（已修复）

```
获取官方API配置 + 快递100配置状态
→ 判断: hasOfficialApi = (配置存在 && 已启用 && 密钥完整)
→ 判断: hasKuaidi100 = 快递100已配置

情况1: 两者都未配置 → 返回"请配置官方API或快递100"
情况2: 官方API可用
  → 调用官方API
  → 成功且有轨迹 → 返回官方数据
  → 失败/无轨迹 → 如果hasKuaidi100 → 降级快递100
  → 异常 → 如果hasKuaidi100 → 降级快递100
情况3: 仅快递100可用
  → 直接调用快递100 → 返回结果
```
