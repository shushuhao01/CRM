## 一、管理后台「短信管理」一级菜单（5个二级菜单）

```
📱 短信管理 (一级菜单，图标: ChatDotRound)
  ├── 📊 短信记录            /sms-management/records           — 全平台CRM短信发送记录
  ├── 📋 模板审核            /sms-management/template-review    — 审核租户/私有客户申请的模板
  ├── 📑 模板管理            /sms-management/templates          — 后台预设模板 + 查看所有租户模板(增删改)
  ├── ⚙️ CRM短信配置        /sms-management/crm-config         — 管控租户短信服务商密钥(从配置下发迁出)
  └── 📈 统计总览            /sms-management/statistics         — 全平台用量统计(后期可选)
```

> **管理后台自身的系统短信配置**（验证码/通知等）**仍留在** 系统设置 → 运营配置 → 短信配置Tab，不移动。

### 1.1 短信记录（Records）
- 跨租户查看所有CRM短信发送记录
- 字段：租户名/ID、发送人、收件人(脱敏)、内容、模板名称、模板CODE、时间、状态(成功/失败)
- 筛选：租户、时间范围、状态、模板
- 操作：查看详情

### 1.2 模板审核（TemplateReview）
- 待审核列表：显示所有租户/私有客户提交的模板申请
- 四阶段审核流程：
  1. `pending_admin` → 管理员审核内容合规性
  2. `pending_vendor` → 管理员手动去服务商平台提交
  3. 服务商通过后 → 管理员填入模板CODE → `active`
  4. 任一阶段可拒绝 → `rejected`
- 已审核列表：历史审核记录
- **拒绝需填写原因**

### 1.3 ★ 模板管理（Templates）— **新增**
- **后台预设模板**：管理后台可创建运营方预设的通用短信模板（如"订单发货通知"、"付款提醒"等），预设模板自带服务商模板CODE，创建后自动为所有租户可用
- **查看所有租户模板**：列表显示所有租户的模板（含预设 + 租户自建），支持按租户筛选
- **增删改操作**：
  - 新建预设模板（选择适用范围：全部租户/指定租户）
  - 编辑模板（修改名称、内容、变量、模板CODE）
  - 删除模板（逻辑删除，已使用的模板不可物理删除）
- **模板分类标签**：预设模板/租户自建模板/通用/营销/通知/服务

### 1.4 CRM短信配置（CrmConfig）
- 从配置下发的 sms 卡片迁出
- 配置内容：服务商选择、AccessKey、SecretKey、签名、限额、启用开关、审核开关
- 管控开关：开启后下发给所有CRM端
- 存储路径不变（`distributedConfig.sms`）

## 三、消息通知机制设计 — **新增**

### 3.1 CRM端通知（租户内部）

| 触发事件 | 通知对象 | 通知方式 | 内容 |
|---------|---------|---------|------|
| 成员申请发送短信 | 租户管理员 | 系统消息 + WebSocket实时推送 | "员工XXX申请发送短信给N位客户，请审核" |
| 短信审核通过 | 申请人 | 系统消息 + WebSocket | "您的短信发送申请已通过，将自动发送" |
| 短信审核拒绝 | 申请人 | 系统消息 + WebSocket | "您的短信发送申请被拒绝，原因：XXX" |
| 模板申请结果通知 | 申请人 | 系统消息 + WebSocket | "您申请的短信模板'XXX'已通过/被拒绝" |
| 模板申请状态变更 | 租户管理员 | 系统消息 | "模板'XXX'状态变更为：服务商审核中/已激活/已拒绝" |

**实现方式**：
- 后端：在短信审核/模板状态变更时调用 `OrderNotificationService.sendBatchMessages()` 给目标用户发送站内消息
- 前端：使用已有的 `MessageType` 枚举新增短信相关类型
- WebSocket：通过已有的 `WebSocketService.sendToUser()` 实时推送

#### 新增 MessageType 枚举值：
```typescript
// 短信相关
SMS_SEND_REQUEST = 'sms_send_request',          // 短信发送申请（通知管理员）
SMS_SEND_APPROVED = 'sms_send_approved',        // 短信发送已通过（通知申请人）
SMS_SEND_REJECTED = 'sms_send_rejected',        // 短信发送被拒（通知申请人）
SMS_TEMPLATE_APPROVED = 'sms_template_approved', // 模板审核通过（通知申请人+管理员）
SMS_TEMPLATE_REJECTED = 'sms_template_rejected', // 模板审核拒绝（通知申请人）
SMS_TEMPLATE_STATUS = 'sms_template_status',     // 模板状态变更（通知管理员）
```

### 3.2 管理后台通知（运营方）

| 触发事件 | 通知对象 | 通知方式 | 内容 |
|---------|---------|---------|------|
| 新短信模板申请 | 管理后台管理员 | 系统消息 + 配置渠道(钉钉/邮件/短信等) | "租户XXX提交了新短信模板申请'模板名称'，请及时审核" |
| 模板审核提醒(超时) | 管理后台管理员 | 系统消息 | "有N个短信模板申请待审核超过24小时" |

**实现方式**：
- 后端：在 `AdminNotificationService` 的 `EVENT_TYPES` 中新增短信相关事件
- 通知渠道：复用已有的通知服务架构（系统消息+钉钉+企微+邮件+短信）

#### 新增 AdminNotification EVENT_TYPES：
```typescript
sms_template_applied: { label: '新短信模板申请', level: 'info', category: '短信管理' },
sms_template_review_timeout: { label: '模板审核超时提醒', level: 'warning', category: '短信管理' },
```

### 3.3 通知配置说明
- CRM端：复用已有的消息订阅配置（`MessageSubscription`），管理员可在系统设置中配置是否接收短信审核通知
- 管理后台：在通知服务的**通知规则**中新增 `sms_template_applied` 和 `sms_template_review_timeout` 事件，管理员可选择启用哪些渠道接收

## 四、模板审核完整流程（四阶段）不变

```
pending_admin   → 租户提交，等管理员初审
pending_vendor  → 管理员初审通过，提交服务商
vendor_approved → 服务商通过（中间态）
active          → 管理员填入模板CODE激活
rejected        → 拒绝（任一阶段）
withdrawn       → 租户自行撤销（新增状态）
```

> 新增 `withdrawn` 状态：CRM端用户在 `pending_admin` 阶段可以撤销自己的申请。

## 五、SmsTemplate 实体最终字段设计

### 5.1 新增字段（共8个）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `vendor_template_code` | VARCHAR(100) | 服务商模板CODE（如 SMS_123456789） |
| `vendor_status` | VARCHAR(20) | 服务商审核状态：null/pending/approved/rejected |
| `vendor_submit_at` | TIMESTAMP | 提交服务商时间 |
| `vendor_reject_reason` | TEXT | 服务商拒绝原因 |
| `admin_reviewer` | VARCHAR(50) | 管理后台审核人 |
| `admin_review_at` | TIMESTAMP | 管理后台审核时间 |
| `admin_review_note` | TEXT | 管理后台审核备注/拒绝原因 |
| `is_preset` | TINYINT DEFAULT 0 | 是否为后台预设模板（1=预设, 0=租户自建） |

### 5.2 status 字段枚举扩展

```
pending_admin  — 待管理员审核
pending_vendor — 已提交服务商
active         — 已激活可用
rejected       — 已拒绝
withdrawn      — 已撤销
deleted        — 已删除（逻辑删除）
```
