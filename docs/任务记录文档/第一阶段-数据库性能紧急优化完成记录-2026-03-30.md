## 📝 各任务详细说明

### 任务 1.1：核心业务表添加 tenant_id 列和单列索引 ✅

**涉及24张表**：customers、orders、products、product_categories、departments、roles、customer_tags、customer_groups、customer_shares、follow_up_records、call_records、after_sales_services、service_records、logistics、performance_records、operation_logs、data_records、notifications、sms_records、sms_templates、messages、order_items、order_status_history、performance_metrics

**操作内容**：
- 为每张表执行 `ALTER TABLE ADD COLUMN IF NOT EXISTS tenant_id VARCHAR(36) NULL`
- 为每张表创建 `idx_xxx_tenant_id` 单列索引
- 使用存储过程 `safe_create_index` 确保索引幂等创建

**安全保障**：
- `ADD COLUMN IF NOT EXISTS` 确保重复执行不报错
- 存储过程先检查 `information_schema.STATISTICS` 再决定是否创建索引

### 任务 1.3：修复唯一索引为租户级唯一 ✅

**修复内容**：

| 表名 | 旧索引 | 新索引 | 说明 |
|------|--------|--------|------|
| customers | `UNIQUE(customer_code)` | `UNIQUE(tenant_id, customer_code)` | 允许不同租户客户编号重复 |
| orders | `UNIQUE(order_number)` | `UNIQUE(tenant_id, order_number)` | 允许不同租户订单号重复 |

**操作步骤**：
1. `DROP INDEX IF EXISTS` 删除旧的全局唯一索引
2. 使用 `safe_create_unique_index` 存储过程创建新的租户级唯一索引

### 任务 1.4：清理 orders 表冗余索引 ✅

**删除11个冗余/低效索引**：

| 索引名 | 删除原因 |
|--------|---------|
| `idx_order_number` | 被 `uk_tenant_order_number` UNIQUE覆盖 |
| `idx_customer` | 被 `idx_orders_tenant_customer` 复合索引覆盖 |
| `idx_status` | 被 `idx_orders_tenant_status` 复合索引覆盖 |
| `idx_created_by` | 被 `idx_orders_tenant_created_by` 复合索引覆盖 |
| `idx_created_at` | 被 `idx_orders_tenant_created_at` 复合索引覆盖 |
| `idx_payment_status` | 被 `idx_orders_tenant_payment_status` 复合索引覆盖 |
| `idx_mark_type` | 极低基数（仅3-5个值），MySQL优化器不会使用 |
| `idx_is_todo` | 布尔值（0/1），基数仅2，索引几乎无效 |
| `idx_order_source` | 低基数（5-10种来源），独立查询极少 |
| `idx_shipping_time` | VARCHAR字段，极少作为独立查询条件 |
| `idx_expected_delivery_date` | 独立查询频率很低 |

**优化效果**：
- 索引数量：27个 → 16个（含PK），减少41%
- 预期写入性能提升：20-30%
- 磁盘空间节省：减少11个索引的B+树存储

### 任务 1.6：日志表添加过期清理策略 ✅

**新增服务**：`backend/src/services/DataCleanupService.ts`

**清理策略**：
| 表 | 清理条件 | 默认保留期 |
|----|---------|-----------|
| operation_logs | created_at < 90天前 | 90天 |
| logs | createdAt < 90天前 | 90天 |
| notifications | is_read = 1 AND created_at < 90天前 | 90天（仅清理已读） |

**定时执行**：
- 注册到 `SchedulerService`，任务名：`data-cleanup`
- 执行周期：每月1日凌晨3:00（cron表达式：`0 3 1 * *`）
- 支持手动触发：`schedulerService.triggerTask('data-cleanup')`
- 支持环境变量自定义保留天数：`LOG_RETENTION_DAYS=90`

**容错处理**：
- 每个清理操作独立 try/catch，单个失败不影响其他
- notifications 表兼容两种可能的列名（is_read/status + created_at/createdAt）
