## 🔴 严重漏洞 (Critical) — 3项

### C-1: mysql2 依赖存在远程代码执行漏洞 (RCE)

- **文件**：`backend/package.json` → `mysql2@3.6.5`
- **CVE编号**：CVE-2024-21508 (CRITICAL)、CVE-2024-21511 (CRITICAL)、CVE-2024-21512 (HIGH)、CVE-2024-21507 (MEDIUM)、CVE-2024-21509 (MEDIUM)
- **风险描述**：mysql2 3.6.5 存在 **5个已知CVE漏洞**，其中2个为严重级别（远程代码执行），攻击者可通过精心构造的请求实现RCE。
- **影响范围**：整个后端数据库层，所有使用mysql2的查询均受影响。
- **修复方案**：升级 `mysql2` 至 `>=3.9.8`
- **修复命令**：
  ```bash
  cd backend && npm install mysql2@latest
  ```

### C-2: TypeORM SQL注入漏洞

- **文件**：`backend/package.json` → `typeorm@0.3.17`
- **CVE编号**：CVE-2025-60542 (HIGH)
- **风险描述**：TypeORM 0.3.17 的 `repository.save()` 和 `repository.update()` 方法在处理嵌套对象时存在 **SQL注入漏洞**。攻击者可通过构造 `{city: {name: "x", role: "admin"}}` 形式的payload注入额外的SQL列更新。
- **影响范围**：所有使用 `save()` / `update()` 的路由，包括客户管理、订单管理、用户管理等核心业务。
- **修复方案**：升级 `typeorm` 至 `>=0.3.26`
- **修复命令**：
  ```bash
  cd backend && npm install typeorm@latest
  ```

### C-3: JWT默认密钥未强制替换

- **文件**：`backend/src/config/jwt.ts` (第18-19行)
- **代码**：
  ```typescript
  private static readonly ACCESS_TOKEN_SECRET: string = process.env.JWT_SECRET || 'crm-secret-key';
  private static readonly REFRESH_TOKEN_SECRET: string = process.env.JWT_REFRESH_SECRET || 'your-refresh-secret-key';
  ```
- **风险描述**：JWT密钥有硬编码的默认值 `'crm-secret-key'` 和 `'your-refresh-secret-key'`。如果部署时未设置环境变量 `JWT_SECRET`，攻击者可使用默认密钥伪造任何用户的JWT Token，获取系统全部权限。
- **影响范围**：所有认证体系，攻击者可冒充任何用户（包括管理员）。
- **当前状态**：`.env` 文件中 `JWT_SECRET` 仍为模板值 `your_jwt_secret_key_here_change_in_production`，且 `JWT_REFRESH_SECRET` 未配置。
- **修复方案**：
  1. 启动时检测JWT密钥是否为默认值，如果是则拒绝启动或强制报错
  2. 生产环境 `.env` 必须设置随机生成的强密钥
  3. 删除代码中的硬编码默认值，改为启动时强制校验

## 🟠 高危漏洞 (High) — 8项

### H-1: multer 文件上传组件存在7个DoS漏洞

- **文件**：`backend/package.json` → `multer@1.4.5-lts.1`
- **CVE编号**：CVE-2025-47935, CVE-2025-47944, CVE-2025-48997, CVE-2025-7338, CVE-2026-2359, CVE-2026-3304, CVE-2026-3520 (均为HIGH)
- **风险描述**：当前版本multer存在 **7个高危DoS漏洞**，攻击者通过发送恶意构造的文件上传请求即可导致服务器崩溃。
- **影响范围**：所有文件上传接口（头像上传、系统图片上传、版本包上传、录音上传、COD凭证上传等）。
- **修复方案**：升级 `multer` 至 `>=2.1.1`（注意：multer 2.x API有breaking changes，需适配）
- **特别注意**：multer 2.x 移除了 `diskStorage`，需要改用新的存储引擎API

### H-2: nodemailer 存在DoS和命令注入漏洞

- **文件**：`backend/package.json` → `nodemailer@6.9.7`
- **CVE编号**：CVE-2025-14874 (HIGH), CVE-2025-13033 (MEDIUM), GHSA-9h6g-pr28-7cqp (MEDIUM), GHSA-c7w3-x93f-qmm8 (LOW)
- **风险描述**：nodemailer存在递归调用导致的DoS漏洞（通过恶意邮件地址触发栈溢出）和SMTP命令注入。
- **影响范围**：邮件发送功能（密码重置、通知邮件、订阅扣款通知等）。
- **修复方案**：升级 `nodemailer` 至 `>=8.0.4`

### H-3: axios 存在DoS漏洞

- **文件**：`backend/package.json` → `axios@1.12.2`
- **CVE编号**：CVE-2026-25639 (HIGH)
- **风险描述**：`mergeConfig` 函数在处理含 `__proto__` 键的配置对象时崩溃，可通过用户控制的JSON输入触发。
- **影响范围**：所有使用axios发起HTTP请求的服务（支付回调、快递API、短信服务、版本检查等）。
- **修复方案**：升级 `axios` 至 `>=1.13.5`

### H-4: 登出未实现Token黑名单机制

- **文件**：`backend/src/routes/auth.ts` (第109-116行)
- **代码**：
  ```typescript
  router.post('/logout', authenticateToken, (req, res) => {
    // 在实际应用中，这里可以将令牌加入黑名单
    // 目前只是返回成功响应，客户端负责清除令牌
    res.json({ success: true, message: '登出成功' });
  });
  ```
- **风险描述**：用户登出后，旧的JWT Token仍然有效（直到过期）。如果Token泄露，攻击者可在Token有效期内冒充该用户。生产环境Token有效期为7天，风险窗口较大。
- **修复方案**：
  1. 实现Token黑名单（推荐使用Redis存储已注销的Token JTI）
  2. 或者实现Token版本号机制（用户表增加token_version字段，登出时递增）

### H-5: Admin后台验证码可跳过

- **文件**：`backend/src/routes/admin/auth.ts` (第73-88行)
- **代码**：
  ```typescript
  // 🔒 验证码校验（安全加固 - 向下兼容：前端传了验证码才校验）
  if (captchaId && captchaCode) { ... }
  ```
- **风险描述**：Admin后台登录验证码是**可选的**（为了向下兼容）。攻击者只需不传`captchaId`和`captchaCode`参数即可完全绕过验证码保护，使验证码形同虚设。结合15分钟100次的登录限流，管理后台仍存在暴力破解风险。
- **修复方案**：生产环境强制要求验证码（删除可选逻辑），或至少添加配置开关 `ADMIN_CAPTCHA_REQUIRED=true`

### H-6: Admin认证中间件JWT密钥回退值不一致

- **文件**：`backend/src/middleware/adminAuth.ts` (第37行)
- **代码**：
  ```typescript
  const decoded = jwt.verify(token, process.env.JWT_SECRET || 'admin-secret-key') as any;
  ```
- **文件**：`backend/src/config/jwt.ts` (第18行)
- **代码**：
  ```typescript
  private static readonly ACCESS_TOKEN_SECRET: string = process.env.JWT_SECRET || 'crm-secret-key';
  ```
- **风险描述**：Admin认证和CRM认证使用不同的默认密钥回退值（`'admin-secret-key'` vs `'crm-secret-key'`）。如果环境变量未配置，两个系统的JWT密钥不一致会导致认证异常。更重要的是，两个弱默认密钥都可能被攻击者利用。
- **修复方案**：统一密钥管理，删除所有硬编码的默认密钥值，启动时强制校验环境变量

### H-7: 公开注册接口缺少限流保护

- **文件**：`backend/src/routes/public/register.ts`、`backend/src/routes/public/payment.ts`
- **风险描述**：所有 `/api/v1/public/` 下的公开接口（注册、支付创建、验证码发送等）仅受通用限流保护（15分钟10000次），没有像登录接口那样的专用严格限流。
  - **注册接口**：攻击者可批量注册租户消耗资源
  - **验证码发送**：虽有1分钟内频率限制，但按IP维度无独立限流，可消耗短信配额
  - **支付创建**：可批量创建无效订单
- **修复方案**：为敏感公开接口添加专用限流中间件（如注册5次/小时、验证码10次/小时）

### H-8: 文件上传路径穿越风险（Admin上传）

- **文件**：`backend/src/routes/admin/upload.ts` (第24行)
- **代码**：
  ```typescript
  const category = (req.query.category as string) || 'general';
  const dir = path.join(uploadBaseDir, category);
  ```
- **风险描述**：`category` 参数直接来自用户查询字符串，未做路径穿越检查。攻击者可传入 `category=../../etc` 等值，将文件上传到任意目录。
- **影响范围**：Admin管理后台上传接口（需Admin认证，但仍有风险）。
- **修复方案**：
  1. 对 `category` 进行白名单校验（仅允许 `general|image|cert|version`）
  2. 或使用 `path.basename()` 过滤路径分隔符
  3. 校验最终路径是否在 `uploadBaseDir` 内

## 🟡 中危漏洞 (Medium) — 7项

### M-1: Express框架存在开放重定向和XSS漏洞

- **文件**：`backend/package.json` → `express@4.18.2`
- **CVE编号**：CVE-2024-29041 (MEDIUM), CVE-2024-43796 (LOW)
- **风险描述**：Express 4.18.2 存在通过 `res.redirect()` 的开放重定向和XSS漏洞。
- **修复方案**：升级 `express` 至 `>=4.20.0`

### M-2: 验证码存储使用内存Map（无分布式支持）

- **文件**：
  - `backend/src/routes/admin/auth.ts` (第16行) — Admin验证码
  - `backend/src/routes/public/register.ts` (第34行) — 注册验证码
- **风险描述**：验证码存储在内存 `Map` 中。如果使用PM2多进程或多服务器部署，验证码在不同进程间不共享，导致验证码验证失败或可被绕过。攻击者可通过多次请求命中不同进程来绕过频率限制。
- **修复方案**：使用Redis存储验证码（项目已有可选的ioredis依赖）

### M-3: 错误响应泄露敏感信息

- **文件**：`backend/src/middleware/errorHandler.ts` (第116-134行)
- **风险描述**：错误处理中间件在日志中记录了完整的请求头（`req.headers`，包含Authorization Token）和请求体（`req.body`，可能包含密码）。虽然日志不直接暴露给用户，但如果日志文件被泄露，将导致大量敏感数据泄露。
- **代码**：
  ```typescript
  const logData = {
    request: {
      headers: req.headers,    // ⚠️ 包含 Authorization Token
      body: req.body,          // ⚠️ 可能包含密码
    },
  };
  ```
- **修复方案**：日志中过滤敏感字段（Authorization、password、token等）

### M-4: 部分Controller直接返回error.message给客户端

- **文件**：多个Controller文件（如 `ProductController.ts`、`QRConnectionController.ts` 等）
- **代码示例**：
  ```typescript
  res.status(500).json({
    success: false,
    message: '操作失败',
    error: error instanceof Error ? error.message : '未知错误'  // ⚠️ 泄露内部错误信息
  });
  ```
- **风险描述**：在生产环境中，内部错误信息（如数据库错误、文件系统错误）不应返回给客户端，可能泄露数据库表结构、文件路径等敏感信息。
- **影响范围**：约20+个Controller/路由文件。
- **修复方案**：生产环境统一返回通用错误消息，仅开发环境返回详细错误

### M-5: 密码强度要求过低

- **文件**：`backend/src/middleware/validation.ts` (第124-129行)
- **代码**：
  ```typescript
  password: Joi.string().min(6).max(128).required()
  ```
- **风险描述**：密码最低仅要求6个字符，无复杂度要求（大小写、数字、特殊字符）。对于SaaS系统管理后台，此密码策略过于宽松。
- **修复方案**：至少要求8字符，包含大写字母+小写字母+数字（可配置化）

### M-6: API速率限制使用内存缓存

- **文件**：`backend/src/middleware/apiRateLimit.ts`
- **风险描述**：API调用次数计数器存储在内存 `Map` 中，进程重启后清零。多进程部署时各进程独立计数，实际限流效果大打折扣（N个进程 = 允许N倍的请求量）。
- **修复方案**：使用Redis作为限流计数器的共享存储

### M-7: WebSocket连接未设置心跳超时清理

- **文件**：`backend/src/services/WebSocketService.ts`
- **风险描述**：WebSocket连接建立后，如果客户端异常断开（未发送断开包），连接对象可能长期驻留在内存Map中，造成内存泄漏。在高并发场景下，僵尸连接会消耗服务器资源。
- **修复方案**：添加心跳检测机制，定期清理超时未响应的连接

## 🔵 低危漏洞 (Low) — 6项

### L-1: CORS配置包含localhost开发地址

- **文件**：`backend/.env` (第48行)
- **代码**：
  ```
  CORS_ORIGIN=https://yunkes.com,...,http://localhost:5173,http://localhost:3000
  ```
- **风险描述**：生产环境 `.env` 的 CORS 配置中包含 `http://localhost:5173` 和 `http://localhost:3000`。虽然攻击者无法直接利用，但在某些场景下（如DNS重绑定攻击）可能存在风险。
- **修复方案**：生产环境移除所有localhost来源

### L-2: 健康检查端点泄露系统信息

- **文件**：`backend/src/app.ts` (第196-216行)
- **代码**：
  ```typescript
  app.get('/health', (req, res) => {
    res.json({
      version: process.env.npm_package_version || '1.0.0',
      environment: process.env.NODE_ENV || 'development',
      onlineUsers: webSocketService.getOnlineUsersCount()
    });
  });
  ```
- **风险描述**：健康检查端点对外暴露了系统版本号、运行环境和在线用户数。攻击者可据此判断系统类型和版本，辅助针对性攻击。
- **修复方案**：健康检查仅返回 `{ status: "ok" }`，详细信息仅在认证后提供

### L-3: 根路由暴露全部API端点列表

- **文件**：`backend/src/app.ts` (第219-238行)
- **风险描述**：根路由 `GET /` 返回完整的API端点列表（auth、users、customers、products、orders、dashboard），相当于为攻击者提供了API目录。
- **修复方案**：生产环境移除或简化根路由响应

### L-4: 开发环境登录跳过限流

- **文件**：`backend/src/app.ts` (第152-155行, 第242-246行)
- **代码**：
  ```typescript
  skip: (_req) => {
    return process.env.NODE_ENV === 'development';  // 开发环境跳过登录限流
  }
  ```
- **风险描述**：如果生产服务器 `NODE_ENV` 未正确设置为 `production`，登录限流将被完全跳过。
- **修复方案**：将逻辑改为仅在明确的生产环境启用限流 → 改为除非明确标记为开发环境否则默认启用限流（安全默认原则）

### L-5: 前端v-html存在潜在XSS（但已有sanitize）

- **文件**：多个Vue组件（Dashboard.vue、MessageBell.vue、AnnouncementPopup.vue等）
- **风险描述**：前端有16处使用了 `v-html` 指令。经审查，大部分使用了 `sanitizeHtml()` 进行净化（`src/utils/sanitize.ts` 实现了白名单标签+属性过滤），但有少数地方直接使用了 `v-html` 未经净化：
  - `SearchDebug.vue` — 5处直接v-html（调试页面，影响较小）
  - `AnnouncementPopup.vue` (第14行) — `truncateContent` 而非 `sanitizeHtml`
  - `Shipping.vue` / `OrderDetailDialog.vue` — `highlightKeywords` 未确认是否经过净化
- **修复方案**：所有 `v-html` 使用统一经过 `sanitizeHtml()` 处理

### L-6: bcrypt salt rounds偏低

- **文件**：`backend/src/routes/admin/auth.ts` (第240行, 第292行)
- **代码**：
  ```typescript
  const hashedPassword = await bcrypt.hash(password, 10);
  ```
- **风险描述**：Admin密码哈希使用 `salt rounds = 10`，而 `.env.example` 配置的 `BCRYPT_ROUNDS=12`。部分代码未读取环境变量配置，直接硬编码为10。虽然10轮在当前算力下尚可接受，但不够统一。
- **修复方案**：统一从环境变量读取 `BCRYPT_ROUNDS`，默认值设为12
