## 问题清单与修复方案

### 问题1（严重）: 支付回调未触发扩容激活

**现象**: 扩容订单支付成功后，`PaymentService.updateOrderStatus()` 调用 `activateTenant()` 处理，但扩容订单的 `payment_orders` 中无 `package_id`，导致 `activateTenant()` 静默失败。`CapacityService.activateCapacity()` 虽已实现但从未被调用。

**修复**: 在 `updateOrderStatus()` 中通过 `orderNo.startsWith('CAP')` 识别扩容订单，改为调用 `capacityService.activateCapacity(orderNo)`。

**影响路径**:
- 微信支付回调 → `handleWechatNotify()` → `updateOrderStatus()` ✅
- 支付宝回调 → `handleAlipayNotify()` → `updateOrderStatus()` ✅
- 模拟支付 → `/mock-pay/:orderNo` → `updateOrderStatus()` ✅

---

### 问题2（严重）: 续费/升级会重置扩容额度

**现象**: `activateTenant()` 中 SQL `max_users = ?` 使用 `pkg.max_users || 3`，续费后 `max_users` 被重置为套餐基础值，不包含 `extra_users`。

**修复**: SQL 改为：
```sql
max_users = ? + COALESCE(extra_users, 0),
max_storage_gb = ? + COALESCE(extra_storage_gb, 0),
```

---

### 问题3（中）: 自动关闭待支付订单逻辑误伤扩容订单

**现象**: 套餐支付成功后，自动关闭同租户所有 pending 订单，包括未支付的扩容订单。

**修复**: 按订单类型分离关闭逻辑：
- 扩容订单 → 只关闭同租户其他 CAP 开头的 pending 订单
- 套餐订单 → 只关闭同租户非 CAP 开头的 pending 订单

---

### 问题4（中）: 关闭订单未同步 capacity_orders

**现象**: `closeOrder()` 只更新 `payment_orders`，不更新 `capacity_orders`，导致两表状态不一致。

**修复**: `closeOrder()` 检测 CAP 前缀，同步关闭 `capacity_orders` 中的对应记录。

---

### 问题5（中）: 自动关闭时 capacity_orders 未同步

**现象**: `updateOrderStatus()` 自动关闭同类型 pending 订单时，只操作 `payment_orders`，遗漏 `capacity_orders`。

**修复**: 在自动关闭逻辑中，若为扩容订单，额外执行 `UPDATE capacity_orders SET status='closed' WHERE tenant_id=? AND status='pending' AND order_no!=?`。

- 扩容相关全部 API 路由（Admin + Public）

### 审查结果

| 检查项 | 结果 | 说明 |

扩容管理功能的租户数据隔离设计完善，不同租户即使存在同一手机号也无法访问对方数据。

## 扩容功能完整流程

```
用户操作                           后端处理                              数据库变更
──────                           ──────                              ──────────
1. 会员中心选择扩容方案            →  验证 JWT + 查价格配置
2. 确认购买 + 选支付方式           →  createOrder()                    → capacity_orders (pending)
                                                                     → payment_orders (pending)
3. 生成二维码                     →  /public/payment/repay/:orderNo   → payment_orders.qr_code
4. 用户扫码支付                   →  微信/支付宝回调
5. 支付回调                      →  updateOrderStatus()
   ↓ 识别 CAP 前缀               →  capacityService.activateCapacity()
   ↓ 更新扩容订单状态                                                 → capacity_orders (paid)
   ↓ 增加扩容额度                                                    → tenants.extra_users += N
   ↓ 同步更新上限                                                    → tenants.max_users = pkg基础 + extra
   ↓ 关闭同类待支付订单                                               → 其他 CAP pending → closed
6. 前端轮询查询                   →  /public/payment/query/:orderNo   → 返回 status=paid
7. 刷新会员资料 + 订单列表         →  MemberService.getProfile()       → 包含最新 extra_users/extraStorageGb
```

## 续费时扩容保留机制

```
续费/升级支付成功 → activateTenant()
  ↓ SQL: max_users = pkg.max_users + COALESCE(extra_users, 0)
  ↓ SQL: max_storage_gb = pkg.max_storage_gb + COALESCE(extra_storage_gb, 0)
  → 扩容额度自动保留，不会因续费重置
```

## 注意事项

1. **扩容订单号规则**: 以 `CAP` 开头，如 `CAP20260405143050ABCDEF`，用于区分套餐订单
2. **双表关联**: 每个扩容订单同时在 `capacity_orders` 和 `payment_orders` 中创建记录
3. **状态同步**: 支付成功/关闭时两表状态保持一致
4. **幂等性**: `activateCapacity()` 查询 `status='pending'` 防止重复激活
5. **向后兼容**: `extra_users`/`extra_storage_gb` 默认值为 0，不影响未扩容的租户
