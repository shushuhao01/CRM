## 二、问题清单（按优先级分类）

### 🔴 高优先级（安全+数据完整性）

#### P1. 支付配置加密使用固定零向量IV — 安全漏洞

**位置**：`backend/src/services/PaymentService.ts:16-19`，以及 `SubscriptionService.ts`、`WechatPayService.ts`、`AlipayService.ts`、`routes/public/payment.ts`、`routes/admin/payment.ts`（共6处）

**问题**：AES-256-CBC 加密使用 `Buffer.alloc(16, 0)` 作为固定IV（全零初始化向量），这严重削弱了加密安全性。AES-CBC 模式要求每次加密使用随机且唯一的IV。

**当前代码**：
```typescript
const decipher = crypto.createDecipheriv('aes-256-cbc',
  crypto.scryptSync(ENCRYPT_KEY, 'salt', 32),
  Buffer.alloc(16, 0))  // ← 全零IV
```

**风险等级**：🔴 高 — 攻击者可利用相同明文产生相同密文的特性进行密文分析

**修复方案**：由于此为存量数据解密逻辑（admin后台保存时已用固定IV加密），**不能直接更改IV**，否则旧数据无法解密。建议：
1. 短期：在默认加密密钥 `crm-payment-secret-key-2024` 上添加与JWT相同的生产环境强制检查
2. 中期：新增加密函数使用随机IV（IV拼接在密文前），同时保留旧解密兼容

#### P2. wecom.ts 路由使用 console.log/console.error — 生产环境信息泄露

**位置**：`backend/src/routes/wecom.ts`（39处 console.log/console.error）

**问题**：整个 wecom.ts 路由文件未使用项目统一的 `log` logger，而是直接使用 `console.log/console.error`。在生产环境中：
- console.log 输出不受日志等级控制，可能泄露敏感信息（access token、用户ID、配置信息等）
- 不进入 Winston 日志文件，无法审计
- 第28行直接打印 `msg_signature`、`timestamp` 等回调参数

**风险等级**：🔴 高 — 生产环境信息泄露 + 审计失效

**修复方案**：将所有 `console.log` 替换为 `log.info`，`console.error` 替换为 `log.error`

#### P3. 支付加密密钥硬编码默认值 — 在6个文件中重复声明

**位置**：6个文件中均有 `process.env.PAYMENT_ENCRYPT_KEY || 'crm-payment-secret-key-2024'`

**问题**：
1. 硬编码弱默认密钥 `crm-payment-secret-key-2024`（无生产环境强制检查）
2. 同一密钥在6个文件中重复声明，违反DRY原则，修改一处容易遗漏其他

**风险等级**：🔴 高 — 支付配置密钥安全性不足

**修复方案**：
1. 创建统一的加密工具模块 `backend/src/utils/paymentCrypto.ts`
2. 添加生产环境弱密钥检查（参考 jwt.ts 的 `INSECURE_SECRETS` 机制）
3. 所有6个文件引用统一模块

#### P4. 租户资源限制中间件 checkTenantLimits 核心功能为 TODO 注释

**位置**：`backend/src/middleware/tenantAuth.ts:164-211`

**问题**：`checkTenantLimits` 中间件的用户数限制和存储空间限制均为 TODO 注释，实际代码直接 `next()` 放行。虽然 `backend/src/middleware/checkTenantLimits.ts` 中有独立实现，但 `tenantAuth.ts` 中导出的同名中间件是空壳。

**风险等级**：🟡 中 — 如果某些路由引用了 tenantAuth.ts 中的 checkTenantLimits，则租户资源限制将形同虚设

**修复方案**：移除 tenantAuth.ts 中的空壳 checkTenantLimits，确保所有引用指向 `middleware/checkTenantLimits.ts` 的完整实现

#### P5. LicenseService 中用户数统计未过滤租户ID

**位置**：`backend/src/services/LicenseService.ts:50-52`

**问题**：
```typescript
const userCountResult = await AppDataSource.query(
  `SELECT COUNT(*) as count FROM users WHERE status = 'active'`
)
```
此查询统计所有活跃用户而非当前租户的用户，在SaaS多租户模式下会返回全平台用户数。

**风险等级**：🔴 高 — SaaS模式下授权用户数校验不准确，可能导致超限使用

**修复方案**：添加 `tenant_id` 过滤条件，或使用 `tenantHelpers` 工具函数

### 🟡 中优先级（性能+代码质量）

#### P6. apiRateLimit.ts 在模块顶层 setInterval — 影响测试和模块加载

**位置**：`backend/src/middleware/apiRateLimit.ts:84`

**问题**：模块文件末尾直接执行 `setInterval(cleanupRateLimitCache, 60 * 60 * 1000)` 作为副作用，导致：
1. Jest 测试时会创建无法清理的定时器（需 `--forceExit`）
2. 模块被 import 就自动启动定时器，无法控制生命周期

**修复方案**：改为导出 `startCleanupSchedule()` 函数，在 `app.ts` 启动时显式调用

#### P7. 前端测试覆盖极低 — 仅2个测试文件21个测试

**位置**：`src/__tests__/`

**问题**：CRM前端约254个Vue组件 + 151个TS文件，仅有 `customerCode.test.ts`(8) 和 `dataMask.test.ts`(13) 两个工具函数测试。核心业务逻辑（认证流程、API请求拦截、状态管理、路由守卫）无测试。

**修复方案**：优先补充以下测试：
1. `authApiService` 登录/登出/Token刷新流程测试
2. `request.ts` HTTP拦截器和错误处理测试
3. 关键 Store（user、permission）的单元测试

#### P8. Admin后台和Website官网无任何测试

**位置**：`admin/`、`website/`

**问题**：两个子系统均无测试框架配置和测试文件。Admin后台管理租户、支付、授权等核心商业功能，完全无测试覆盖。

**修复方案**：至少为Admin后台添加Vitest配置和API层测试

#### P9. 后端多个定时任务使用 setInterval 而非 node-cron

**位置**：`backend/src/app.ts:415`（消息清理24h间隔）

**问题**：`setInterval` 不能精确控制执行时间（如每天凌晨3点），且进程重启后计时器重置。项目已依赖 `node-cron`，但部分定时任务仍用 `setInterval`。

**修复方案**：将 `scheduleMessageCleanup` 中的 `setInterval(cleanupExpiredMessages, 24 * 60 * 60 * 1000)` 改为使用 `SchedulerService` 统一调度

#### P10. 数据库连接池大小硬编码50 — 可能不够大规模SaaS使用

**位置**：`backend/src/config/database.ts:139`

**问题**：连接池默认 `connectionLimit: 50`，可通过环境变量 `DB_CONNECTION_LIMIT` 配置。但 `connectTimeout: 60000`（60秒）过长，大量并发请求在连接池满时会排队60秒才超时。

**修复方案**：将 `connectTimeout` 降为 `10000`（10秒），添加 `acquireTimeout: 10000`

### 🟢 低优先级（代码规范+优化建议）

#### P11. tenantAuth.ts 中 checkTenantLimits 与 checkTenantLimits.ts 命名冲突

**位置**：`backend/src/middleware/tenantAuth.ts:164` vs `backend/src/middleware/checkTenantLimits.ts`

**问题**：两个不同文件导出同名 `checkTenantLimits`，一个是空壳（TODO），一个是完整实现。容易导入错误。

**修复方案**：重命名 tenantAuth.ts 中的为 `tenantLimitsPlaceholder` 或直接删除

#### P12. PaymentService 创建订单前每次都 loadConfig + ensurePaymentOrderColumns

**位置**：`backend/src/services/PaymentService.ts:133-134`

**问题**：每次创建订单都执行 `ALTER TABLE` 检查（虽然有条件判断），是不必要的性能开销。

**修复方案**：增加静态标志位 `private columnsChecked = false`，首次检查后跳过

#### P13. MobileWebSocketService 使用旧的默认密钥

**位置**：`backend/src/services/MobileWebSocketService.ts:194`

**问题**：`const jwtSecret = process.env.JWT_SECRET || 'crm-secret-key'` 使用了不安全的默认密钥 `crm-secret-key`，且不走 `JwtConfig.getAccessTokenSecret()` 统一路径。

**修复方案**：改为使用 `JwtConfig.getAccessTokenSecret()`

#### P14. 数据库实体注册遗漏

**位置**：`backend/src/config/database.ts:99-118`

**问题**：对照 entities 目录，以下实体未在 `database.ts` 的 entities 数组中注册：
- `Package.ts` — 租户套餐实体
- `PrivateCustomer.ts` — 私有客户实体
- `TenantLog.ts` — 租户日志实体
- `TenantSettings.ts` — 租户设置实体
- `AdminOperationLog.ts` — 管理员操作日志实体
- `Changelog.ts` — 变更日志实体

这些实体虽然定义了但未注册到TypeORM DataSource中，如果被路由引用会导致运行时错误。

**修复方案**：在 entities 数组中补充注册缺失的实体
