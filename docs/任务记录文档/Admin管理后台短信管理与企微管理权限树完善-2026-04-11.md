## 五、权限系统架构总结

```
用户登录 → 获取角色权限列表(permissions[])
                ↓
         存入 Pinia userStore
                ↓
    ┌─────────────────────────────────────┐
    │                                     │
    ↓                                     ↓
路由守卫                             侧边栏菜单
(meta.permission)                 (hasMenu/hasPerm)
    ↓                                     ↓
页面访问控制                      一级/二级菜单显隐
                ↓
         v-permission 指令
                ↓
         按钮级操作控制
```

### 权限码命名规范

```
{模块}:{子模块/页面}:{操作}
```

- **模块**: `sms-management`、`wecom-management`、`settings` 等
- **子模块**: `template-review`、`chat-archive`、`admins` 等
- **操作**: `view`（查看）、`edit`（编辑）、`create`（新建）、`delete`（删除）

### 权限检查流程

1. **一级菜单显隐**: `hasMenuPermission(moduleCode)` — 只要有该模块下任一子权限即可显示
2. **二级菜单显隐**: `hasPermission(permCode)` — 需要具体的 `*:view` 权限
3. **路由访问**: 路由守卫检查 `meta.permission` 字段
4. **按钮操作**: `v-permission` 指令检查 `*:edit`/`*:create`/`*:delete` 权限

## 六、验证方法

### 测试步骤

1. 登录超级管理员账号
2. 进入「系统设置 > 角色管理」
3. 点击「新增角色」
4. 在权限树中确认「短信管理」和「企微管理」节点完整显示
5. 勾选部分权限（如只勾选 `查看模板审核`，不勾选 `执行模板审核`）
6. 保存角色并分配给测试用户
7. 用测试用户登录，验证：
   - 能看到模板审核页面 ✅
   - 通过/拒绝按钮不可见 ✅（因为没有 edit 权限）

### 注意事项

- `v-permission` 指令只控制**前端UI显隐**，后端API仍需做权限校验
- 超级管理员（`role: 'super_admin'`）自动拥有所有权限，不受限制
- 角色权限存储为 JSON 数组，包含叶子节点和半选父节点的权限码
