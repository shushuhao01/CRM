## 六、支付闭环流程

```
用户操作                           后端处理                              数据库变更
──────                           ──────                              ──────────
1. 会员中心选择扩容方案            →  验证 JWT + 查价格配置
2. 确认购买 + 选支付方式           →  createOrder()                    → capacity_orders (pending)
                                                                      → payment_orders (pending)
3. 生成支付二维码                  →  /public/payment/repay/:orderNo   → payment_orders.qr_code
4. 用户扫码支付                   →  微信/支付宝回调通知
5. 支付回调                      →  updateOrderStatus()
   ↓ 识别 CAP 前缀               →  capacityService.activateCapacity()
   ↓ CAS更新状态(防重复)                                               → capacity_orders (paid)
   ↓ 增加扩容额度                                                     → tenants.extra_users += N
   ↓ 同步更新上限值                                                    → tenants.max_users = 基础 + extra
   ↓ 关闭同类待支付订单                                                → 其他 CAP pending → closed
6. 前端轮询                      →  /public/payment/query/:orderNo    → 返回 status=paid
7. 刷新页面                      →  MemberService.getProfile()        → 包含最新扩容数据
```

## 七、关键设计决策

### 7.1 订单号规则
- 扩容订单以 `CAP` 开头，如 `CAP20260405143050ABCDEF`
- 套餐订单使用普通编号
- 通过前缀区分，在支付回调中决定走不同的激活逻辑

### 7.2 双表关联
每个扩容订单同时在两个表中创建记录：
- `capacity_orders` — 扩容业务专用表，记录扩容类型、数量、单价
- `payment_orders` — 通用支付订单表，复用现有支付流程

### 7.3 续费保留扩容额度
`activateTenant()` 中的SQL改为：
```sql
max_users = ? + COALESCE(extra_users, 0),
max_storage_gb = ? + COALESCE(extra_storage_gb, 0)
```
确保续费/升级套餐后，扩容额度不被重置。

### 7.4 数据隔离
- 会员端：tenantId 从 JWT token 中获取（memberAuth 中间件），不可伪造
- 管理端：由 adminAuthMiddleware 保护，仅管理员可访问
- 扩容订单查询：严格按 `WHERE tenant_id = ?` 过滤

### 7.5 幂等性
- `activateCapacity()` 使用 `WHERE status='pending'` 的CAS操作，防止并发重复激活
- MySQL 行锁保证同一订单不会被多次处理

## 十一、注意事项

1. `capacity_orders` 和 `payment_orders` 双表状态必须同步（支付成功/关闭时）
2. 扩容额度在续费时通过 `COALESCE(extra_users, 0)` 自动保留
3. 私有客户的扩容记录通过 licenses → tenants 的关联查询实现
4. 管理端全部路由受 `adminAuthMiddleware` 保护
5. 会员端全部数据操作以 JWT 中的 tenantId 为准，不接受前端传入
6. `ensureTables()` 在应用启动时自动执行，确保表结构存在
7. `extra_users` 和 `extra_storage_gb` 字段默认值为 0，不影响未扩容的租户
