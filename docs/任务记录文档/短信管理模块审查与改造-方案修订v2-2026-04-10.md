## 一、两套短信的本质区分

### ⚠️ 极其重要：不要搞混！

| 维度 | 管理后台系统短信 | CRM业务短信 |
|------|----------------|-------------|
| **谁在用** | 运营方（管理后台） | 租户/私有客户（CRM系统） |
| **用途** | 官网注册验证码、开通通知、续费提醒、支付成功通知等**平台级**短信 | 租户给自己的客户发的**业务短信**：营销推广、订单通知、发货通知、催收等 |
| **密钥归属** | 运营方自己的阿里云账号 | 租户自己的短信服务商账号（阿里云/腾讯云/华为云） |
| **配置位置(当前)** | Admin → 系统设置 → 运营配置 → 短信配置Tab | Admin → 模块服务 → 配置下发 → 短信设置卡片 / CRM → 系统管理 → 系统设置 → 短信设置 |
| **配置存储** | `system_config` 表 `sms_config` key | `system_config` 表 `distributedConfig.sms` + CRM端 `sms_settings` |
| **短信模板** | 运营方在阿里云自己申请的模板（VERIFY_CODE等） | **租户需要申请 → 管理后台审核 → 提交服务商 → 获得模板CODE** |

## 二、修订后的管理后台"短信管理"一级菜单

### 2.1 菜单结构（4个二级菜单）

```
📱 短信管理 (一级菜单)
  ├── 📊 短信记录        /sms-management/records          — 查看所有租户CRM短信发送记录
  ├── 📋 模板审核        /sms-management/template-review   — 审核租户申请的CRM短信模板
  ├── ⚙️ CRM短信配置    /sms-management/crm-config        — 配置/管控租户的短信服务商密钥等
  └── 📈 统计总览        /sms-management/statistics        — 全平台CRM短信用量统计(可选后期)
```

> **注意**：管理后台自身的"系统短信配置"（用于发验证码等）**仍留在** 系统设置 → 运营配置 → 短信配置Tab，**不移动**。因为那是运营方自己的事情，和CRM业务短信完全无关。

### 2.2 各页面说明

#### 2.2.1 短信记录 `/sms-management/records`
- **数据来源**: `sms_records` 表，跨租户查询（`AppDataSource.getRepository(SmsRecord)`，不用 `getTenantRepo`）
- **显示字段**: 租户名称/ID、发送人、收件人（脱敏）、短信内容、使用模板、模板CODE、发送时间、状态（成功/失败/发送中）
- **筛选**: 按租户、时间范围、状态、模板筛选
- **搜索**: 搜索内容、收件人手机号

#### 2.2.2 模板审核 `/sms-management/template-review`
- **数据来源**: `sms_templates` 表，跨租户查询
- **功能**: 
  - 查看所有租户提交的模板申请（待审核 / 已审核列表）
  - 审核操作：初审通过 → 手动填入服务商模板CODE → 最终激活 / 拒绝
  - 显示租户名称、模板名称、模板内容、变量、服务商模板CODE、审核状态

#### 2.2.3 CRM短信配置 `/sms-management/crm-config`
- **从配置下发迁出**: 原 `Distribute.vue` 的 `sms` 卡片的编辑表单迁到这里
- **配置内容**: 短信服务商选择、AccessKey、SecretKey、短信签名、每日/月限额、启用开关、发送审核开关
- **管控开关**: 开启管控后统一下发给所有CRM端
- **存储不变**: 仍存储在 `distributedConfig.sms`，CRM端读取逻辑不变

## 三、模板审核完整流程（四阶段）

```
┌──────────────────┐   ┌────────────────────────┐   ┌──────────────────┐
│   CRM端（租户）   │   │   管理后台（运营方）     │   │  服务商（阿里云等）│
│                  │   │                        │   │                  │
│ ① 租户提交模板申请 ──→ 收到待审核模板           │   │                  │
│   (名称/内容/变量) │   │ status: pending_admin   │   │                  │
│                  │   │                        │   │                  │
│                  │   │ ② 管理员审核内容合规     │   │                  │
│                  │   │   - 内容有违规 → 拒绝    │   │                  │
│                  │   │   - 内容合规 → 初审通过  │   │                  │
│                  │   │   管理员去服务商平台提交  ──→  ③ 服务商审核       │
│                  │   │   status: pending_vendor │   │   （1-2个工作日）  │
│                  │   │                        │   │                  │
│                  │   │   管理员查到审核结果     ←── 通过：返回模板CODE │
│                  │   │   或 服务商拒绝         ←── 拒绝：返回原因    │
│                  │   │                        │   │                  │
│                  │   │ ④ 管理员最终操作        │   │                  │
│                  │   │   服务商通过:            │   │                  │
│                  │   │   → 填入模板CODE        │   │                  │
│                  │   │   → 标记 active          │   │                  │
│                  │   │   服务商拒绝:            │   │                  │
│                  │   │   → 填拒绝原因          │   │                  │
│                  │   │   → 标记 rejected        │   │                  │
│                  │   │                        │   │                  │
│ 租户获得可用模板  ←── 下发到该租户可用模板列表   │   │                  │
│ 发短信时选择模板  │   │                        │   │                  │
│ 实际用模板CODE发送 │   │                        │   │                  │
└──────────────────┘   └────────────────────────┘   └──────────────────┘
```

### 3.1 状态流转枚举

```
pending_admin     → 租户已提交，等待管理员初审
pending_vendor    → 管理员初审通过，已提交/正在提交给服务商
vendor_approved   → 服务商审核通过（中间态）
vendor_rejected   → 服务商审核拒绝
active            → 管理员最终确认激活，租户可用
rejected          → 管理员拒绝（任一阶段可拒绝）
```

### 3.2 SmsTemplate 实体新增字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `vendor_template_code` | VARCHAR(100) | 服务商返回的模板CODE（如 SMS_123456789） |
| `vendor_status` | VARCHAR(20) | 服务商审核状态：pending/approved/rejected |
| `vendor_submit_at` | TIMESTAMP | 提交服务商时间 |
| `vendor_reject_reason` | TEXT | 服务商拒绝原因 |
| `admin_reviewer` | VARCHAR(50) | 管理后台审核人 |
| `admin_review_at` | TIMESTAMP | 管理后台审核时间 |
| `admin_review_note` | TEXT | 管理后台审核备注/拒绝原因 |

### 3.3 关键说明：服务商审核是手动流程

第一期不做服务商API自动对接（阿里云AddSmsTemplate等），原因：
- 各服务商API接口不同，对接成本高
- 模板审核通常1-2个工作日，异步流程
- 多数中小客户直接在服务商控制台操作

**实际操作流程**：
1. 管理员初审通过 → 标记 `pending_vendor`
2. 管理员手动登录服务商控制台提交模板（或通知租户自行在服务商提交）
3. 收到服务商审核结果后，管理员在后台填入模板CODE → 标记 `active`
4. 如服务商拒绝，管理员填入拒绝原因 → 标记 `rejected`
