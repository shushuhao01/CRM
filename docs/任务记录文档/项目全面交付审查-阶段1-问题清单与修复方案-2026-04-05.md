## 🔴 高优先级问题（数据隔离/安全）

### P1-01：auth.ts 认证中间件用户查询未添加租户隔离

**文件**：`backend/src/middleware/auth.ts` 第66-69行
**问题**：`authenticateToken` 中间件使用 `dataSource.getRepository(User)` 直接查询用户，未按 `tenantId` 过滤。虽然用户ID是UUID不太可能冲突，但在理论上存在跨租户风险，且不符合纵深防御原则。

```typescript
// 当前代码（第66-69行）：
const userRepository = dataSource.getRepository(User);
user = await userRepository.findOne({
  where: { id: payload.userId }
});
```

**风险等级**：🟡 中（UUID冲突概率极低，但安全审计不通过）
**修复方案**：在查询条件中添加 `tenantId` 过滤（从JWT payload中获取）

```typescript
const where: any = { id: payload.userId };
if (payload.tenantId) {
  where.tenantId = payload.tenantId;
}
user = await userRepository.findOne({ where });
```

**同样影响**：`optionalAuth`（第207-209行）存在相同问题

### P1-02：auth.ts 用户认证缓存未按租户隔离

**文件**：`backend/src/middleware/auth.ts` 第53行
**问题**：缓存key为 `auth:user:${payload.userId}`，未包含 `tenantId`。理论上不同租户的用户ID不同（UUID），但缓存设计不够严谨。

**修复方案**：缓存key改为 `auth:user:${payload.tenantId || 'default'}:${payload.userId}`

---

### P1-03：systemConfig.ts 支付方式路由使用 AppDataSource.getRepository 但手动处理了租户

**文件**：`backend/src/routes/system/systemConfig.ts` 第500、536、580、645、704行
**问题**：支付方式相关路由使用 `AppDataSource.getRepository(PaymentMethodOption)` 而非 `getTenantRepo()`，虽然手动通过 `TenantContextManager.getTenantId()` 添加了过滤条件（包含全局预设 `tenant_id IS NULL`），但PUT/DELETE操作中存在风险——查询条件是 `pm.id = :id AND (pm.tenant_id = :tenantId OR pm.tenant_id IS NULL)`，这意味着任何租户都可以**修改全局预设的系统支付方式**。

**风险等级**：🟡 中
**修复方案**：
- PUT操作：对系统预设(`isSystem && tenantId === null`)只允许修改 `isEnabled` 和 `sortOrder`，不允许修改 `label` 和 `value` ✅ 已部分实现
- DELETE操作：已有保护（不允许删除系统预设）✅ 

**结论**：支付方式路由的租户隔离**已做好**，但建议统一使用 `getTenantRepo` 以保持一致性。

### P1-04：systemSettings.ts 模块状态查询未按租户隔离

**文件**：`backend/src/routes/system/systemSettings.ts` 第56行
**问题**：`AppDataSource.getRepository(Module)` 查询所有模块状态，但 `modules` 表是全局管理的（Admin后台统一配置），所有租户共享相同的模块启用状态。

**风险等级**：🟢 低（模块是全局配置，非租户数据）
**结论**：**设计合理**，后续已通过租户features取交集过滤。

---

### P1-05：wecom.ts 回调接口遍历全部租户配置

**风险等级**：🟢 低（回调接口无认证上下文，遍历查找是标准做法）
**结论**：**设计合理**，企微回调接口需要遍历配置才能识别来源租户。

```typescript
await AppDataSource.query(
  `INSERT INTO customer_assignments (id, customer_id, ...) VALUES (?, ?, ...)`,
  [...]

```

### P2-02：Proxy缓存key可能导致租户切换后数据混淆

**文件**：`backend/src/utils/tenantRepo.ts` 第104、105行
**问题**：Proxy缓存TTL为5分钟。在SaaS模式下，同一Node进程处理不同租户请求时，`getTenantRepo()` 先获取tenantId再查缓存，逻辑正确。但如果同一请求中tenantId发生变化（极端情况），可能使用错误的缓存。

**风险等级**：🟢 低（实际场景中不会发生）
**结论**：可保持现状。

---

### P2-03：前端localStorage存储敏感数据

**文件**：多个前端Vue文件
**问题**：部分前端文件将角色列表、用户数据存储到 `localStorage`（如 `crm_mock_users`、`crm_roles`），虽然这些是前端本地缓存不涉及跨租户，但命名中包含"mock"可能是历史遗留。

**风险等级**：🟢 低
**修复方案**：清理mock相关的localStorage键名。

---

### P2-04：静态文件上传目录无租户隔离

**文件**：`backend/src/app.ts` 第185-186行
**问题**：`/uploads` 和 `/recordings` 静态文件服务没有按租户隔离目录，所有租户共享相同的上传路径。

```typescript
app.use('/uploads', express.static(path.join(process.cwd(), 'uploads')));
app.use('/recordings', express.static(path.join(process.cwd(), 'recordings')));
```

**风险等级**：🟡 中（如果文件名可预测，其他租户可能访问到文件）
**修复方案**：生产环境建议使用OSS存储+签名URL，或在静态文件路径中包含tenant_id。当前文件名使用UUID，实际被猜中的概率极低。
