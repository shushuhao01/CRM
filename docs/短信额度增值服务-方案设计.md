# 短信额度增值服务 — 整体方案设计

> 📅 编写日期：2026-04-10
> 🏷️ 状态：待确认

---

## 一、需求概述

短信发送作为增值服务，每条短信消耗一定金额。需要建立完整的**短信额度管理体系**，包括额度购买、消耗、账单、套餐管理，覆盖 **CRM端**、**会员中心**、**管理后台** 三端。

---

## 二、功能总览

| 端 | 功能 | 说明 |
|---|---|---|
| **CRM端** | 短信额度按钮 + 弹窗 | 查看额度、购买套餐、支付、账单记录 |
| **CRM端** | 发送短信前额度校验 | 余额不足时禁止发送 |
| **会员中心** | 短信额度一级菜单页面 | 客户直接购买额度，购买后CRM同步生效 |
| **管理后台** | 短信管理 > 额度套餐（二级菜单） | 创建/编辑/删除套餐、设置单条价格 |
| **管理后台** | 支付管理 | 短信额度订单的支付记录、报表、退款 |

## 三、数据库设计（2张新表）

### 3.1 `sms_quota_packages` — 短信额度套餐表

| 字段 | 类型 | 说明 |

| sms_count | INT | 套餐包含短信条数 |

| description | VARCHAR(500) | 套餐描述 |
| sort_order | INT DEFAULT 0 | 排序权重 |

| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 3.2 `sms_quota_orders` — 短信额度购买订单表

| 字段 | 类型 | 说明 |
|---|---|---|

| order_no | VARCHAR(50) UNIQUE | 订单号 |

| status | VARCHAR(20) DEFAULT 'pending' | 订单状态: pending/paid/refunded/closed |

### 3.3 利用已有 `system_configs` 表存储租户额度

通过 `system_configs` 表（已存在）存储每个租户的短信额度状态，键值对方式：

| config_key | 含义 | 示例值 |
|---|---|---|
| `sms_quota_total` | 总额度（累计购买条数） | 5000 |
| `sms_quota_used` | 已使用条数 | 1234 |
| `sms_unit_price` | 单条短信价格（全局，管理后台设置） | 0.045 |

### 4.4 发送短信额度校验（修改已有接口）

在 `backend/src/routes/sms.ts` 的 `POST /send` 接口中增加额度校验：
- 发送前：检查 `sms_quota_total - sms_quota_used >= recipients.length`
- 发送后：`sms_quota_used += successCount`

## 七、关键技术要点

1. **支付复用**：复用现有 `PaymentService.ts` 的微信/支付宝支付能力，调用 `createOrder` 创建订单生成二维码
2. **额度同步**：CRM端和会员中心共享同一个 `system_configs` 表中的额度数据（按 tenant_id 隔离），购买后实时生效
3. **退款逻辑**：退款金额 = (剩余未用条数 / 购买总条数) × 购买金额，退款后从总额度中扣减对应条数
4. **支付回调**：支付成功后回调自动增加租户短信额度（`sms_quota_total += sms_count`）
5. **并发安全**：额度扣减使用数据库事务 + 乐观锁（`UPDATE ... WHERE used <= total`）
