# 会话存档购买模式双轨制改造记录

> 日期：2026-04-21
> 需求：支持代购+服务费自购双模式，管理后台可控，会员中心自适应

---

## 一、需求概述

### 业务背景
会话存档是企微官方收费功能，客户需要购买席位。原有流程仅支持"平台代购"一种模式，
实际业务中存在两种场景：
1. **代购模式**：客户通过平台一站式购买（含官方席位费+平台服务费）
2. **服务费模式**：客户自行在企微后台购买席位，仅向平台支付服务费

### 需求要点
- 管理后台可配置全局购买模式：「仅代购」或「代购+服务费自购」
- 管理后台可配置每席位服务费单价（元/人/年）
- 会员中心根据后台配置自动展示：仅代购 / 双模式可选
- 套餐是预设优惠人数包，客户也可自定义任意人数购买
- 两种模式都支持预设套餐和自选人数

## 二、改动文件清单

### 2.1 管理后台

**`admin/src/views/wecom/Pricing.vue`**
- Tab2「会话存档定价」重构
- 新增：全局购买模式 Radio（`proxy_only` / `both`）
- 新增：每席位服务费输入框（当模式为 `both` 时显示）
- 新增：`archiveGlobalConfig` 数据字段，保存到后端
- 简化档位卡片：移除每卡片的 `archiveType` 字段（已提升为全局配置）

### 2.2 后端

**`backend/src/routes/admin/wecom-management.ts`**
- `PUT /pricing-config`：保存时将 `archivePricing` + `archiveGlobalConfig` 自动同步到 `wecom_vas_config`
- 同步内容包括：`purchaseMode`、`seatServiceFee`、`tierPricing`、`presetPackages`
- 解决了原有的 admin 保存 `wecom_pricing_config` 但会员中心读取 `wecom_vas_config` 的数据不互通问题

**`backend/src/routes/public/member-wecom.ts`**
- `POST /purchase`：新增 `purchaseMode` 请求参数
- 根据 `purchaseMode` 和后台 `configPurchaseMode` 计算实际费用：
  - `proxy`：使用阶梯定价（代购售价）
  - `service_fee`：使用 `seatServiceFee × userCount`
- 订单名称和备注区分模式（代购/服务费）
- `packageId` 区分：`vas_chat_archive` / `vas_chat_archive_service_fee`

### 2.3 会员中心

**`website/src/views/member/wecom/WecomPricing.vue`**
- 新增：购买模式切换 UI（当 `vasConfig.purchaseMode === 'both'` 时显示）
- 代购模式：保持原有流程（预设套餐+阶梯定价+自定义人数）
- 服务费模式：仅显示人数输入+服务费单价，无阶梯定价
- 费用说明文案根据模式动态切换
- 提交购买时携带 `purchaseMode` 参数
- 节省金额仅在代购模式下显示

## 三、数据流

```
管理后台 Pricing.vue
  → PUT /admin/wecom-management/pricing-config
    → 保存到 system_config.wecom_pricing_config（完整配置）
    → 同步到 system_config.wecom_vas_config（会员中心专用格式）
      ├─ chatArchive.purchaseMode: 'proxy_only' | 'both'
      ├─ chatArchive.seatServiceFee: number
      ├─ chatArchive.tierPricing: [{min, max, price}]
      └─ chatArchive.presetPackages: [{users, label}]

会员中心 WecomPricing.vue
  → GET /public/member/wecom（读取 wecom_vas_config）
    → 根据 purchaseMode 展示 UI
  → POST /public/member/wecom/purchase（下单）
    ← purchaseMode: 'proxy' | 'service_fee'
    → 后端根据模式计算金额
```

---

## 四、权限控制逻辑

| 管理后台配置 | 会员中心展示 | 允许的购买模式 |
|-------------|-------------|--------------|
| `proxy_only` | 仅代购流程（无模式切换） | proxy |
| `both` | 双模式可选（代购/服务费） | proxy 或 service_fee |

## 五、第二轮修复（2026-04-21 续）

### 问题定位

1. **会员中心看不到自购模式**：后端 `GET /public/member/wecom` 返回的 `vasConfig` 遗漏了 `purchaseMode` 和 `seatServiceFee` 字段
2. **会员中心套餐UI不够直观**：原有6列小方块不显示套餐详情，需要改为CRM风格的卡片式
3. **CRM只看到服务费套餐**：`GET /wecom/pricing-config` 未返回 `archiveGlobalConfig`，CRM前端无法判断购买模式
4. **CRM无模式切换和自选人数**：`PackagePurchaseTab.vue` 的 archive 类型硬编码按旧的 per-tier `archiveType` 渲染

### 修改文件

| 文件 | 修改说明 |
|------|---------|
| `backend/src/routes/public/member-wecom.ts` | vasConfig 返回增加 `purchaseMode`、`seatServiceFee` |
| `website/src/views/member/wecom/WecomPricing.vue` | 套餐区重设计为卡片式；两种模式均展示套餐卡；新增价格计算辅助函数；支付操作栏美化 |
| `backend/src/routes/wecom/pricing.ts` | `GET /pricing-config` 附加 `archiveGlobalConfig`；`POST /purchase-archive` 支持 `userCount`+`purchaseMode`，自选任意人数 |
| `src/api/wecom.ts` | `purchaseArchivePackage` 参数扩展支持 `userCount`、`purchaseMode` |
| `src/views/Wecom/components/PackagePurchaseTab.vue` | archive 类型完全重写：代购/自购模式切换按钮、卡片式套餐展示、自选人数区域、阶梯定价联动 |

### 数据流更新

```
CRM PackagePurchaseTab.vue
  → GET /wecom/pricing-config
    ← archivePricing: [{tierLabel, maxMembers, salePrice, ...}]
    ← archiveGlobalConfig: {purchaseMode, seatServiceFee}
  → 根据 archiveGlobalConfig.purchaseMode 展示模式切换
  → POST /wecom/purchase-archive
    ← {tierId?, userCount, purchaseMode}
    → 后端根据模式计算金额（代购用 salePrice，服务费用 seatServiceFee）

会员中心 WecomPricing.vue
  → GET /public/member/wecom
    ← vasConfig.purchaseMode / vasConfig.seatServiceFee（新增）
    ← vasConfig.tierPricing（卡片数据来源）
  → 卡片式展示，选中套餐后自动填充自定义人数区
```

## 六、注意事项

1. 保存定价配置时自动同步两个 config key，无需手动维护
2. 后端对 `purchaseMode` 做了安全校验：即使前端传 `service_fee`，后台配置为 `proxy_only` 时仍按代购处理
3. 订单的 `package_id` 和 `remark` 均标记了购买模式，便于后续对账和统计
4. CRM 后端 purchase-archive 同时支持预设套餐（tierId）和自选人数（userCount），两种路径均根据 purchaseMode 计算
5. 所有改动均通过 lint 检查，未引入新错误
