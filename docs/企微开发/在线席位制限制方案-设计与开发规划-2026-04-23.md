## 一、必要性分析

### 1.1 当前模式的局限性

当前系统采用 **"限制注册用户总数"** 模式：

```
tenant_packages.max_users = 10   →  该租户最多创建 10 个用户账号
tenants.max_users = 10           →  同步到租户表
tenants.extra_users = 0          →  扩容的额外用户数
```

**问题**：

| 问题 | 说明 |
|------|------|
| **1. 人员流动浪费名额** | 员工离职后账号虽可禁用，但已占用名额（`COUNT(*) WHERE tenant_id=?`统计的是所有创建过的用户），除非物理删除用户，否则新员工无法创建。实际实现中统计的是 `status='active'` 的用户，稍好一些，但禁用再启用会反复触发上限。 |
| **2. 不适合大团队 + 轮班制** | 一个 50 人团队可能同时在线只有 20 人（早晚班轮换），但需要购买 50 人套餐。 |
| **3. 增购粒度粗** | 扩容只能扩"总账号数"，客户会觉得为离职员工买单不合理。 |
| **4. 竞品已普遍采用席位制** | 企业微信、钉钉、Salesforce、HubSpot 等均采用"在线席位/活跃席位"模式，更符合 SaaS 行业惯例。 |

### 1.2 席位制的优势

| 优势 | 说明 |
|------|------|
| **降低客户采购门槛** | 50 人团队只需买 20 个在线席位，套餐价格更低，转化率更高 |
| **持续增购动力** | 业务增长 → 同时在线人数增加 → 自然触发席位增购，MRR 持续增长 |
| **更灵活的人员管理** | 注册不限数量，人员变动无压力 |
| **符合行业标准** | 与企微/钉钉/Salesforce 模式一致，客户理解成本低 |

### 1.3 结论：有必要，但建议两种模式并存

> **推荐方案：保留"限制总用户数"，新增"限制在线席位数"，由管理后台为每个套餐/租户选择限制模式。**

理由：
- **私有部署**客户通常偏好"总用户数"限制（买断制，简单明确）
- **SaaS** 客户更适合"在线席位"制（按需付费，灵活高效）
- 两种模式可在套餐层面切换，同一平台可同时运营两类套餐

## 二、方案概述

### 2.1 限制模式定义

| 模式 | 代码标识 | 限制对象 | 适用场景 |
|------|---------|---------|---------|
| **总用户数** | `user_limit_mode = 'total'` | 注册用户总数 | 私有部署、买断制 |
| **在线席位** | `user_limit_mode = 'online'` | 同时在线人数 | SaaS订阅制 |

### 2.2 在线席位的核心逻辑

```
在线席位模式下：
├── 注册用户数 → 不限制（或设一个很大的上限如 99999）
├── 同时在线人数 → 限制 ≤ max_online_seats
├── "在线"定义 → 持有有效 JWT 且在最近 30 分钟内有 API 请求活动
├── 超出时 → 新用户登录被拒绝，提示"在线席位已满"
├── 增购 → 增加 extra_online_seats
└── 管理员 → 可强制踢出指定用户释放席位
```

### 2.3 "在线"判定策略

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **A. WebSocket 连接数** | 实时准确 | 刷新页面断连/多标签页多算/移动端无WS | ❌ |
| **B. JWT Token 存在即在线** | 简单 | Token 7天有效，远超"在线"含义 | ❌ |
| **C. 心跳活跃时间** ⭐ | 准确、可控 | 需前端定期心跳 | ✅ **推荐** |
| **D. API 请求活跃时间** ⭐ | 无需额外心跳 | 精度稍低 | ✅ **备选** |

**推荐方案 C+D 混合**：
- 前端每 5 分钟发一次心跳 `POST /api/v1/heartbeat`
- 后端同时在任意 API 请求中间件里更新 `last_active_at`
- 判定标准：**最近 30 分钟内有活动即算"在线"**
- 超过 30 分钟无活动自动释放席位

### 2.4 整体架构变更图

```
                    ┌─────────────────────────────────┐
                    │         套餐/租户配置             │
                    │  user_limit_mode: 'total'|'online'│
                    │  max_users: 10 (总数模式)         │
                    │  max_online_seats: 5 (席位模式)   │
                    └──────────┬──────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
    ┌─────▼─────┐      ┌──────▼──────┐     ┌───────▼──────┐
    │ 创建用户时  │      │  用户登录时   │     │  心跳/请求时  │
    │ checkUser  │      │ checkLogin  │     │ updateActive │
    │ Limit      │      │ Seats       │     │ At           │
    └─────┬─────┘      └──────┬──────┘     └───────┬──────┘
          │                    │                    │
    mode='total'?        mode='online'?       更新user_sessions
    → COUNT users        → COUNT active         .last_active_at
    → vs max_users         sessions
                         → vs max_online_seats
```

### 3.6 字段关系总结

```
tenant_packages (套餐定义)
├── max_users            ← 总用户数限制 (mode=total 时生效)
├── max_online_seats     ← 在线席位限制 (mode=online 时生效)  [新增]
└── user_limit_mode      ← 'total' | 'online'               [新增]

tenants (租户实例)
├── max_users            ← 同步自套餐 + extra_users
├── max_online_seats     ← 同步自套餐 + extra_online_seats   [新增]
├── extra_users          ← 已有，扩容用户数
├── extra_online_seats   ← 扩容在线席位                      [新增]
└── user_limit_mode      ← 继承套餐                         [新增]

user_sessions (会话追踪)                                      [新建表]
├── 登录时创建记录
├── 心跳/API请求时更新 last_active_at
├── 30分钟无活动 → 标记 expired
└── 管理员可手动 kicked
```

### 4.4 新增 Service：`OnlineSeatService`

**核心方法**：

| `createSession(userId, tenantId, token, deviceType, ip, ua)` | 登录时创建会话记录 |
| `updateActivity(sessionToken)` | 心跳/请求时更新活跃时间 |
| `getOnlineCount(tenantId)` | 获取租户当前在线人数（30min内活跃） |
| `getOnlineSessions(tenantId)` | 获取在线会话列表（含用户信息） |
| `canLogin(tenantId)` | 检查是否可以登录（席位是否已满） |
| `kickSession(sessionId, adminId)` | 管理员踢出指定会话 |
| `kickUser(userId, adminId)` | 管理员踢出用户所有会话 |
| `cleanExpiredSessions()` | 定时清理过期会话（30min无活动） |
| `getMaxSeats(tenantId)` | 获取最大席位 = max_online_seats + extra_online_seats |
| `getSeatStats(tenantId)` | 获取席位统计（在线/总席位/剩余） |

**关键实现逻辑**：

```typescript
class OnlineSeatService {
  private readonly ACTIVE_THRESHOLD_MINUTES = 30; // 30分钟无活动视为离线

  /**
   * 检查租户是否可以登录新用户
   */
  async canLogin(tenantId: string): Promise<{
    allowed: boolean;
    message?: string;
    onlineCount: number;
    maxSeats: number;
  }> {
    // 1. 查询租户限制模式
    const tenant = await this.getTenant(tenantId);
    if (!tenant || tenant.userLimitMode !== 'online') {
      return { allowed: true, onlineCount: 0, maxSeats: 99999 };
    }

    // 2. 先清理过期会话
    await this.cleanExpiredSessions(tenantId);

    // 3. 统计当前在线人数（去重 user_id，一个用户多设备只算1个席位）
    const onlineCount = await this.getOnlineCount(tenantId);
    const maxSeats = tenant.maxOnlineSeats + tenant.extraOnlineSeats;

    if (onlineCount >= maxSeats) {
      return {
        allowed: false,
        message: `在线席位已满（${onlineCount}/${maxSeats}），请等待其他用户下线或联系管理员增加席位`,
        onlineCount,
        maxSeats
      };
    }

    return { allowed: true, onlineCount, maxSeats };
  }

  /**
   * 获取在线用户数（按 user_id 去重）
   */
  async getOnlineCount(tenantId: string): Promise<number> {
    const threshold = new Date(Date.now() - this.ACTIVE_THRESHOLD_MINUTES * 60 * 1000);
    const result = await AppDataSource.query(
      `SELECT COUNT(DISTINCT user_id) as count
       FROM user_sessions
       WHERE tenant_id = ? AND status = 'active' AND last_active_at >= ?`,
      [tenantId, threshold]
    );
    return result[0]?.count || 0;
  }
}
```

export const checkUserLimit = async (req, res, next) => {
  // ... 已有逻辑 ...

  // SaaS 模式
  const tenant = await tenantRepository.findOne({ where: { id: tenantId } });

  if (tenant.userLimitMode === 'online') {
    // 在线席位模式：创建用户不限制（只在登录时限制）

    if (currentUserCount >= softLimit && softLimit > 0) {

        success: false,
        message: `注册用户数已达上限（${currentUserCount}/${softLimit}），请清理不活跃账号`,
        code: 'USER_SOFT_LIMIT_EXCEEDED'
      });
    }
    return next();
  }

  // 总用户数模式：保持原有逻辑
  if (currentUserCount >= tenant.maxUsers) {
    return res.status(403).json({ ... });
  }
  next();
};

class HeartbeatService {
  private timer: number | null = null;
  private readonly INTERVAL = 5 * 60 * 1000; // 5分钟

  start() {
    this.stop();
    this.timer = window.setInterval(() => {
      api.post('/heartbeat').catch(() => {});
    }, this.INTERVAL);
  }

  stop() {
    if (this.timer) {
      clearInterval(this.timer);
      this.timer = null;
    }
  }
}

## 九、风险与注意事项

### 9.1 技术风险

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| **心跳增加 API 请求量** | 5分钟一次，100用户 = 每分钟20请求 | 心跳接口极轻量，无DB写入可用内存缓存批量更新 |
| **多标签页/多设备重复计算** | 一个用户打开3个标签页算1个还是3个席位？ | 按 `user_id` 去重，一个用户只占1个席位 |
| **PM2 Cluster 模式下的内存状态** | 多进程各自维护在线列表不一致 | 所有状态存 DB/Redis，不依赖进程内存 |
| **时钟偏差** | 30分钟阈值在高负载时可能误判 | 使用数据库时间 `NOW()`，不用 JS `Date.now()` |
| **数据库写入频繁** | 每个 API 请求都更新 last_active_at | 使用内存缓存+定时批量写入（如每60秒批量UPDATE） |

### 9.2 产品风险

| 风险 | 应对 |
|------|------|
| **用户体验：被踢出** | 被踢出时 WebSocket 推送通知，前端弹窗提示"您的席位已被管理员释放" |
| **用户投诉：30分钟太短** | 阈值可配置化，存入 `tenant_settings` 或系统配置 |
| **定价混乱** | 套餐页清晰标注"席位制"/"注册制"标签 |

### 9.3 兼容性保障

| 项目 | 保障措施 |
|------|---------|
| **现有租户不受影响** | `user_limit_mode` 默认值 = `'total'`，所有现有租户自动使用原有逻辑 |
| **渐进式迁移** | 管理员可在 Admin 后台为单个套餐/租户切换模式 |
| **数据库向后兼容** | 全部使用 `ALTER TABLE ADD COLUMN`，不修改现有列 |
| **API 向后兼容** | 所有新字段在 API 响应中为可选字段，前端做 `v-if` 判断 |

### 9.4 "活跃时间"批量更新优化方案

为避免每个 API 请求都写数据库，推荐内存缓存 + 定时批量写入：

```typescript
class OnlineSeatService {
  // 内存中缓存最近的活跃时间
  private activeCache: Map<string, number> = new Map(); // sessionToken → timestamp

  updateActivity(sessionToken: string): void {
    // 只更新内存，不写DB
    this.activeCache.set(sessionToken, Date.now());
  }

  // 每 60 秒批量写入数据库
  async flushActivityToDb(): Promise<void> {
    if (this.activeCache.size === 0) return;
    
    const batch = Array.from(this.activeCache.entries());
    this.activeCache.clear();

    // 批量 UPDATE（使用 CASE WHEN 或多条合并）
    for (const [token, ts] of batch) {
      await AppDataSource.query(
        `UPDATE user_sessions SET last_active_at = ? WHERE session_token = ? AND status = 'active'`,
        [new Date(ts), token]
      );
    }
  }
}
```
