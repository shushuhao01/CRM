# CRM 1.8.0 全项目深度分析与企微三端/虚拟商品专项审计

> 版本：v1  
> 日期：2026-04-21  
> 范围：当前仓库代码现状（CRM系统、官网会员中心、管理后台、后端API、移动端相关能力）  
> 重点：企微管理三端开发现状 + 虚拟商品链路完成度、风险与优化

---

## 1. 项目全景与架构拆解

### 1.1 多端工程分层

- CRM系统（主业务前台）：`src`
- 官网+会员中心：`website/src`
- 管理后台（Admin）：`admin/src`
- 后端API服务：`backend/src`
- 文档与过程沉淀：`docs`

当前仓库是典型的“多前端 + 单后端”单体仓库组织，三端都以 Vue3 为主，后端以 Express + TypeORM 提供统一业务服务。

### 1.2 技术栈与工程基础

#### 前端技术栈

- 核心框架：Vue 3（CRM/Admin/Website）
- 构建工具：Vite 5
- 路由：Vue Router 4
- 状态管理：Pinia
- UI：Element Plus（CRM/Admin）
- 测试：主 CRM 有 Vitest（`src/__tests__`），其余端测试覆盖较弱

#### 后端技术栈

- Node.js + TypeScript + Express
- ORM：TypeORM（MySQL）
- 安全：helmet、cors、rate-limit、JWT
- 日志：winston + morgan
- 测试：Jest（`backend/src/__tests__`）

### 1.3 鉴权与租户模型

- CRM 端：`auth_token`，路由守卫 + 业务权限码
- Admin 端：`admin_token`，菜单权限较细，但后端接口粒度不完全对齐
- 会员中心：`member_token` + 独立 memberAuth 中间件
- 后端核心业务普遍围绕 `tenantId` 做数据隔离

## 2. 核心业务能力地图

### 2.1 CRM主线能力（总体成熟）

- 客户：客户档案、分组标签、跟进关联已形成闭环
- 订单：新增、编辑、审核、发货、状态历史完整
- 商品：实体/虚拟商品并行，库存与类型字段已扩展
- 财务支付：支付与订阅能力可用，但实现复杂度较高
- 短信：模板、审核、发送记录、统计、额度管理较完整
- 企微：V4方向下多模块并行推进，覆盖面广但稳定性不均

### 2.2 后端服务分域（高耦合特征明显）

后端已形成分域路由体系，但部分路由文件体量较大，存在“路由层承载过多业务细节”现象：

- `backend/src/routes/customers/*`
- `backend/src/routes/orders/*`
- `backend/src/routes/wecom/*`
- `backend/src/routes/public/*`
- `backend/src/routes/virtual*`

风险是后续在并发、一致性、权限细分、回归测试上维护成本继续上升。

## 3. 代码风格、质量与工程治理评估

### 3.1 正向信号

- 多端均已 TypeScript 化（程度不一）
- 三端均存在明确入口与路由组织
- 后端日志、错误处理、基础安全中间件较完整
- 企微与虚拟商品有持续迭代文档沉淀

### 3.2 关键治理问题

1. **规范分裂**  
   主 CRM 与 Admin ESLint/依赖版本不统一，工程规范未完全对齐。

2. **迁移体系不完全统一**  
   既有 migration 脚本，也有运行时自动建表/补列逻辑，导致环境一致性风险。

3. **临时脚本与过渡文件较多**  
   根目录和 backend 下 `_*.js/_*.cjs/_*.py` 辅助脚本密集，存在技术债沉淀。

4. **测试覆盖深度不足**  
   关键业务链路（企微套餐、虚拟商品履约、跨表一致性）缺少成体系集成测试。

## 5. 企微管理三端专项审计（重点）

## 5.1 CRM端（`src/views/Wecom/*`）

### 已有能力

- 配置、通讯录、客户/客户群、获客、活码、会话存档、客服、AI、侧边栏、支付套餐均有页面与API入口
- 企微业务在 CRM 端覆盖最广，功能面最完整

### 问题与风险

- 部分 API 命名/路径与后端不一致，存在局部 404 或功能空转风险
- 演示态与真实态代码并存，可能造成用户行为预期偏差
- 某些详情类接口已封装但后端未完整实现

## 5.2 官网会员中心（`website/src/views/member/wecom/*`）

### 已有能力

- 企微服务页、授权、购买、续费、用量、订单闭环已基本打通
- 与 `public/member-wecom` 专用路由契合度较高

### 问题与风险

- AI助手页面仍偏示例化，真实业务联动弱
- 局部交互仍是 `alert` 级别，体验一致性不足

## 5.3 管理后台（`admin/src/views/wecom/*`）

### 已有能力

- 功能菜单覆盖非常广：总览、套餐、配额、租户套餐、订单、系统配置、审计等
- 后台对“运营治理”场景支持较充分

### 问题与风险

- 部分后端管理接口仍是占位返回（开发中）
- 前端有细粒度权限，但后端部分接口仅做 admin 级粗校验，存在越权边界风险

## 5.4 三端一致性结论

- **接口一致性：中等偏上，但存在关键错配点**
- **权限一致性：三套体系并存，后端粒度需补齐**
- **状态一致性：套餐/席位/支付/授权存在“先改状态后支付”类风险点**

## 7. 虚拟商品专项审计（重点）

## 7.1 当前业务链路（已形成雏形）

1. 商品配置：虚拟类型、发货类型、模板等字段
2. 下单：识别虚拟商品，预占库存
3. 审核：虚拟订单分流到 `virtual_pending`
4. 发货：虚拟发货路由推进状态并写记录
5. 领取：用户通过 claim 流程确认领取

链路可跑通，但可靠性能力还未达“高并发生产级”。

## 7.2 主要缺陷与风险

### P0（高优先）

- 预占库存并发安全不足，存在超卖或重复预占风险
- 订单状态变更与库存状态变更缺少统一事务边界
- 发货/领取幂等设计不完整，重复请求风险较高

### P1（中高优先）

- 权限粒度偏粗（谁可改库存/发货/作废需精细化）
- 关键配置（如邮箱密码）安全保护不足
- 支付失败/流程中断后的补偿机制不完善

### P2（中优先）

- 运行时DDL与迁移脚本并存，应收敛到标准迁移
- 缺少库存流水审计表，不利于追溯和对账
- 缺少虚拟商品全链路自动化测试

## 7.3 建议改造路径

### 第一阶段（立即止损，1周）

- 预占改为原子更新（条件更新 + 影响行校验）
- 发货、领取加入幂等键和唯一约束
- 订单/库存/记录三者关键路径事务化

### 第二阶段（稳定化，2~3周）

- 增加补偿任务：超时释放、失败回滚、异常扫描修复
- 增加虚拟库存流水审计表
- 完善 RBAC 权限点与操作审计

### 第三阶段（可运维化，持续）

- 全面移除运行时DDL
- 增加并发与一致性专项测试
- 统一前后端状态机与可视化监控指标

## 8. 交互体验与产品一致性评估

### 8.1 主要体验问题

- 三端在提示风格、异常展示、状态文案上不统一
- 部分“开发中”功能暴露给正式用户路径
- 复杂模块（企微、虚拟履约）缺少明确流程反馈与防误操作机制

### 8.2 建议

- 建立三端统一交互规范（反馈、加载、空态、错误态）
- 对未完成能力使用 feature flag 控制可见性
- 为高风险操作加二次确认、可撤销、操作日志回查入口

## 10. 最终结论

项目整体已经具备较高的业务覆盖度，核心CRM主链路成熟度较好；当前最大挑战不在“有没有功能”，而在“多端高频演进后的一致性、稳定性、可维护性和可回归能力”。

其中：

- **企微管理**：已进入“功能广度完成后，稳定性与一致性治理”的关键阶段；
- **虚拟商品**：主流程已成型，但需尽快补齐并发/事务/幂等/补偿，避免随着业务量上升暴露高成本故障；
- **管理后台与会员中心**：可用性已建立，但仍需将占位实现与体验粗糙点拉齐至生产级标准。

建议你把后续1~2个迭代目标从“继续铺新功能”调整为“稳定化治理优先”，先把上述 P0/P1 问题清掉，再进入下一轮扩展。
