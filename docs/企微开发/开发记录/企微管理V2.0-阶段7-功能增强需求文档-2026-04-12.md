## 需求1：会话存档席位管控

### 1.1 业务背景

企业微信的会话存档接口需按人数向企微官方购买席位。CRM平台作为增值服务独立售卖存档功能。**核心逻辑：租户在CRM购买了N个席位，即使在企微官方购买了更多人数，CRM端也只处理N个人的存档数据**。

> ⚠️ **关键概念澄清**：这里的"席位"和"人数"指的是**企微账号数量**，而非CRM租户成员数量。一个CRM租户成员可以绑定多个企微账号（`wecom_user_bindings` 中同一 `crm_user_id` 可对应多条记录）。收费和席位限制以**绑定的企微账号数量**为准。
>
> 例如：张三和李四两个CRM成员，共绑定了20个企微账号。租户只购买了10个CRM会话存档席位，那么只能选择其中10个企微账号纳入存档范围，不得超出购买数量。

### 1.2 现状分析

| 模块 | 现状 | 差距 |
|------|------|------|
| `wecom_archive_settings` 表 | 已有 `max_users`、`used_users` 字段 | ❌ 购买后未写入 `max_users` |
| `chatArchive.ts` 购买流程 | 仅更新 `status='active'` + `expire_date` | ❌ 未写入购买人数 |
| 同步聊天记录 | 拉取企微所有授权用户聊天 | ❌ 未过滤CRM席位范围 |
| 前端 ChatArchive.vue | 无席位信息展示 | ❌ 无法感知剩余/已用席位 |

### 1.3 功能规格

#### 1.3.1 席位绑定-选择生效企微账号

- 管理员在**存档设置Tab**中选择哪些**企微账号**纳入存档范围
- 可选列表 = `wecom_user_bindings` 中 `isEnabled=true` 的所有企微账号（展示：企微账号名称 + 绑定的CRM用户）
- **一个CRM用户可能对应多条企微账号**，每个企微账号独立占一个席位
- 选择数量 ≤ `max_users`（购买席位数），超出无法选择，弹出提示"当前已用X个/共购Y个企微账号席位，超出需增购"
- 选择的企微账号保存到 `wecom_archive_members` 表

#### 1.3.2 同步过滤

```
同步会话记录时：
1. 读取 wecom_archive_members 表中当前租户的"生效成员" userId 列表
2. 仅拉取这些成员的会话记录（作为 from_user_id 或 to_user_id）
3. 不在生效范围内的成员聊天记录 → 不同步、不存储
```

#### 1.3.3 席位用量展示

- **会话存档页面顶部**显示席位用量条：  
  `📊 存档席位：已用 8/10 个企微账号 ████████░░ 80%` [管理席位]
- 用量 ≥ 90% 时显示橙色警告
- 用量 = 100% 时显示红色，"增购"按钮高亮
- 点击"管理席位"→ 跳到存档设置Tab

#### 1.3.4 增购逻辑

- 已购套餐过期前增购 → 新席位立即生效，到期时间不变
- 增购弹窗复用 PurchaseDialog，显示"当前已购X人，增购后共Y人"

### 1.4 数据模型

**新增表：`wecom_archive_members`**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT AUTO_INCREMENT | 主键 |
| tenant_id | VARCHAR(50) | 租户ID |
| wecom_user_id | VARCHAR(100) | 企微成员userId |
| wecom_user_name | VARCHAR(200) | 成员名称（冗余） |
| crm_user_id | VARCHAR(50) | 关联CRM用户ID |
| is_enabled | BOOLEAN DEFAULT true | 是否启用存档 |
| created_at | DATETIME | 加入时间 |
| updated_at | DATETIME | 更新时间 |

**修改：`wecom_archive_settings` 表**

| 字段变更 | 说明 |
|---------|------|
| `max_users` | 购买后写入购买人数 |
| `used_users` | 实时计算 = `wecom_archive_members` 中 `is_enabled=true` 的数量 |

### 1.5 API设计

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/wecom/chat-archive/seats` | 获取席位信息：`{ maxUsers, usedUsers, members: [...], expireDate }` |
| PUT | `/wecom/chat-archive/seats/members` | 更新生效成员列表（管理员） |
| POST | `/wecom/chat-archive/purchase` | 修改：购买后写入 `max_users` |

#### 2.2.2 后端权限过滤

查询会话记录/会话列表时：

```typescript
// 伪代码逻辑
const visibility = await getArchiveVisibility(tenantId)  // 'self' | 'department' | 'all'
const currentUser = req.currentUser

if (currentUser.role === 'admin' || currentUser.role === 'super_admin') {
  // 管理员不受限制
} else if (visibility === 'self') {
  // 仅查看当前用户绑定的企微成员的会话
  const myWecomUserIds = await getBindingsByUserId(currentUser.id)
  queryBuilder.andWhere('r.from_user_id IN (:...ids) OR r.to_user_id IN (:...ids)', { ids: myWecomUserIds })
} else if (visibility === 'department') {
  // 查看当前用户所在部门所有成员的会话
  const deptUserIds = await getDepartmentWecomUserIds(currentUser.departmentId)
  queryBuilder.andWhere('r.from_user_id IN (:...ids) OR r.to_user_id IN (:...ids)', { ids: deptUserIds })
}
// 'all' 不加过滤
```

## 需求3：增值服务-套餐与定价配置增强

### 3.1 业务背景

当前预设套餐包（5人入门/10人基础/20人标准等）在前端 PurchaseDialog.vue 中**硬编码**。阶梯定价从 `system_config.wecom_vas_config` 读取。需要将套餐包也配置化，并完善收费管理能力。

### 3.2 现状分析

| 项目 | 现状 | 差距 |
|------|------|------|
| 预设套餐包 | 硬编码在 PurchaseDialog.vue | ❌ 无法管理后台配置 |
| 阶梯定价 | 存储在 system_config.wecom_vas_config | ✅ 可配置 |
| 支付方式 | 微信/支付宝/对公转账 | ✅ 已支持 |
| 订单管理 | 存储在 payment_orders 表 | ✅ 基础已有 |
| 续费/增购 | 无 | ❌ 缺失 |

### 3.3 功能规格

#### 3.3.1 预设套餐包配置化

`GET /wecom/chat-archive/vas-pricing` 响应新增字段：

    "wecomFeeNote": "..."
  }

```

#### 3.3.3 续费与增购

**续费**：到期前30天内可续费，延长1年，享受续费折扣  
**增购**：在当前套餐基础上增加席位，到期时间不变，按剩余天数折算单价

```
增购价格 = 增购人数 × 适用单价 × (剩余天数 / 365)
```

PurchaseDialog 新增模式：`type: 'new' | 'renew' | 'upgrade'`

#### 3.3.4 Admin后台配置页面

Admin 后台中的 VAS 配置页面新增预设套餐包编辑功能（可增删改套餐项）。

|------|------|
| `PurchaseDialog.vue` | 删除硬编码 `presetPackages`，改为从API获取；新增续费/增购模式 |
| `ChatArchive.vue` | 顶部增加当前套餐信息卡片 |

### 3.5 API变更

| 方法 | 路径 | 变更 |

```
┌──────────────────────────────────────────────────────────┐
│ 会话存档                           [🔲 全屏] [↗ 新窗口] │
│ ......                                                   │
└──────────────────────────────────────────────────────────┘

**修改：`customers` 表（CRM客户主表）**

> ⚠️ **2026-04-12 补充需求**：CRM客户列表新增USID字段

| 新增字段 | 类型 | 说明 |
|---------|------|------|
| `wecom_external_userid` | VARCHAR(100) NULL | 客户唯一企业微信编码（USID），企微 `external_userid`，租户隔离 |

**USID字段规格说明**：

1. **字段含义**：`wecom_external_userid` 即客户在企业微信中的唯一标识（external_userid），用于CRM客户与企微客户的关联匹配
2. **数据来源**：
   - **自动同步**：当CRM系统用户绑定企微后，同步企微客户时自动匹配（通过手机号/姓名等维度），写入该字段
   - **手动填入**：管理员在CRM客户详情中手动输入USID进行关联
3. **租户隔离**：每个租户的USID独立存储和查询，通过 `tenant_id` + `wecom_external_userid` 组合唯一
4. **侧边栏展示**：在企微侧边栏的客户详情页面中展示USID，支持一键复制（方便客服快速查找定位）
5. **客户列表展示**：CRM客户管理列表中新增"企微USID"列，支持搜索和筛选
6. **唯一索引**：`UNIQUE INDEX uk_tenant_wecom_userid (tenant_id, wecom_external_userid)` — 同一租户下USID不重复

**USID关联流程**：

```
方式1：自动关联
  企微客户同步 → 匹配CRM客户(手机号/姓名) → 自动写入 wecom_external_userid
  
方式2：手动关联
  CRM客户详情 → 编辑USID字段 → 输入/粘贴 external_userid → 保存
  → 后端校验USID格式 + 租户内唯一性 → 写入

方式3：侧边栏命中关联
  侧边栏识别企微客户 → 匹配CRM客户 → 若未关联则自动写入USID
```
