## 五、核心业务逻辑说明

### 5.1 USID唯一性校验

- 更新USID时先查询同租户下是否已有相同USID的其他客户
- 唯一索引 `IDX_customers_tenant_wecom_userid (tenant_id, wecom_external_userid)` 提供数据库级保障
- 冲突时返回409状态码 + 已使用客户名称提示

### 5.2 侧边栏客户详情增强

| 新增字段 | 数据来源 | 脱敏规则 |
|---------|---------|---------|
| shippingAddress | Customer表(地址/手机) | 手机:138****5678, 地址:前6字+*** |
| afterSales | AfterSalesService表 | 无需脱敏 |
| bindingInfo | WecomUserBinding + Tenant | 完整展示 |
| crmCustomer.tenantCode | Tenant表 | 完整展示 |
| crmCustomer.wecomExternalUserid | Customer表 | 完整展示 |

### 5.3 JSON规范遵循

- 所有JSON字段读取使用 `safeJsonParse()` 包裹
- 所有JSON字段存储使用 `JSON.stringify()`
- 标签名称解析: `JSON.parse(tagNames)` 包裹 try-catch
- API响应通过 `res.json()` 自动序列化，禁止手动拼接

### 5.4 换绑流程

```
用户点击[换绑] → 确认弹窗
  → PUT /wecom/sidebar/binding/:id/rebind
  → 后端: binding.isEnabled = false
  → 前端提示: "已解除绑定，请在侧边栏重新登录完成换绑"
  → 侧边栏重新打开 → verify-binding 发现未绑定 → 进入登录绑定流程
```
