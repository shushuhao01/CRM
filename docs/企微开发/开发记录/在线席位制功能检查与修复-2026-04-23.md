## 十、第三轮严重Bug修复（2026-04-23 深夜）

> 发现在线席位统计显示0人、在线成员列表为空、多用户可突破1席位上限登录等严重功能失效问题。

### 10.1 根因分析

经全链路排查，发现 **5个严重bug** 导致在线席位功能完全失效：

| # | Bug | 根因 | 影响 |
|---|-----|------|------|
| 1 | `user_sessions` 表不存在 | `synchronize: false` + 无迁移脚本，所有会话CRUD静默失败 | 全功能失效 |
| 2 | 私有部署模式登录时席位检查被跳过 | `deployConfig.isSaaS()` 条件限制，私有模式直接跳过 | 不限制登录人数 |
| 3 | 心跳被踢状态前端读不到 | 后端返回 `{ success, kicked }` 未包装在 `data` 中，axios拦截器解包后为 undefined | 踢出提醒无效 |
| 4 | 已登录无会话用户被误判为过期 | 心跳接口 `sessionStatus !== 'active'` 对 `not_found` 也返回过期 | 旧会话用户误踢 |
| 5 | `both` 模式在线人数返回0 | `getTenantSeatStats` 和 `checkLoginAllowed` 只检查 `==='online'`，`both` 模式被忽略 | 双模式功能失效 |

### 10.2 修复详情

#### 修复1：OnlineSeatService 自动建表

**文件**：`backend/src/services/OnlineSeatService.ts`

- 新增 `ensureTable()` 方法，使用 `CREATE TABLE IF NOT EXISTS` 自动创建 `user_sessions` 表
- 带 `tableEnsured` 缓存标记，仅首次执行DDL
- 在 `createSession`、`checkLoginAllowed`、`getOnlineCount`、`getOnlineSessions`、`getSessionStatus`、`flushActivityCache` 中调用

#### 修复2：移除 isSaaS() 条件限制

**文件**：`backend/src/controllers/UserController.ts`

- 登录时席位检查从 `if (deployConfig.isSaaS() && user.tenantId)` 改为 `if (user.tenantId)`
- 私有部署和SaaS模式均支持在线席位检查

#### 修复3：标准化心跳API响应格式

**文件**：`backend/src/routes/onlineSeat.ts`

- 所有心跳响应统一为 `{ success: true, data: { ... } }` 格式
- kicked: `{ success: true, data: { kicked: true, message: '...' } }`
- expired: `{ success: true, data: { expired: true, message: '...' } }`
- 正常: `{ success: true, data: { status: 'ok', timestamp: '...' } }`
- 确保前端 axios 拦截器解包后 `res.data.kicked` 路径正确

#### 修复4：心跳接口容错 not_found 状态

**文件**：`backend/src/routes/onlineSeat.ts`

- 仅对 `'kicked'` 返回被踢状态
- 仅对 `'expired'` 和 `'logged_out'` 返回过期状态
- `'not_found'` 和 `'active'` 视为正常，继续更新活跃时间

#### 修复5：兼容 both 模式

**文件**：`backend/src/services/OnlineSeatService.ts`、`src/views/System/User.vue`

- `getTenantSeatStats`：`online` 或 `both` 模式均统计在线人数
- `checkLoginAllowed`：`online` 或 `both` 模式均执行席位检查
- 前端 `enabled` 判断：`mode === 'online' || mode === 'both'`

## 十一、第四轮修复 — 在线状态全链路打通（2026-04-23 深夜）

> 重启后端后在线状态依然全部显示离线、在线人数统计0、在线成员列表空白。

### 11.1 新发现的根因

| # | Bug | 根因 | 影响 |
|---|-----|------|------|
| 6 | 用户列表在线状态判断使用旧逻辑 | `getUsers` 用 `lastLoginAt` 15分钟阈值判断在线（只在登录那一刻更新），与 `user_sessions` 表完全脱节 | 登录15分钟后全员显示离线 |
| 7 | 已登录用户无会话记录 | 后端重启后表新建，但旧JWT对应的session从未创建，`updateActivity` 只做UPDATE不会INSERT | 在线人数永远为0 |

### 11.2 修复方案

#### 修复6：用户列表在线状态改用 user_sessions 表

**文件**：`backend/src/controllers/UserController.ts`

- 新增调用 `onlineSeatService.getOnlineUserIds(tenantId)` 批量查询在线用户ID
- 优先使用 `user_sessions` 表判断（30分钟活跃阈值）
- 无 tenantId 时降级为 `lastLoginAt` 判断

**文件**：`backend/src/services/OnlineSeatService.ts`

- 新增 `getOnlineUserIds(tenantId)` 方法，返回 `Set<string>` 在线用户ID集合

#### 修复7：auth中间件自动补建会话记录

**文件**：`backend/src/services/OnlineSeatService.ts`

- 新增 `ensureSessionExists()` 方法：
  - 内存缓存 `sessionExistsCache` 避免每次请求查DB
  - 首次遇到未缓存的 sessionToken 时查DB，不存在则INSERT
  - `createSession` 成功后也加入缓存
  - `destroySession` / `forceKickSession` 时清除缓存

**文件**：`backend/src/middleware/auth.ts`

- 在现有 `updateActivity` 后异步调用 `ensureSessionExists`
