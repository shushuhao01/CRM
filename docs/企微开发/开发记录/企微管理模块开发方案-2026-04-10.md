## 七、JSON数据处理规范(强制)

本模块涉及JSON字段: tag_ids, user_ids, department_ids, servicer_user_ids, to_user_ids, content

存储: entity.field = JSON.stringify(array)
读取: const data = safeJsonParse(entity.field, [])
禁止手动拼接JSON字符串
API请求体由axios自动序列化, 不手动JSON.stringify
API响应使用res.json(), Express自动序列化

```typescript
function safeJsonParse<T>(str: string | null, fallback: T): T {
  if (!str) return fallback;
  try { return JSON.parse(str) as T; }
  catch { return fallback; }
}
```

## 八、数据量风险分析与优化方案

### 8.1 数据量预估(100租户规模)

| 表 | 预估数据量 | 风险 | 单行大小 |
|---|---|---|---|
| wecom_configs | 约200条 | 无风险 | 约2KB |
| wecom_user_bindings | 约2000条 | 无风险 | 约0.5KB |
| wecom_customers | 约400万条 | 中等(需索引优化) | 约1KB |
| wecom_acquisition_links | 约1000条 | 无风险 | 约1KB |
| wecom_service_accounts | 约500条 | 无风险 | 约0.5KB |
| wecom_chat_records | 约3650万条/年 | 高危(核心风险点) | 约2KB |
| wecom_payment_records | 约10万条/年 | 低风险 | 约0.5KB |

wecom_customers 预计占用: 400万 x 1KB = 约4GB (可接受)
wecom_chat_records 预计占用: 3650万 x 2KB = 约73GB/年 (需要治理)

### 8.2 优化策略

**策略一: wecom_customers 表优化**
- 添加复合索引: (wecom_config_id, external_user_id) UNIQUE
- 添加索引: (wecom_config_id, follow_user_id, status)
- 添加索引: (wecom_config_id, add_time)
- 同步时使用 upsert(已有external_user_id则更新), 避免重复数据

**策略二: wecom_chat_records 全量OSS存储 + MySQL轻量索引**
- 消息原文(文本+媒体)全部存OSS, 数据库不存content原文
- MySQL的wecom_chat_records表只存索引字段:
  id, tenant_id, wecom_config_id, corp_id, msg_id, msg_type, action,
  from_user_id, from_user_name, to_user_ids, room_id, msg_time,
  oss_key(OSS文件路径), is_sensitive, audit_remark, audit_by, audit_time
- OSS存储路径: wecom-chat/{tenant_id}/{year}/{month}/{msg_id}.json
- 每条消息的完整内容(含媒体URL)作为JSON文件存OSS
- 查询流程: MySQL查索引 -> 获取oss_key -> 按需从OSS读取原文
- 媒体文件(图片/语音/视频)单独存: wecom-chat/{tenant_id}/media/{media_key}
- 优势: MySQL单行约0.3KB(原来2KB), 同样数据量体积缩小85%
- 定时归档: MySQL索引表超过180天的记录可归档到冷表, OSS数据永久保留
- 私有部署: OSS可降级为本地磁盘存储(uploads/wecom-chat/)

**策略三: 同步限流**
- 客户同步: 每个租户每小时最多同步1次, 避免API调用过频
- 会话存档: 每次最多拉取1000条, 分批处理
- 收款同步: 每天同步一次即可

**策略四: 会话存档增值服务授权机制(核心)**

会话存档是增值服务, 需要管理后台单独授权给指定租户, 没有授权的租户无法使用。

授权控制两级机制:
  第一级: 管理员授权(Admin操作)
    - 管理后台 -> 租户详情页 -> "增值服务" 区域 -> "会话存档" 授权开关
    - 管理后台 -> 私有客户详情页 -> 同样的授权开关
    - 授权信息存储: tenant_settings 表
      key=wecom_chat_archive_authorized, value=true/false (默认false)
      key=wecom_chat_archive_expire_date, value=2027-04-10 (授权到期时间,可选)
      key=wecom_chat_archive_max_days, value=90 (最大保留天数)
    - 没有授权的租户: CRM中企微管理菜单正常显示, 但会话存档页面显示
      "此功能为增值服务, 请联系管理员开通"

  第二级: 租户启用(租户管理员操作)
    - 有授权的租户 -> CRM企微配置页面 -> 会话存档开关(启用/停用)
    - 启用后: 需要填写会话存档Secret和RSA私钥
    - 启用后: 系统开始定时拉取该租户的会话数据
    - 停用后: 停止拉取, 已有数据保留

判断逻辑(后端API):
  1. 查 tenant_settings: wecom_chat_archive_authorized == true ?
  2. 否 -> 返回 { authorized: false, message: "未授权" }
  3. 是 -> 检查是否过期 -> 过期返回 { authorized: false, message: "授权已过期" }
  4. 未过期 -> 返回 { authorized: true, enabled: 是否已启用, config: 配置信息 }

判断逻辑(CRM前端):
  1. 进入会话存档页面 -> 调用 GET /wecom/chat-archive/status
  2. authorized=false -> 显示增值服务提示卡片 + 联系管理员按钮
  3. authorized=true, enabled=false -> 显示启用引导(填写Secret/私钥)
  4. authorized=true, enabled=true -> 正常显示会话列表

管理后台UI:
  租户详情页 / 私有客户详情页 -> 增值服务区域:
  +-----------------------------------------------+
  | 增值服务                                       |
  |  [x] 会话存档  授权状态: 已授权                 |
  |      授权时间: 2026-04-10                      |
  |      到期时间: 2027-04-10  [修改]               |
  |      保留天数: 90天  [修改]                     |
  |      [撤销授权]                                |
  +-----------------------------------------------+
  (未来可在此区域扩展更多增值服务)

**策略五: 补充tenant_id字段**
现有问题: WecomCustomer, WecomChatRecord等子表缺少tenant_id字段
虽然通过 wecom_config_id -> wecom_configs.tenant_id 可间接关联
但为了查询性能和安全性, 建议在以下表直接加 tenant_id:
- wecom_customers (加 tenant_id 列 + 索引)
- wecom_chat_records (加 tenant_id 列 + 索引)
- wecom_payment_records (加 tenant_id 列 + 索引)
这样查询时直接 WHERE tenant_id=? 无需 JOIN wecom_configs
