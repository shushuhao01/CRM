## ❗ 常见操作错误提醒

> **`SHOW VARIABLES` / `SHOW BINARY LOGS` 等是 SQL 命令，必须在 `mysql>` 提示符下执行！**  
> 不要在 bash/shell 终端直接输入，否则会报 `SHOW: command not found`。

正确操作流程：
```bash
# 1. 先登录MySQL
mysql -u abc789 -p
# 输入密码

# 2. 在 mysql> 提示符下执行SQL
mysql> SHOW VARIABLES LIKE 'log_bin';
mysql> SHOW VARIABLES LIKE 'datadir';
mysql> SHOW BINARY LOGS;
mysql> exit

# 3. 退出后在bash下找文件、用工具
```

> **如果 `mysqlbinlog` 命令不存在，需要先安装：**
```bash
apt install mysql-client-core-8.0 -y
```

## 恢复方案（按优先级排序）

### 方案1：🔥 MySQL Binlog 恢复（成功率最高）

MySQL 默认开启了 binlog（二进制日志），记录了所有数据变更。这是你**最大的希望**。

#### 1.1 检查 binlog 是否开启

```bash
# SSH登录服务器，进入MySQL
mysql -u abc789 -p

# 检查binlog状态
SHOW VARIABLES LIKE 'log_bin';
# 如果值是 ON，就有希望！

# 查看binlog文件列表
SHOW BINARY LOGS;

# 查看当前binlog位置
SHOW MASTER STATUS;
```

#### 1.2 找到误操作的时间点

```bash
# 查看binlog事件，找到DROP TABLE的时间点
mysqlbinlog --no-defaults --base64-output=decode-rows -v \
  /www/server/data/mysql-bin.000XXX | grep -B5 "DROP TABLE" | head -50

# 或者用时间范围查找（替换为你误操作前的时间）
mysqlbinlog --no-defaults --start-datetime="2026-04-19 00:00:00" \
  --stop-datetime="2026-04-19 HH:MM:SS" \
  /www/server/data/mysql-bin.000XXX > /tmp/recovery.sql
```

#### 1.3 用 binlog 恢复数据

```bash
# 先找到误操作前最近的时间点，假设误操作时间是 14:30:00
# 恢复从最早binlog到误操作前的所有数据

# 步骤1：先用schema.sql重建空表结构（你已经做了）
# 步骤2：重放binlog中误操作之前的所有INSERT/UPDATE数据

mysqlbinlog --no-defaults \
  --stop-datetime="2026-04-19 14:29:00" \
  /www/server/data/mysql-bin.000001 \
  /www/server/data/mysql-bin.000002 \
  ... \
  | mysql -u abc789 -p abc789

# ⚠️ 注意：stop-datetime 要设置为误操作之前的时间
```

#### 1.4 binlog 文件位置

宝塔面板 MySQL binlog 常见位置：
```
/www/server/data/mysql-bin.*
/www/server/data/binlog.*
/var/lib/mysql/mysql-bin.*
/var/lib/mysql/binlog.*
```

查找：
```bash
find / -name "mysql-bin.*" -o -name "binlog.*" 2>/dev/null | head -20
```

### 方案2：🔥 宝塔面板自动备份

宝塔面板可能配置了**定时数据库备份**！

#### 2.1 检查宝塔备份

```bash
# 宝塔默认数据库备份路径
ls -la /www/backup/database/

# 查找所有sql备份文件
find /www -name "*.sql.gz" -o -name "*.sql" 2>/dev/null | head -20
find /www/backup -type f -name "*abc789*" 2>/dev/null
```

#### 2.2 登录宝塔面板检查

1. 登录宝塔面板 → 数据库
2. 点击数据库名旁的「备份」按钮查看历史备份
3. 宝塔 → 计划任务 → 查看是否有数据库备份任务

#### 2.3 恢复备份

```bash
# 如果找到备份文件
gunzip /www/backup/database/abc789_xxx.sql.gz
mysql -u abc789 -p abc789 < /www/backup/database/abc789_xxx.sql
```

### 方案3：MySQL数据文件恢复（InnoDB）

如果 binlog 没开启，尝试从 InnoDB 数据文件恢复。

#### 3.1 检查 innodb_file_per_table

```sql
SHOW VARIABLES LIKE 'innodb_file_per_table';
```

如果是 ON，每个表有独立的 .ibd 文件。虽然 DROP TABLE 会删除 .ibd 文件，但**文件系统可能还没覆盖**。

#### 3.2 使用文件恢复工具

```bash
# 安装 extundelete 或 testdisk 尝试恢复已删除的 .ibd 文件
# ⚠️ 前提是要停止MySQL，避免磁盘写入覆盖
apt-get install extundelete  # Debian/Ubuntu
# 或
yum install extundelete      # CentOS

# 查找MySQL数据目录
ls /www/server/data/abc789/

# 尝试恢复已删除文件
extundelete /dev/sda1 --restore-directory /www/server/data/abc789/
```

#### 3.3 使用 undrop-for-innodb 工具

这是专门恢复 InnoDB 数据的工具：
```bash
git clone https://github.com/twindb/undrop-for-innodb.git
cd undrop-for-innodb
make

# 扫描表空间文件恢复数据
./stream_parser /www/server/data/ibdata1
```

### 遇到的问题及解决
1. **SHOW命令在bash执行** → 需在 `mysql>` 提示符下执行
2. **mysqlbinlog未安装** → `apt install mysql-server-core-8.0 -y`
3. **abc789用户权限不足** → 使用root用户执行binlog重放
4. **Duplicate column错误** → 使用 `-f` (force) 参数跳过，不影响数据恢复
