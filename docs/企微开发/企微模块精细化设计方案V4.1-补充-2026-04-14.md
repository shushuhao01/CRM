## 十五、会话存档席位配额管理

> 会话存档是企微付费增值服务，按「席位」购买
> 席位 = 开通会话存档的企微成员数量
> 选中生效范围的用户数量不能超过已购买的席位数
> 以企微成员为最小单位计数

### 15.1 席位概念说明

```
席位(Seat):
  - 1个席位 = 1个企微成员开通会话存档功能
  - 购买20个席位 = 最多20个成员可被选中生效
  - 超出已购席位数时，不可再选中新成员
  - 席位来源: SaaS模式由admin后台VAS订单管理；私有化由授权码控制

名词对齐:
  - 企微官方: "开通范围" / "许可用户"
  - 本系统: "席位" / "生效成员" / "存档用户"
  - 企微API: get_permit_user_list 返回已许可成员列表

计费模式:
  - SaaS: 购买席位包(如20席位/年)，admin后台配置上限
  - 私有化: 授权码中内嵌席位上限，本地校验
```

### 15.3 席位配额数据来源

```
SaaS模式:
  数据来源: admin后台 VAS订单管理
  表: vas_orders / tenant_settings
  字段: chat_archive_seats (已购席位数)
  查询: GET /admin/tenants/:id/vas -> chatArchiveSeats
  租户端: GET /wecom/chat-archive/quota -> { purchased, used, remaining, expireDate }

私有化模式:
  数据来源: 本地授权码解析
  授权码结构中包含: chatArchiveMaxSeats
  查询: 读取本地授权配置 -> chatArchiveMaxSeats
  若未限制: maxSeats = Infinity (不限席位)

API:
  GET  /wecom/chat-archive/seat-quota
  返回: {
    purchasedSeats: number,      // 已购席位
    usedSeats: number,           // 已生效成员数
    remainingSeats: number,      // 剩余可用
    expireDate: string | null,   // 到期时间(SaaS) / null(私有化永久)
    mode: 'saas' | 'private',   // 当前模式
    members: [{                  // 已生效成员列表
      userId: string,
      userName: string,
      department: string,
      enabledAt: datetime
    }]
  }

  PUT  /wecom/chat-archive/seat-members
  请求: { memberUserIds: string[] }
  校验: memberUserIds.length <= purchasedSeats
  动作: 更新生效成员列表 + 同步到企微(set_permit_user_list 若有此API权限)

  注意: 企微官方的"开通成员"是在企微管理后台设置的
        本系统的"生效成员"是本地管理层面的配额控制
        两者应保持一致，可通过 get_permit_user_list 同步校对
```

### 15.4 席位超限处理

```
场景处理:

1. 选择成员时超限:
   - 前端禁止勾选(按钮置灰 + 提示"剩余席位不足")
   - 部门批量选择时若超限，弹出手动挑选对话框

2. VAS过期/降级导致席位缩减:
   - 原20席位降为10席位，但已有15人生效
   - 后端标记: overQuota = true
   - 前端显示: 红色警告 "当前生效成员(15)超出席位上限(10)"
   - 不强制踢出已生效成员，但禁止新增
   - 管理员需手动取消部分成员的生效状态

3. 同步校验:
   - 定时任务(每日): 对比本系统生效成员 vs 企微许可成员
   - 差异: 提示管理员处理
   - 同步方向: 以本系统为准(管理端) -> 提示在企微后台同步配置

4. 顶部席位状态栏(增强V4第7.2节):
   已购席位不足时:
   +------------------------------------------------------------------+
   | 席位: 15/20 [========75%] 到期:2027-03-15  [续费] [增购]         |
   +------------------------------------------------------------------+

   席位即将用完时(>=90%):
   +------------------------------------------------------------------+
   | ⚠️ 席位: 19/20 [==================95%] 仅剩1个  [立即增购]       |
   +------------------------------------------------------------------+

   席位超限时(降级后):
   +------------------------------------------------------------------+
   | 🔴 席位超限: 15/10 [超出5个] 请减少生效成员或增购  [处理]        |
   +------------------------------------------------------------------+
```

## 十六、多租户安全隔离设计

> 每个二级菜单、操作、配置、企业、用户都需做到安全隔离
> AI模块同样需要租户级别隔离
> 确保租户A无法访问/操作/查看租户B的任何数据

### 16.1 隔离原则

```
隔离层级:

  ┌─────────────────────────────────────────────────┐
  │  租户A (tenantId = 'tenant_001')                │
  │  ┌──────────┐ ┌──────────┐ ┌──────────┐       │
  │  │ 企微配置  │ │ 客户数据  │ │ AI配置    │       │
  │  │ configA1  │ │ 客户a1.. │ │ 模型a1.. │       │
  │  │ configA2  │ │ 客户a2.. │ │ 智能体a. │       │
  │  └──────────┘ └──────────┘ └──────────┘       │
  └─────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────┐
  │  租户B (tenantId = 'tenant_002')                │
  │  ┌──────────┐ ┌──────────┐ ┌──────────┐       │
  │  │ 企微配置  │ │ 客户数据  │ │ AI配置    │       │
  │  │ configB1  │ │ 客户b1.. │ │ 模型b1.. │       │
  │  └──────────┘ └──────────┘ └──────────┘       │
  └─────────────────────────────────────────────────┘

  完全隔离: 租户A 永远看不到/操作不了 租户B 的数据
```

### 16.2 后端隔离实现(已有机制 + 需确认覆盖)

```
现有隔离机制(确认):
  1. 请求上下文: 每个请求通过JWT提取 tenantId
  2. 数据库查询: 所有查询自动加 WHERE tenantId = ?
  3. 中间件: TenantGuard 中间件校验租户身份

隔离实现方式(TypeORM/数据库层面):

  方式1: 查询条件注入(当前使用)
    // 每个Service方法手动加 tenantId 条件
    async getCustomers(tenantId: string, query: any) {
      return this.repo.find({
        where: { tenantId, ...query }
      })
    }

  方式2: 全局查询过滤器(推荐增强)
    // TypeORM Global Subscriber / Scope
    // 自动为所有查询注入 tenantId 条件
    // 防止开发者遗漏

  推荐: 在现有方式1基础上，增加方式2作为安全兜底
```

### 16.4 AI模块租户隔离(重点)

```
AI模块的隔离尤其重要:
  - 模型API Key: 每个租户自己的Key，不可跨租户使用
  - 知识库: 租户A的产品资料不能被租户B的AI读取
  - 会话内容: AI分析的聊天记录严格租户隔离
  - 提示词: 租户自定义的系统提示词属于租户私有
  - 日志: AI调用日志按租户隔离，含Token消耗统计

AI隔离实现要点:

  1. API Key存储隔离:
     WecomAiModel.tenantId = 当前租户
     查询时: WHERE tenantId = ? AND id = ?
     不同租户可以配置同一提供商但用不同Key

  2. 知识库检索隔离:
     知识库embedding搜索时，必须带 tenantId 条件
     SELECT * FROM wecom_knowledge_entries
     WHERE tenant_id = ? AND knowledge_base_id IN (
       SELECT id FROM wecom_knowledge_bases WHERE tenant_id = ?
     )
     ORDER BY similarity DESC

  3. AI调用上下文隔离:
     组装Prompt时，仅查询当前租户的会话记录
     async buildPrompt(tenantId, conversationId) {
       const messages = await chatRecordRepo.find({
         where: { tenantId, conversationId }
       })
       // 绝不会查到其他租户的聊天内容
     }

  4. Token消耗与费用隔离:
     WecomAiLog.tenantId 记录每次调用的租户
     统计Token消耗时按 tenantId 聚合
     费用预估按租户独立计算

  5. 智能体配置隔离:
     租户A创建的"质检分析师"与租户B的完全独立
     即使名称相同，也是不同记录(不同tenantId)
```

### 16.5 跨租户防护措施

```
防护层级(纵深防御):

  Layer 1: 路由中间件
    TenantGuard: 从JWT提取tenantId，注入到request上下文
    所有API路由必须经过TenantGuard

  Layer 2: Service层
    所有数据库操作必须传入tenantId参数
    禁止不带tenantId的查询(代码审查规范)

  Layer 3: 数据库层
    TypeORM Global Subscriber: 自动为INSERT注入tenantId
    查询拦截器: 自动为SELECT/UPDATE/DELETE注入tenantId条件

  Layer 4: API响应过滤
    返回数据前，二次校验tenantId是否匹配
    防止关联查询时意外返回其他租户数据

  Layer 5: 日志审计
    记录所有跨租户访问尝试(即使被拦截)
    异常告警: 同一用户短时间内尝试多个tenantId

安全编码规范(所有新增模块必须遵守):

  // 正确示例
  async getAiModels(tenantId: string) {
    return this.aiModelRepo.find({
      where: { tenantId }
    })
  }

  // 错误示例(禁止)
  async getAiModel(id: number) {
    return this.aiModelRepo.findOne({ where: { id } })
    // 缺少tenantId！可能返回其他租户的模型
  }

  // 正确的单条查询
  async getAiModel(tenantId: string, id: number) {
    return this.aiModelRepo.findOne({
      where: { tenantId, id }
    })
  }
```

### 16.6 企微配置级隔离(多配置场景)


### 16.7 权限隔离(角色+功能级)


## 十七、UI现代化设计规范

> 当前企微管理各页面功能可用但UI体验一般
> 目标: 现代化、简约、高效、美观，体验感好
> 风格: 浅色系为主，轻量卡片，细腻阴影，不使用大块深色填充
> 参考: 主流SCRM产品的视觉风格(微伴、尘锋、探马等)

### 17.1 整体设计原则

```
核心原则:

  1. 简约不简单
     - 去除多余边框线，用间距和阴影区分层次
     - 信息密度适中，不堆砌功能
     - 重要信息突出，次要信息收敛

  2. 浅色系主调
     - 背景: #F5F7FA (浅灰蓝，比纯白更柔和)
     - 卡片: #FFFFFF (纯白卡片 + box-shadow)
     - 主色: #4C6EF5 (明亮蓝，企业感但不沉闷)
     - 辅色: #7C3AED (紫色用于AI相关功能)
     - 成功: #10B981  警告: #F59E0B  危险: #EF4444
     - 禁止: 大面积深色背景、大面积纯黑块

  3. 信息层次清晰
     - 一级标题: 18px 600粗 #1F2937
     - 二级标题: 15px 500粗 #374151
     - 正文:     14px 400正常 #4B5563
     - 辅助文字: 12px 400正常 #9CA3AF
     - 数字强调: 24-32px 700粗 主色

  4. 交互友好
     - 按钮有hover过渡(0.2s ease)
     - 卡片有hover微上浮(translateY -2px + shadow增强)
     - 操作有即时反馈(加载动画/成功提示)
```

### 18.1 H5登录流程增强

```
当前V4方案的登录流程:
  H5加载 -> verify-binding -> 未绑定 -> 输入用户名+密码 -> 绑定成功

增强后的登录流程:

    b. 输入授权码(一次性码，由管理员生成)
    c. 企微管理员预配置(URL中嵌入tenantCode参数)

  步骤2: 账号登录(现有增强)
    在确定租户后，输入该租户下的CRM账号密码
```

### 18.7 H5侧边栏绑定流程中获取通讯录信息

```
侧边栏H5绑定完成后，系统自动做以下事情:

  1. 获取当前企微成员信息:
     wx.invoke('getCurExternalContact') -> externalUserId
     wx.invoke('getContext') -> userId (当前登录的企微成员)

  2. 通过userId查找企微通讯录中的成员:
     GET /wecom/address-book/member?userId=wang_sales
     返回: { name, department, avatar, phone, ... }

  3. 自动创建绑定记录:
     POST /wecom/sidebar/bind-account
     请求: {
       wecomUserId: 'wang_sales',   // 从JS-SDK获取
       corpId: 'ww1234567890',      // 从JS-SDK获取
       tenantCode: 'yunkes',        // 从步骤1获取
       username: 'admin_wx',       // CRM用户名
       password: '******'           // CRM密码
     }

  4. 后端处理:
     a. 验证tenantCode -> 获取tenantId
     b. 验证CRM用户名密码 -> 获取crmUserId
     c. 查询wecomUserId对应的通讯录信息
     d. 检查是否已有绑定 -> 有则更新，无则创建
     e. 创建WecomUserBinding记录
     f. 返回sidebarToken(后续免登录)

  5. 绑定成功后:
     自动同步该成员的通讯录信息到绑定记录
     后续该成员在企微侧边栏打开H5页面时
     自动携带sidebarToken，无需再次登录
```

### 19.1 企微应用接入模式说明

```
企业微信应用接入有两种方式:

模式A: 第三方应用(系统提供应用)
  - 由SaaS平台方注册并认证的第三方应用
  - 企业管理员在企微后台扫码授权安装
  - 平台方持有 suite_id / suite_secret
  - 企业授权后获得 permanent_code(永久授权码)
  - 优势: 企业无需创建应用，扫码即用
  - 限制: 需平台通过企微审核，接口权限由企微控制

模式B: 自建应用(企业自行创建)
  - 企业在自己的企微后台创建自建应用
  - 填入 CorpID / AgentID / Secret
  - 企业自行管理应用权限和可见范围
  - 优势: 权限自控，无需第三方授权审核
  - 限制: 需要企业IT有一定配置能力

适用场景:
  SaaS版:
    推荐: 第三方应用(扫码授权，对企业最简单)
    备选: 自建应用(有IT团队的企业可能更倾向自控)

  私有化部署版:
    默认: 自建应用(独立部署，自行管理)
    不适用: 第三方应用(没有SaaS平台方)
```
