#### 消息统计数据来源说明

```
消息收发统计有两种数据来源:

方式1: 会话存档(已开通) - 精准到每条消息
  - 数据来源: wecom_chat_records 表
  - sentCount: FROM where from_user_id = externalUserId
  - recvCount: FROM where to_user_ids LIKE externalUserId
  - 优点: 精确到条，可查看具体聊天内容
  - 缺点: 需要开通会话存档(付费功能)

方式2: 联系客户统计API(无需会话存档) - 聚合数据
  - API: POST /cgi-bin/externalcontact/get_user_behavior_data
  - 官方文档: https://developer.work.weixin.qq.com/document/path/92132
  - 返回字段:
    * chat_cnt: 聊天总数(单聊/群聊)
    * message_cnt: 发送消息数
    * reply_percentage: 已回复聊天占比
    * avg_reply_time: 平均首次回复时长(秒)
  - 注意: 该API按「成员」维度聚合，不区分具体客户
  - 适用: 无会话存档时展示成员级概览统计

推荐策略:
  - 已开通会话存档 -> 主表格显示精确的 发N/收N
  - 未开通会话存档 -> 主表格消息列显示 "-" 或 "需开通会话存档"
  - 成员级统计(behavior_data) 在获客助手/数据分析模块展示
```

     |         |

     |         |
     v         v
  写入关联   标记rejected

```

#### 新增数据模型

```
WecomAutoMatchSuggestion(自动匹配建议表)
- id: number (PK)
- tenantId: string
- wecomCustomerId: number (FK -> wecom_customers)
- crmCustomerId: string (FK -> customers)
- matchType: 'phone' | 'name' (匹配方式)
- matchField: string (匹配的值，如手机号)
- confidence: 'high' | 'medium' | 'low'
- status: 'pending' | 'confirmed' | 'rejected'
- confirmedBy: string (确认人)
- confirmedAt: datetime
- createdAt, updatedAt
```

### 5.8 智能上下线与权重设计(核心增强)

> 获客链接的接待成员分配策略: 权重、上下线、次日重新上线

#### 概念说明

```
名词解释:
- 权重: 每个接待成员的客户分配比例权重(如 7:5:3)
- 上线: 该成员可以接待新客户
- 下线: 该成员暂停接待(已添加客户不受影响)
- 智能上下线: 系统根据规则自动切换成员的上线/下线状态
- 次日重新上线: 当日下线的成员在次日指定时间自动恢复上线
```

## 十六、多租户安全隔离设计

> 所有菜单/操作/配置/企业/用户/AI都做到租户级安全隔离

### 16.1 隔离原则

```
完全隔离: 租户A 永远看不到/操作不了 租户B 的数据
隔离维度:
  第一级: tenantId (租户间完全隔离)
  第二级: wecomConfigId (同租户不同企微配置间数据分开)
```

### 16.2 隔离检查清单(全模块覆盖)

```
所有模块的读/写操作均需 tenantId 条件:
  企微配置/客户/客户群/获客助手/活码/会话存档/席位配置
  微信客服/通讯录/绑定/侧边栏/收款/群模板/自动匹配建议
  智能上下线规则/AI模型/AI智能体/AI质检策略/AI质检结果
  AI知识库/AI知识条目/AI话术/AI日志/AI敏感词/AI自动标签规则
```

### 16.3 AI模块租户隔离(重点)

```
- API Key存储隔离: 每个租户自己的Key
- 知识库检索隔离: embedding搜索带tenantId条件
- AI调用上下文隔离: Prompt仅含当前租户会话记录
- Token消耗隔离: 按tenantId聚合统计
- 智能体配置隔离: 同名不同租户完全独立
```

### 16.4 纵深防御5层

```
Layer 1: TenantGuard中间件(JWT提取tenantId)
Layer 2: Service层(所有DB操作必须传tenantId)
Layer 3: TypeORM全局查询过滤器(自动注入tenantId)
Layer 4: API响应过滤(二次校验tenantId匹配)
Layer 5: 日志审计(记录跨租户访问尝试)
```

### 16.5 角色权限矩阵

```
| 功能                    | 管理员 | 主管 | 销售 | 客服 |

| 企微授权/配置           | v      |      |      |      |

| AI模型/智能体配置       | v      |      |      |      |

## 十九、企微授权增强 - 第三方授权与自建应用双模式

> 第三方应用: SaaS平台提供，企业管理员扫码授权即用
> 自建应用: 企业自行创建，手动填写CorpID/Secret
> admin后台控制两个按钮的可见性

### 19.1 接入方式选择

```
+----------------------------+  +----------------------------+
| 📱 第三方应用授权          |  | 🔧 自建应用配置            |
| 扫码即可完成授权           |  | 手动填写CorpID和Secret    |
| 推荐SaaS版使用             |  | 适合私有化部署             |
| [开始授权 ->]               |  | [手动配置 ->]               |
+----------------------------+  +----------------------------+

SaaS版: 两个按钮都显示(推荐第三方)
私有化: 默认仅显示自建应用(admin可控制)
```

### 19.2 第三方授权流程

```
点击[开始授权] -> 生成授权链接(suite_id+pre_auth_code)
-> 弹出扫码二维码 -> 企业管理员扫码确认
-> 回调获取auth_code -> 换取permanent_code
-> 自动创建WecomConfig记录 -> 授权完成

企微API:
  POST /cgi-bin/service/get_pre_auth_code
  POST /cgi-bin/service/get_permanent_code
  POST /cgi-bin/service/get_corp_token

WecomConfig新增字段:
  authMode: 'third_party' | 'self_built'
  permanentCode, authCorpName, authAdminUserId, authTime, authScope
```

### 19.3 自建应用分步引导

```
步骤1: 企业信息(CorpID+AgentID)
步骤2: 应用Secret(必填+测试连接)
步骤3: 功能Secret(按需: 通讯录/客户联系/会话存档)
步骤4: 配置完成(状态汇总+下一步建议)
```
