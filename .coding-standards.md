## 一、编码规范 —— 彻底杜绝乱码

### 1.1 文件编码：全部使用 UTF-8 无 BOM

| 规则 | 说明 |
|------|------|
| 所有源代码文件 | UTF-8 无 BOM（`.ts`, `.vue`, `.js`, `.json`, `.css`, `.scss`, `.html`, `.md`） |
| `.env` 配置文件 | UTF-8 无 BOM（中文注释可能导致乱码，**值中避免直接写中文**） |
| SQL 脚本文件 | UTF-8 无 BOM |
| 批处理/Shell脚本 | `.sh` 用 UTF-8 + LF；`.bat`/`.ps1` 用 UTF-8（可带 BOM） |

**IDE 设置（JetBrains / VS Code）：**
```
File → Settings → Editor → File Encodings
  - Global Encoding: UTF-8
  - Project Encoding: UTF-8
  - Default encoding for properties files: UTF-8
  - ☑ Create UTF-8 files: with NO BOM
```

**验证方法：**
```powershell
# 检查文件是否含有 BOM
$bytes = [System.IO.File]::ReadAllBytes("path/to/file.ts")
if ($bytes[0] -eq 0xEF -and $bytes[1] -eq 0xBB -and $bytes[2] -eq 0xBF) {
    Write-Host "⚠️ 文件含有 BOM 头"
}
```

### 1.2 换行符：统一 LF

项目已配置 `.gitattributes`（`* text=auto eol=lf`）和 `.editorconfig`（`end_of_line = lf`）。

**注意事项：**
- Windows 上 Git 默认 `core.autocrlf=true` 会自动转换，需要确认设置：
  ```powershell
  git config core.autocrlf input   # 推荐：提交时转 LF，检出时不转换
  git config core.eol lf           # 工作目录使用 LF
  ```
- 如发现文件换行符不一致，可批量修复：
  ```powershell
  # 重新规范化所有文件的换行符
  git add --renormalize .
  git commit -m "fix: normalize line endings to LF"
  ```

### 1.3 数据库编码：utf8mb4

| 层级 | 配置 |
|------|------|
| 数据库级别 | `CREATE DATABASE crm CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;` |
| 表级别 | `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci` |
| 连接级别 | `.env` 中 `DB_CHARSET=utf8mb4`（已配置） |
| TypeORM 配置 | `database.ts` 中 `charset: 'utf8mb4'`（已配置） |

**建表模板：**
```sql
CREATE TABLE IF NOT EXISTS `table_name` (
  `id` VARCHAR(36) NOT NULL,
  `name` VARCHAR(255) NOT NULL COMMENT '名称',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='表注释';
```

### 1.4 HTTP 响应编码

```typescript
// ✅ JSON 响应（Express 自动设置 Content-Type: application/json; charset=utf-8）
res.json({ message: '操作成功' });

// ✅ HTML 响应

// ✅ CSV 导出（必须加 BOM 头，否则 Excel 打开乱码）

res.setHeader('Content-Disposition', `attachment; filename*=UTF-8''${fileName}`);
```

### 1.5 前端请求编码

```typescript

// ✅ 正确

// ⚠️ URL 参数中的中文必须编码
```

## 二、JSON 处理规范

### 2.1 绝对禁止：手动拼接 JSON

```typescript
// ❌ 绝对禁止 - 手动拼接（特殊字符、引号、换行都会出错）
const json = '{"name":"' + userName + '","desc":"' + desc + '"}';
const json2 = `{"name":"${userName}"}`;

// ✅ 唯一正确方式 - JSON.stringify
const json = JSON.stringify({ name: userName, desc });
```

### 2.2 数据库存取 JSON

```typescript
// ❌ 错误 - 直接拼 SQL（SQL注入 + 转义问题）
await db.query(`UPDATE config SET value = '${jsonStr}' WHERE key = 'settings'`);

// ✅ 正确 - 参数化查询 + JSON.stringify
await db.query(
  `UPDATE config SET value = ? WHERE key = ?`,
  [JSON.stringify(data), 'settings']
);

// ✅ 正确 - 读取时 JSON.parse 并做容错处理
const row = await db.query(`SELECT value FROM config WHERE key = ?`, ['settings']);
let config: MyConfig;
try {
  config = JSON.parse(row[0]?.value || '{}');
} catch (e) {
  console.error('JSON 解析失败:', e);
  config = {} as MyConfig;  // 给默认值，不要让程序崩溃
}
```

### 2.3 API 响应

```typescript
// ❌ 错误 - 手动序列化
res.send(JSON.stringify({ success: true, data }));
res.setHeader('Content-Type', 'application/json');
res.end('{"code":0}');

// ✅ 正确 - 让 Express 框架处理
res.json({ success: true, data });

// ✅ 正确 - 统一响应格式
res.json({
  code: 0,
  message: 'success',
  data: result,
});
```

### 2.4 前端 Vue 模板中的对象

```vue
<!-- ❌ 错误 - 在模板中写 JSON 字符串 -->
<MyComponent :config="'{"key": \"value\"}'">
<MyComponent :config='{"key": "value"}'>

<!-- ✅ 正确 - 使用 JavaScript 对象 -->
<MyComponent :config="{ key: 'value' }">

<!-- ✅ 正确 - 使用 data/computed -->
<MyComponent :config="myConfig">
```

```typescript
// script setup 中定义
const myConfig = reactive({ key: 'value' });
```

### 2.5 JSON 文件（配置文件等）

```jsonc
// ✅ 标准 JSON 要求：
// 1. 键名必须用双引号包裹
// 2. 字符串值必须用双引号（不能用单引号）
// 3. 最后一个属性后不能有尾逗号
// 4. 不支持注释（.jsonc 文件除外）
// 5. 不支持 undefined、NaN、Infinity

// ❌ 错误
{
  name: "test",          // 键名没有引号
  'value': 'hello',     // 用了单引号
  "list": [1, 2, 3,],   // 尾逗号
  "data": undefined,     // 不支持 undefined
}

// ✅ 正确
{
  "name": "test",
  "value": "hello",
  "list": [1, 2, 3],
  "data": null
}
```

### 2.6 JSON.parse 安全调用

```typescript
/**
 * 安全的 JSON 解析，失败时返回默认值而非抛异常
 */
function safeJsonParse<T>(str: string, defaultValue: T): T {
  if (!str || str === 'undefined' || str === 'null') {
    return defaultValue;
  }
  try {
    return JSON.parse(str) as T;
  } catch {
    console.warn(`JSON 解析失败: ${str.substring(0, 100)}`);
    return defaultValue;
  }
}

// 使用示例
const config = safeJsonParse<AppConfig>(rawString, { theme: 'light', lang: 'zh-CN' });
const list = safeJsonParse<string[]>(rawString, []);
```

## 三、TypeScript 编码规范

### 3.1 类型安全

```typescript
// ❌ 避免 any（除非有特殊理由并写注释）
function process(data: any) { ... }

// ✅ 使用具体类型
interface CustomerData {
  id: string;
  name: string;
  phone?: string;
}
function process(data: CustomerData) { ... }

// ✅ 不确定类型时用 unknown + 类型守卫
function process(data: unknown) {
  if (typeof data === 'object' && data !== null && 'id' in data) {
    const customer = data as CustomerData;
    // ...
  }
}
```

### 3.2 空值安全

```typescript
// ❌ 危险 - 直接访问可能为空的属性
const name = response.data.customer.name;

// ✅ 安全 - 可选链 + 空值合并
const name = response?.data?.customer?.name ?? '未知';

// ✅ 安全 - 数组访问
const first = list?.[0] ?? null;

// ✅ 安全 - 函数调用
const result = callback?.();
```

### 3.3 字符串模板 vs 拼接

```typescript
// ❌ 不推荐
const msg = '用户 ' + name + ' 在 ' + time + ' 登录了系统';

// ✅ 推荐 - 模板字符串
const msg = `用户 ${name} 在 ${time} 登录了系统`;

// ⚠️ 但 SQL 语句中，绝不能用模板字符串拼接值
// ❌ SQL注入风险
const sql = `SELECT * FROM users WHERE name = '${name}'`;

// ✅ 参数化查询
const sql = `SELECT * FROM users WHERE name = ?`;
await db.query(sql, [name]);
```

### 3.4 异步错误处理

```typescript
// ❌ 异步操作没有 try-catch
router.get('/api/data', async (req, res) => {
  const data = await service.getData();
  res.json(data);
});

// ✅ 必须捕获异常
router.get('/api/data', async (req, res) => {
  try {
    const data = await service.getData();
    res.json({ code: 0, data });
  } catch (error) {
    console.error('获取数据失败:', error);
    res.status(500).json({ code: -1, message: '服务器内部错误' });
  }
});
```

## 七、错误预防清单

### 7.1 上线前自查表

- [ ] 所有文件编码为 UTF-8 无 BOM
- [ ] 数据库字符集为 utf8mb4
- [ ] HTML 页面包含 `<meta charset="UTF-8">`
- [ ] CSV 导出添加了 BOM 头（`\uFEFF`）
- [ ] 所有 SQL 使用参数化查询（无字符串拼接）
- [ ] JSON 使用 `JSON.stringify` / `JSON.parse`（无手动拼接）
- [ ] API 响应使用 `res.json()`（非 `res.send(JSON.stringify(...))`）
- [ ] 异步路由全部有 `try-catch`
- [ ] 新增路由已在 `index.ts` 中注册
- [ ] 无未使用的 import
- [ ] 无循环依赖
- [ ] 新增依赖检查了 ESM/CJS 兼容性

### 7.2 常见乱码场景与解决方案

| 场景 | 原因 | 解决方案 |
|------|------|----------|
| 数据库存取中文乱码 | 字符集不是 utf8mb4 | 确保数据库、表、连接都用 utf8mb4 |
| CSV 导出 Excel 乱码 | 缺少 BOM 头 | 输出内容前加 `\uFEFF` |
| API 返回中文乱码 | Content-Type 未指定 charset | 用 `res.json()` 或手动设 `charset=utf-8` |
| 前端页面乱码 | HTML 缺少 meta charset | 加 `<meta charset="UTF-8">` |
| 文件上传中文文件名乱码 | multer latin1 编码 | `Buffer.from(name, 'latin1').toString('utf8')` |
| Node.js 控制台乱码 | Windows 默认 GBK 编码 | 设置 `chcp 65001` 或 `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8` |
| Git diff 中文乱码 | Git 默认不转义 | `git config --global core.quotepath false` |
| `.env` 文件中文乱码 | 编辑器保存编码不对 | 确保用 UTF-8 保存，值中避免直接写中文 |
| JSON 中特殊字符报错 | 手动拼接未转义 | 使用 `JSON.stringify` |
| 邮件内容乱码 | 未指定 Content-Type | 设置 `charset: 'utf-8'`（已在项目中配置） |

### 7.3 常见 JSON 错误与修复

| 错误信息 | 原因 | 修复 |
|----------|------|------|
| `SyntaxError: Unexpected token` | JSON 格式不合法 | 检查引号、逗号、括号匹配 |
| `SyntaxError: Unexpected end of JSON input` | JSON 字符串不完整 | 检查字符串是否被截断 |
| `SyntaxError: Bad control character` | 含有未转义的控制字符 | 用 `JSON.stringify` 代替手动拼接 |
| `TypeError: Converting circular structure to JSON` | 对象有循环引用 | 移除循环引用或用自定义 replacer |
| `JSON.parse(undefined)` | 传入了 undefined | 加空值检查 `str \|\| '{}'` |

## 八、实用工具函数

建议在项目中统一维护以下工具函数：

### 8.1 后端工具 (`backend/src/utils/encoding.ts`)

```typescript
/**
 * 安全 JSON 解析
 */
export function safeJsonParse<T = unknown>(str: string | null | undefined, defaultValue: T): T {
  if (!str || str === 'undefined' || str === 'null') return defaultValue;
  try {
    return JSON.parse(str);
  } catch {
    return defaultValue;
  }
}

/**
 * CSV 添加 BOM 头（防止 Excel 乱码）
 */
export function addBom(content: string): string {
  return '\uFEFF' + content;
}

/**
 * 安全的中文文件名编码（用于 Content-Disposition）
 */
export function encodeFileName(fileName: string): string {
  const encoded = encodeURIComponent(fileName);
  return `attachment; filename="${encoded}"; filename*=UTF-8''${encoded}`;
}

/**
 * 解码 multer 上传的中文文件名
 */
export function decodeMulterFileName(name: string): string {
  try {
    return Buffer.from(name, 'latin1').toString('utf8');
  } catch {
    return name;
  }
}
```

### 8.2 前端工具 (`src/utils/encoding.ts`)

```typescript
/**
 * 安全 JSON 解析
 */
export function safeJsonParse<T = unknown>(str: string | null | undefined, defaultValue: T): T {
  if (!str || str === 'undefined' || str === 'null') return defaultValue;
  try {
    return JSON.parse(str);
  } catch {
    return defaultValue;
  }
}

/**
 * URL 安全编码（中文参数）
 */
export function safeEncodeURIComponent(str: string): string {
  return encodeURIComponent(str).replace(/[!'()*]/g, (c) =>
    '%' + c.charCodeAt(0).toString(16).toUpperCase()
  );
}

/**
 * 下载文件时安全设置文件名
 */
export function downloadWithFileName(blob: Blob, fileName: string): void {
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = fileName;
  document.body.appendChild(a);
  a.click();
  document.body.removeChild(a);
  URL.revokeObjectURL(url);
}
```

## 十、AI 辅助开发注意事项

使用 AI 工具（GitHub Copilot / Kiro 等）生成代码时，需特别注意：

1. **检查 AI 生成的 JSON 字符串** —— AI 可能手动拼接 JSON，必须替换为 `JSON.stringify`
2. **检查 AI 生成的 SQL** —— AI 可能用字符串模板拼接，必须改为参数化查询
3. **检查 AI 生成的文件编码** —— 确保新文件以 UTF-8 无 BOM 保存
4. **检查 AI 生成的中文注释** —— 确保不在 `.env` 值中包含中文
5. **检查 AI 生成的 import** —— 确保新路由在 `index.ts` 中注册
